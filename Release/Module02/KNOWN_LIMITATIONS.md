# Known Limitations — Module 02 (Classification)

## Deployment model — read this first

**No autonomous production `ClassificationProvider` exists.** `ClaudeLiveClassifier` — v1's only concrete provider — requires a live, agent-driven Claude session; its `classify()` method raises `NotImplementedError` if invoked any other way, by design. Running Module 02 outside a live session (e.g. a scheduled/unattended `python -m src.main` invocation) is **safe** — every text/vision-dependent file gracefully falls back to `Category.UNKNOWN` (`fallback_reason: "provider_exception"`) rather than crashing or silently guessing — but it is **not functional** for judgment-dependent classification. Building an autonomous provider (a networked LLM API call, a local model, or a rule-based fallback classifier) is explicitly out of scope for v1 (`Build-out/02 Classification/Module 02 Design.md` §23-A, §25) and is the single largest gap between "feature complete" and "fully autonomous."

(Promoted to the top of this document after an independent release audit, 2026-07-06 — `Release/Module02/RELEASE_AUDIT.md`, finding F1 — found this fact was previously only one bullet among many below, while the release status fields elsewhere used unqualified "production-ready" language that didn't surface it. See `MODULE_STATUS.md`, `RELEASE_NOTES.md`, and `MODULE_CONTRACT.md` for the corrected wording.)

## Known limitations

- **Screenshot heuristic false-positives on real photos with no camera EXIF.** `Rules/Classification Rules.md`'s Screenshot-vs-Image split treats "no camera EXIF present" as one of three OR'd conditions — correct for its intended purpose (screenshots never carry camera EXIF), but it means any real photo that also lacks camera EXIF (common for images stripped of metadata by messaging apps, some cloud uploads, or older cameras) will be classified as Screenshot, not Image. Confirmed against both a synthetic test sample and a live UAT file (`IMG_4821.jpg`, camera-style filename, no EXIF) — both landed on Screenshot. Not a code defect; the heuristic is implemented exactly as designed. Worth revisiting the OR-condition weighting if this proves too aggressive against real user photos.
- **`_COMMON_SCREEN_RESOLUTIONS` (in `src/core/images.py`) is a code-level constant, not yet formalized in `Rules/Classification Rules.md`.** Implemented at implementation time to make the resolution-matching condition concrete; the Rules doc should be updated to list the actual resolution set if it needs to be treated as a governed business rule rather than an implementation detail.
- **Vision-mode request contract carries no image bytes.** `ClassificationRequest` (the provider-boundary dataclass) has `path`/`mode`/`extracted_text`/`mime_type` but nothing for rendered image bytes. Fine for v1's `ClaudeLiveClassifier` (Claude has direct file access and can look at `path` itself when `mode == "vision"`), but a future network-based vision provider would need actual image bytes threaded through some other mechanism. `ClassificationEngine._extract()` only validates that a PDF page *can* be rendered (`render_page_as_image()` as a fail-fast check) without passing the bytes anywhere yet.
- **No Receipt category in v1's taxonomy.** A document that reads more like a payment receipt than a classic invoice (confirmed live during UAT: `payment_confirmation.pdf`, worded "RECEIPT / INVOICE," describing a completed payment) has no dedicated home — Invoice is the closest fit. Handled correctly via the `ambiguous` signal (for Module 06 to weigh later), not by inventing an unapproved category. Worth considering for a future Classification Rules revision if this pattern turns out to be common in real Downloads folders.
- **Fallback vocabulary has grown beyond the three reasons named in the frozen design (§11).** The design specified `provider_exception`/`invalid_response` for provider-side failures; implementation added `extraction_failed` (malformed file content, discovered during initial implementation) and `unreadable_content` (unreadable/non-image content in the image-split path, discovered during integration testing — see the defect writeup in `TEST_RESULTS.md`). Both are small, documented extensions of the same fallback discipline, not architecture changes — flagged here for visibility since the design doc itself only names three. Every fallback reason now also carries a sanitized `error_detail` diagnostic string in the action log (added after the release audit, F3), so this vocabulary growth is at least fully diagnosable going forward.
- **`ClassificationEngine`'s error handling, while now covering both the text-bearing and image-split branches, still can't anticipate every possible failure mode.** `classify_batch()`'s outer try/except remains the last line of defense for anything genuinely unanticipated (verified via `test_classify_batch_outer_safety_net_still_covers_truly_unanticipated_failures`, which forces a failure in an internal routing function itself). A record whose `current_path` breaks path handling in some way neither branch's try/except was designed around would still surface as an `error` log entry with `category` left `None`, not a graceful classify entry.
- **Judgment quality has only been UAT-tested against one batch of 18 files, and that validation was self-graded.** The single UAT run (`Runtime/UAT/Module02_UAT_2026-07-06_015818/`) covered a deliberately varied but still small set of cases (8 judgment calls total), graded by the same agent that implemented the classifier and authored the test files' expected answers — not an independent rater. This validates the architecture's plumbing and demonstrates sound reasoning on one deliberately hard case (an invoice/receipt worded ambiguously); it should not be read as statistically meaningful validation of judgment quality at scale. Broader real-world Downloads folders (larger volume, more categories of ambiguity, more languages, more file conditions) — ideally reviewed by someone other than the implementer — haven't been exercised yet.
- **`Tests/Large Batch/`'s image/PDF fixtures are still synthetic placeholder bytes**, inherited from Module 01 (which never needed real content). This is what surfaced the image-read-failure defect during integration testing — useful as an adversarial-content stress test, but not representative of a batch of real photos/PDFs at volume. Worth adding a large batch of *real* varied content before Module 03 needs to validate against it.
- **`_EXTENSION_CATEGORY_MAP`'s Archive extensions (`.rar`/`.7z`/`.gz`/`.tar`) are ahead of Module 01's current scope.** Module 01's `SUPPORTED_EXTENSIONS` only ingests `.zip` for archives today, so these four mappings are currently unreachable in practice — a deliberate forward-compatibility choice (found and made explicit during the release audit, F4; `.tar` itself was actually missing from the map until that audit caught the drift against `Rules/Classification Rules.md`). Now guarded by a permanent regression test (`test_extension_category_map_matches_rules_taxonomy`) so the Rules doc and the code can't silently diverge again, and by a second test (`test_every_module01_supported_extension_is_routed_by_module02`) that will fail loudly the moment Module 01 adds an extension Module 02 doesn't yet route.
- **`save_file_record()` performs a full read-modify-write of the entire cumulative `metadata_store.json` on every call, and `classify_batch()` calls it once per file in its loop.** Inherited unchanged from Module 01 (which already flagged the read side, `find_by_current_path()`'s linear scan, as a future concern "if the store grows into the thousands"). Module 02 makes this cost more prominent: classifying a batch of N files against a store that has already accumulated M records from all prior runs costs roughly O(N × M) total I/O, not just O(N). Not a problem at current/expected v1 volumes (found during the release audit, F9) — flagged here explicitly since it's now this module's problem too, not just Module 01's, and it will be the first thing worth optimizing (e.g. one load + one write per batch instead of per file) if store size or batch frequency grows materially.

## Intentional design decisions

- **`category` stays a plain `str` at the `ClassificationResult`/provider boundary**, not the `Category` enum — providers (especially a future non-Claude/LLM-API provider) naturally produce free text; `ClassificationEngine._validate_category()` is the deliberate trust boundary that converts/validates it into a real `Category` member. An unrecognized string is exactly the `invalid_response` fallback case, not a crash — confirmed adversarially (M02-S02).
- **`category=None` vs. `Category.UNKNOWN` are deliberately different states.** `None` means Module 01 never had readable bytes to try (`status == "unreadable"`); `Category.UNKNOWN` means Module 02 tried on a readable file and found no match or hit a known failure mode. Losing track of this distinction is an easy mistake for a future module to make — called out explicitly here and in the code's own comments.
- **`mode` (`deterministic`/`text`/`vision`) is logged, not persisted on `FileRecord`.** Deliberately scoped to the action log per the frozen design (§12) — `FileRecord` itself doesn't need to carry how a classification was reached, only what it is.
- **No retry, no secondary-provider chain in v1.** Any provider failure (unavailable, exception, invalid response) falls straight to `Category.UNKNOWN` — simple, auditable, and consistent with `Rules/Confidence Rules.md`'s existing hard floor (Unknown → always `review_required`), rather than inventing new retry/confidence logic at this layer.
- **`ClassificationEngine` is the only thing that knows about deterministic-vs-AI-vs-fallback decision-making; `ClassificationProvider` performs classification only.** This three-layer split (`classify_batch()` → `Engine` → `Provider`) was the design's most significant refinement and is fully implemented as specified — verified by `provider.received_requests == []` assertions throughout the test suite for every path that shouldn't reach a provider.
- **`ClassificationResult.notes` and `ProviderMetadata.reasoning` are two distinct free-text fields, both currently unused by `ClaudeLiveClassifier`.** No documented guidance yet on which a future provider should populate for what (noted during the release audit, F10) — `notes` lives on the classification answer itself, `reasoning` on the call's observability metadata. Worth a one-line clarification in the design doc before a second provider is built, to avoid inconsistent usage across providers.

## Deferred improvements for future modules

- Acting on `classification_signals` (ambiguous, multi_document_detected, non_english_detected, no_extractable_text, locked) to actually influence confidence scoring — Module 06.
- A second, networked `ClassificationProvider` implementation (to validate `ProviderMetadata`'s observability goal — comparing providers over time — which can't be meaningfully exercised with only one provider in existence).
- Formalizing `_COMMON_SCREEN_RESOLUTIONS` into `Rules/Classification Rules.md` if it needs governance beyond a code constant.
- Considering a dedicated Receipt category (or an explicit Rules-level tie-breaker for invoice-vs-receipt wording) if this pattern proves common.
- Replacing `Tests/Large Batch/`'s synthetic image/PDF fixtures with real varied content.
- Broader UAT coverage across more languages, larger volumes, and more categories of ambiguity than the single run executed so far.
