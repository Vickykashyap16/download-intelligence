# Known Limitations — Module 06 (Confidence & Review)

## Deployment model — read this first

Like Modules 04/05, there is no deployment-model caveat to state here. Module 06 has no provider, live or autonomous, of any kind (`Module 06 Design.md` §2, confirmed across four independent Design Review passes) — every deduction, hard floor, and tier lookup is a direct field read or null-check against an already-computed upstream signal. Module 06 is identically functional whether invoked during a live Claude session or an unattended/scheduled run; there is no judgment-dependent field that could degrade between the two.

## Known limitations

- **The "Corrupted file" hard floor is complete and correct for 7 of 11 categories, and inherited-incomplete for the other 4 — a disclosed gap in Modules 02/03's existing signal set, not a Module 06 defect.** `Module 06 Design.md` §2.4 traces this precisely: for Invoice/Resume/Bank Statement/Contract/Document/Image/Screenshot, `category == Category.UNKNOWN` is a real, verified, sufficient signal for "this file could not be meaningfully read," and the `unknown_category` hard floor fully covers it. For Archive/Audio, a corrupted file degrades to only an ordinary −8 missing-required-field deduction, not a forced `review_required`. For Application/Video, category is parsed from the filename string alone and content is never opened or validated at all — a corrupted Application or Video file is completely undetectable as corrupted anywhere in the current pipeline. Closing this gap would require Module 02 or Module 03 to detect and report corruption for these four categories — a change to an already-frozen module's contract, and therefore a matter for `Governance/FROZEN_MODULE_CHANGE_POLICY.md`, not something Module 06 can fix within its own scope. Module 06 was explicitly instructed at design time not to introduce a new corruption signal or duplicate existing state.
- **"Unknown category" and "Corrupted file" are logged as one identifier, `unknown_category`, never two.** `Rules/Confidence Rules.md` names these as two separate business rules, but Module 06 has no signal anywhere on `FileRecord` that distinguishes *why* a record is `Category.UNKNOWN` (ambiguous content, unsupported content, and a genuine parse failure all converge on the same value). A deliberate, disclosed design decision (§2.4's corollary, §13), not an oversight — listing two identifiers for one underlying fact would misleadingly suggest two independent causes to a reviewer.
- **No `tier`-aware filing decision — by design, not oversight.** Module 06 only computes and stores `tier`; it never reads or writes `processed_at`/`approved_by`/`approved_at`, never touches the filesystem, and never checks `suggested_destination` against a real directory listing. Deciding what to actually do about a low tier is entirely Module 07's job (`Module 06 Design.md` §4).
- **A disclosed housekeeping artifact, since resolved: `m06_perf_measurement.py` was left at the project root.** Written during the Release Audit's performance-measurement step (PCV Check 12), an initial attempt to remove it failed (`Operation not permitted` — the connected workspace folder did not permit file deletion once written, and the project owner declined to grant an exception when asked at that time). It was a real, reviewed, working script that touched no real `Database/`/`Runtime/` state and made no implementation change — inert leftover, not a pipeline component, for as long as it existed. Every prior module's own performance-measurement harness was ephemeral (never persisted); this was the first exception, and it was disclosed rather than silently left unexplained. **Update (2026-07-12):** the file has since been successfully removed from the project root in a later housekeeping pass; this limitation no longer applies.
- **`save_file_record()`'s O(N×M) storage read/write cost is now also Module 06's problem, not just Modules 02–05's.** `score_confidence_batch()` calls `save_file_record()` once per file processed — each call does a full read-modify-write of the entire cumulative `metadata_store.json`. Not a problem at current/expected v1 volumes (first flagged in Module 01's own `KNOWN_LIMITATIONS.md`, restated by each subsequent module as it became newly relevant) — flagged here for the same reason, per `Governance/ENGINEERING_STANDARD.md` §21's requirement that each additional module re-disclose a shared inherited cost. The real measurement taken this release (40.122 seconds for 75 files through the full six-module chain, +1.0% versus Module 05's own five-module baseline) shows no current bottleneck from this cost at v1 scale.
- **The UAT judgment-quality caveat applies to this module's upstream inputs, not to Module 06 itself.** The same session that implemented Module 06 also defined the "correct" expected Module 02/03 judgment answers used in UAT, which `Governance/ENGINEERING_STANDARD.md` §6.3 requires disclosing as a non-solved limitation. Module 06's own correctness, however, is verified by deterministic, re-derivable arithmetic over whatever category/fields it's handed — a stronger form of evidence than judgment-quality sampling, and not itself subject to this caveat, mirroring the same distinction Modules 04/05's own release records drew.
- **Deduction values, cap thresholds, and hard-floor mappings are hardcoded literals in `confidence.py`, not loaded dynamically at runtime.** A change to `Rules/Confidence Rules.md`'s weights requires a corresponding code change and a fresh regression run — the same pattern every earlier module's rule-consuming code already follows. Not a defect; a disclosed v1 architectural choice. `Rules/Confidence Rules.md`'s own "Tuning note" explicitly anticipates these weights being adjusted after real batches are observed.

## Intentional design decisions

- **No Provider layer at all — the narrowest architectural surface of any module built so far.** Every one of `Rules/Confidence Rules.md`'s nine deductions and four hard floors resolves to a direct field read or null-check against an already-computed upstream signal — confirmed through an explicit, dedicated architectural-decision section (`Module 06 Design.md` §2) that traced every single rule to its source field before concluding no provider is needed, not assumed from Module 04/05's own precedent alone.
- **Required/optional field taxonomy independently defined in `confidence.py`, cross-checked by a dedicated regression test — never imported from `pipeline/metadata.py`.** A module's internal constants are not part of another module's contract (`Governance/ARCHITECTURE_DECISIONS.md` decision 15); importing directly would create a hidden coupling a future `metadata.py` refactor could silently break. Deliberate, disclosed duplication, the same pattern Module 02/03's Provider classes already established.
- **Deduction caps enforced inside `compute_deductions()` itself, per category, in a fixed field order.** Fields within the cap are recorded at full nominal value; every field beyond the cap is still recorded — never hidden — at a value of `0`, so `confidence_breakdown` always lists every genuinely missing field while its sum never exceeds the cap. A deliberate, single-interpretation rule (Design Review finding M3), not left to an implementer's judgment.
- **`hard_floors_applied` lists every hard floor whose trigger condition was true, regardless of whether it changed the outcome.** A deliberate choice (§16) to keep the log line reproducible from the record's own fields alone, rather than requiring a second, hidden "did this floor matter" computation that would itself need auditing.
- **No cross-record batch state of any kind.** Unlike Module 04's collision-sensitive ordering or Module 05's within-batch naming-collision ordering, no record's *output value* in Module 06 depends on any other record in the same batch — batch order is preserved purely for log-ordering consistency with the rest of the pipeline.
