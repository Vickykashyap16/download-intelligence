# Known Limitations — Module 04 (Duplicate & Version Detection)

## Deployment model — read this first

Unlike Modules 02/03, there is no deployment-model caveat to state here. Module 04 has no provider, live or autonomous, of any kind (`Module 04 Design.md` §14) — every decision (hash equality, perceptual-hash distance, filename similarity, version-token/date comparison) is a deterministic computation over already-structured data. Module 04 is identically functional whether invoked during a live Claude session or an unattended/scheduled run; there is no judgment-dependent field that could degrade between the two.

## Known limitations

- **`main.py`'s `detect_duplicates()` filter has no `category is not None` precondition, unlike its sibling `extract()`.** Found during the first Independent Implementation Audit (finding L1). `extract()`'s filter requires `category is not None and category.value != "Unknown"` before calling Module 03; `detect_duplicates()`'s filter has no equivalent, because exact-duplicate detection is deliberately designed to run on every discovered record regardless of category (§3/§9) — gating on category here would silently skip files Module 04 is specifically designed to still cover. This is now explicitly documented inline in `main.py`'s own docstring (not a silent gap), and the Engine degrades gracefully in any case (membership tests against the near-duplicate/version-chain category sets simply return `False` for `category is None`). No functional impact; recorded here for completeness.
- **No automated regression test guards the deliberate duplication between `storage/database.py`'s `_normalize_for_index()` and `pipeline/duplicate_detector.py`'s `normalize_filename()`.** Found during the first Independent Implementation Audit (finding L2). The duplication itself is intentional and already approved — `storage/database.py` cannot import from `pipeline/duplicate_detector.py` without inverting this project's established storage→pipeline dependency direction, so this follows Module 03 Design.md §21's precedent of independent, convention-following duplication across a module boundary. Hand-checked consistent across 13 representative filenames during the audit, but there is no permanent test enforcing the two stay behaviorally identical as either function's regex evolves — a real, if currently unrealized, drift risk. **Not fixed as part of this release** — a small parametrized test asserting the two functions agree on a representative filename set is the smallest fix, deferred as non-blocking.
- **No backfill path for records that predate an index entry.** Found during the first Independent Implementation Audit (finding L3). A record only ever enters `Database/FileIndex/*.json` when that specific record passes through `update_indexes()`. Now that the H1 idempotency fix (post-freeze correction #2) means a genuinely-settled "nothing found" record correctly stops being reprocessed, this is the normal, permanent state for most files rather than a passing one — but there is no tooling to backfill indexes for records that existed before Module 04 shipped, or that were processed once and correctly never revisited. No current test or workflow depends on backfilling; named here for awareness ahead of any future migration.
- **`conflict_type`'s two values are verified at the Engine level, not independently against the persisted `action_log.jsonl` entry.** Found during the second Independent Implementation Audit (observation G4, Cosmetic). `detect_duplicates_batch()`'s log-writing code assigns `details["conflict_type"] = result.conflict_type` as a direct, zero-transformation passthrough, so the existing Engine-level test already exercises the only real logic involved — a log-level test would mostly re-confirm that Python assignment works. Lower-value than it sounds; not fixed, not blocking.
- **`save_file_record()`'s O(N×M) storage read/write cost is now also Module 04's problem, not just Module 02/03's.** `detect_duplicates_batch()` calls `save_file_record()` once per file processed (plus once more for the one disclosed side-effect record, when applicable) — each call does a full read-modify-write of the entire cumulative `metadata_store.json`. Not a problem at current/expected v1 volumes (first flagged in Module 01's own `KNOWN_LIMITATIONS.md`, restated by Module 02 and Module 03 as it became newly relevant to each) — flagged here for the same reason, per `Governance/ENGINEERING_STANDARD.md` §21's requirement that each additional module re-disclose a shared inherited cost.
- **`lookup_phash_matches()`/`lookup_name_matches()` are linear scans over `Database/FileIndex/*.json`, not indexed nearest-neighbor structures.** A deliberate, disclosed v1 trade-off (`Module 04 Design.md` §20/§27) — fine at v1 volumes (a single user's Downloads folder); a future SQLite migration (`ROADMAP.md` Version 2) would let `lookup_phash_matches()` use a real indexed structure (e.g. a BK-tree) instead, without requiring a contract change, since `storage/database.py`'s function signatures were written not to presume JSON specifically.
- **Two intentionally-identical files that aren't really "the same download twice" cannot be distinguished from an accidental re-download.** A disclosed limitation of Module 04's purely content-based approach (`Module 04 Design.md` §26) — no semantic understanding is attempted in v1, consistent with §14's no-provider scope decision. A future version could add a live-judgment provider if this proves to matter in practice; not built, and not currently planned, for v1.
- **Two exact-hash-duplicate records with different `category` are not actively flagged as a mismatch.** Hash equality is authoritative for `duplicate_of` regardless of category agreement (`Module 04 Design.md` §26) — Module 04 does not require or check category agreement for exact matches, only for near-duplicate/version-chain matches, where category scoping is load-bearing. A disclosed, accepted limitation, verified via M04-F03/M04-C03 at the integration level.
- **Version-chain candidacy still has two disclosed, bounded residual false-positive/false-negative risks after the 2026-07-23 PT-003 correction (`v1.0.1`, `Release/Module04/RELEASE_NOTES.md` post-freeze correction #5).** Before this correction, `_check_version_chain()` accepted any same-category filename pair scoring above the similarity threshold with no further corroboration — confirmed, via 2 real-world validation runs, to produce false "version chain" groupings between unrelated files sharing a generic template or tool-assigned name pattern (`PATTERN_TRACKER.md` PT-003). The correction requires a corroborating signal (an explicit version token, or an identical filename plus a size-proximity check) before accepting a candidate, and real-world re-validation confirmed all 4 known instances are resolved (`PT003_VALIDATION_REPORT.md`). Two residual risks remain, by design, not by oversight: (1) a genuine version chain using neither an explicit token nor an identical filename (e.g. an organic rename like "Meeting Notes.docx" → "Meeting Notes Updated.docx") is not detected — a real recall trade-off, not yet exercised by any real dataset either way; (2) two unrelated files that coincidentally share both a generic identical filename and a similar size can still form a false version-chain group — narrower than the pre-correction exposure, not eliminated by it. Neither residual has been observed in real data to date (searched directly in both validation datasets, neither found), but neither is proven absent from a broader real Downloads folder.

## Intentional design decisions

- **No Provider layer at all — a deliberate departure from Modules 02/03's pattern.** Every decision Module 04 makes is a computation over already-extracted, already-structured data; the "judgment" the original pre-build brief anticipated (occasional semantic override for "latest") is resolved not by asking Claude to adjudicate, but by flagging the disagreement (`version_conflict`) and deferring the actual decision to the human review path Modules 06/07 already provide. A future version could add a live-judgment provider if the deterministic signals prove insufficiently discriminating in practice — deferred, not pre-built speculatively (`Module 04 Design.md` §14/§27).
- **Image and Screenshot are strictly non-overlapping categories for both near-duplicate and version-chain grouping, with no equivalence mapping.** Confirmed as designed (`Module 04 Design.md` §9, F5) — a Screenshot never groups with a plain Image, and vice versa, matching how `category` is already treated as a strict partition everywhere else in this pipeline. A future version could introduce an equivalence mapping if real-world data shows this is too strict; that would be a new, separately-reviewed decision.
- **`duplicate_signals` records only the single best-scoring match per detection type, never a list of several simultaneous candidates.** An intentional v1 limitation (`Module 04 Design.md` §17, F4), disclosed so it is never mistaken for an oversight — surfacing multiple candidates to a human reviewer would require a schema change, not something this design speculatively builds for now.
- **§10's similarity threshold (90) and §11's max perceptual-hash distance (5) are configurable implementation parameters, not architectural constants, and are not yet relocated into a dedicated `Rules/Duplicate & Version Rules.md`.** Deferred until a later governance cleanup, tracked alongside Module 03's own still-pending `Rules/Metadata Rules.md` recommendation (`Module 04 Design.md` §15) — does not block this release.
- **Sequential-execution assumption.** Module 04 assumes strictly sequential pipeline execution — it never runs concurrently with Module 07, and does not re-validate that a record's `current_path` still exists on disk at processing time (`Module 04 Design.md` §23, F7) — the same non-defensive assumption every earlier module already makes about its own upstream dependencies.
