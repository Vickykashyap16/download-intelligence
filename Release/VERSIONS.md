## Versioning convention

- **Module Version** — standard semver (`MAJOR.MINOR.PATCH`), independent per module:
  - `PATCH` — bug fix within the module's current, frozen contract (e.g. the symlink fix would have been a patch release had this convention existed at the time).
  - `MINOR` — additive change that doesn't break the module's `MODULE_CONTRACT.md` (e.g. a new optional field, a new supported extension).
  - `MAJOR` — a change to the module's contract (its declared INPUT/OUTPUT/guarantees) that could require downstream modules to adapt.
- **Pipeline Version** — one number for the whole project's overall maturity. Bumped deliberately at meaningful milestones (e.g. "all 8 modules built and passing end-to-end" would justify `1.0.0`), not automatically derived from module versions.
- Modules evolve independently and at different paces. Illustrative example of where this goes (not the current state): `Pipeline 0.4.0` with `Module01 v1.0.0`, `Module02 v1.2.0`, `Module05 v0.9.0` — each module has iterated on its own schedule; the pipeline number reflects overall progress, not an average or maximum of the module numbers.
- **This file (`Release/VERSIONS.md`) is the single authoritative source of truth for every module's current version** (line 3, above) — it is updated on every release or re-release, including post-freeze patches. Each module's own `MODULE_STATUS.md` (in `Release/ModuleNN/`) is, by contrast, a **permanent, point-in-time release record**, generated once and never updated afterward — every existing instance (Modules 01–05) explicitly self-labels as "the permanent release record for Module NN," and every one of their `Pipeline Version` fields stays fixed at whatever the pipeline's version was on the date that module's record was created, never bumped as the pipeline version later advances. **Clarified 2026-07-11** (Module 01 post-freeze correction #1, see `Release/Module01/RELEASE_NOTES.md`): this line previously read "records its current version," which was imprecise and could be misread as implying MODULE_STATUS.md is kept live — it is not, and by design. If a module's version changes after its `MODULE_STATUS.md` was generated, this ledger (and the affected module's own dated `RELEASE_NOTES.md` addendum) is what reflects that; `MODULE_STATUS.md` correctly continues to describe the module's state as of its own generation date, exactly as `CHANGELOG.md` entries do for their own dates.
