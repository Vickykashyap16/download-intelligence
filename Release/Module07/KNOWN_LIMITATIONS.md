# Known Limitations — Module 07 (Preview, Approval & Execution)

## Deployment model — read this first

**Module 07 does not have a fully autonomous deployment path in v1.** `auto`-tier records execute with zero human input (G4) in either an interactive Claude-assisted session or an unattended scheduled run. `approval_required` and `review_required` records require an `ApprovalDecision` set to be supplied to `execute(decisions=...)` from somewhere — and how that set is actually produced (Open Decision OD-3: an interactive chat table, a generated markup file the user marks up and returns, or something else) is explicitly out of this module's architectural scope, by design (`Module 07 Design.md` §2/§26, `ARCHITECTURE_DECISIONS.md` decision 22). In practice today, that means a real `approval_required` batch can only be executed with a live Claude session in the loop, constructing the `decisions` set from a human's conversational responses — the same "supplied during a live session, not autonomous code" pattern already established for `classify(provider=...)`/`extract(provider=...)`'s judgment calls, but here for an entire category of record, not just a fallback path. There is currently no unattended-scheduled-run story for `approval_required`/`review_required` records at all; a scheduled run would file every `auto`-tier record and leave everything else untouched, exactly as designed (NG5), but this is a real, disclosed capability gap for anyone expecting Module 07 to run fully unattended end-to-end.

## Known limitations — blocking release certification

- **Integration Testing has not been performed.** No `Tests/Module 07 Integration Test Plan.md` exists; no harness has exercised the real Module 01→07 chain end-to-end (all three tiers, the adversarial `review_required`-forced-approval case, execution failure/partial-batch continuation, execution-time collision re-check, undo/rollback at both granularities, idempotency). Every prior module (01–06) completed this stage before its own Release Audit. See `RELEASE_AUDIT.md`.
- **UAT has not been performed.** No run has been executed against a real external Downloads-like folder and a real external destination-library-like folder, as `Module 07 Design.md` §0.3 requires as a measurable release criterion. See `RELEASE_AUDIT.md`.
- **No measured performance number exists** for the real Module 01→07 chain (`Module 07 Design.md` §0.3, `Governance/PIPELINE_CONTRACT_VERIFICATION.md` check 12). Module 06's own baseline (40.122s / 75 files) has no Module-07-inclusive successor measurement to compare against.

## Known limitations — non-blocking, disclosed

- **Four legacy stub functions remain in the codebase, fully dead.** `build_preview()`, `execute_approved()`, `log_rejected_edit()` (`src/pipeline/execution.py`) and `undo_batch()`'s own stub (`src/storage/runtime_io.py`) all predate the frozen design, were each individually superseded by a differently-named real implementation, and are confirmed (by grep, across production code and every test file) to have zero real call sites anywhere. Each still raises `NotImplementedError`. No functional risk — nothing calls them — but they are release-cleanup candidates, deliberately not removed during the composed-system architecture audit per explicit instruction to change no production code during a review-only pass. Recommended for removal at or before Module 07's actual release, once Integration Testing/UAT confirm nothing external depends on their presence (nothing should, but this is exactly the kind of assumption Integration Testing exists to verify).
- **`ARCHITECTURE_DECISIONS.md` decision 20's "Consequences" text is stale in one narrow respect.** It predicted `destination_root`'s config-reading code would be added in WP-2; it was actually added in WP-12, disclosed at the time. The architectural decision itself (the config key's shape and location) remains fully accurate. Per explicit instruction, this correction belongs in WP-13's governance-document update pass, not this release-engineering pass.
- **`main.py`'s `_EXECUTION_MOVE_ACTIONS`/`_EXECUTION_OUTCOME_ACTIONS` constants are a manually-kept-in-sync, unimported copy** of `execution.py`'s own private action-value vocabulary (not imported directly, since the source constant is underscore-private). No drift-guard regression test exists. If a future change ever renamed one of the three move-action strings, this CLI-summary code would silently miscount without a corresponding action-log entry actually being missed (the underlying execution behavior would remain correct; only the CLI's own summary counts could drift).
- **The `save_file_record()` O(N×M) cost is inherited, not newly introduced.** Every call in `ExecutionEngine.execute_file()`, `reconcile_batch()`'s `REPAIRED` branch, and `undo_single_action()` performs a full read-modify-write of the entire cumulative metadata store, the same disclosed cost every module since Module 02 carries (`ARCHITECTURE_DECISIONS.md` decision 11). Not expected to matter at v1 volumes; re-measurable once Integration Testing's performance step actually runs.
- **`Metadata & Log Schema.md`'s `undo` action `details` shape was undocumented until WP-13.** Now resolved (`{"reversed_action": ...}` documented as of this release-engineering pass) — listed here only because it was a real, dated gap during the implementation window, not because it remains one.

## Intentional design decisions (not limitations)

- **No trash folder, no backup copy for undo (G5).** Undo works by replaying the action log with `from`/`to` swapped — a deliberate simplification over maintaining a separate backup mechanism, consistent with G1's "never delete" guarantee making a backup unnecessary for the reversal case it exists to cover.
- **`review_required` records are never auto-revisited (NG2).** They stay flagged in the preview and the day's Daily Summary; no automatic re-queue exists in v1. A deliberate scope boundary, not a gap — automatic re-queue would require some signal that circumstances changed, which nothing in this pipeline currently produces.
- **No protection against concurrent external changes to the destination library beyond one synchronous re-check immediately before each move (NG3).** No distributed lock, no whole-batch transaction — appropriate for a single-user, single-batch-at-a-time tool (`ARCHITECTURE_DECISIONS.md` decision 17), not a gap for a different deployment model this project doesn't target.
- **`Database/Learning/User Corrections.json` is passive-capture only (NG6).** Every edit/rejection is recorded; nothing reads it back or acts on it automatically in v1. A deliberate, disclosed scope boundary for a future learning/personalization feature, not an oversight.
- **Undo is not guaranteed to succeed in every case (NG7).** A `reversible = false` action is surfaced for human review rather than silently attempted; an undo cannot restore a location something outside this pipeline has since altered. Both are inherent to "the log is the only undo mechanism," not implementation gaps.
