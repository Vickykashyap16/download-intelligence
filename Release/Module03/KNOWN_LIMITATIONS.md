# Known Limitations — Module 03 (Metadata Extraction)

## Deployment model — read this first

**No autonomous production `MetadataExtractionProvider` exists.** `ClaudeLiveExtractor` — v1's only concrete provider — requires a live, agent-driven Claude session; its `extract()` method raises `NotImplementedError` if invoked any other way, by design. Running Module 03 outside a live session (e.g. a scheduled/unattended `python -m src.main` invocation) is **safe** — every judgment-dependent field gracefully falls back to `null` (`fallback_reason: "provider_exception"`) rather than crashing or silently guessing — but it is **not functional** for judgment-dependent extraction. Building an autonomous provider (a networked LLM API call, a local model, or a rule-based fallback extractor) is explicitly out of scope for v1 and is the single largest gap between "feature complete" and "fully autonomous" — the same posture Module 02's `KNOWN_LIMITATIONS.md` states for `ClassificationProvider`.

## Known limitations

- **`Rules/Naming Rules.md`'s Contract/Audio templates don't cleanly match Module 03's actual taxonomy field names.** Found during the independent release audit (`Release/Module03/RELEASE_AUDIT.md`, finding F5). Contract's naming template placeholder is `{PartyName}`, but Module 03's real taxonomy field (`REQUIRED_FIELDS`/`OPTIONAL_FIELDS` in `src/pipeline/metadata.py`) is `counterparty`. Audio's documented fallback template is `{Description}_{Date}`, but Audio's taxonomy has no `description` field at all — only `track_title`, `artist`, `duration`, `recording_date`. Unlike `Rules/Confidence Rules.md` (regression-tested against the taxonomy — see the Implementation Audit's F2 fix), nothing currently checks `Naming Rules.md`'s placeholders against real field names, so this class of mismatch isn't caught automatically.
  **Deliberately not fixed as part of Module 03's release:** `Rules/Naming Rules.md` is explicitly marked "a draft to react to, not a locked spec," and Module 05 (Naming & Destination) — the module that actually consumes these templates — doesn't exist yet. Changing the templates now, before Module 05's design resolves how placeholders map to real taxonomy fields, risks guessing at a decision that belongs to that module's design phase. **This will be addressed during Module 05's design** (either by correcting `Naming Rules.md`'s placeholder names to match the taxonomy, or by having Module 05 define an explicit placeholder→field mapping) — flagged here now so it isn't rediscovered cold later.
  No functional impact today: no code reads `Naming Rules.md` programmatically yet, and Module 03 itself never touches naming or destinations.
- **Video's `content_date` and `duration` are always `null` in v1.** No video-tag/duration-reading library was approved during design (`mutagen` covers audio only; a video-specific dependency was explicitly deferred, `Module 03 Design.md` §16/§27). Video's naming date slot will need to fall back to `FileRecord.modified_at` (a filesystem timestamp, not the video's actual recording date) once Module 05 exists — a real, disclosed limitation for phone-recorded videos downloaded well after they were shot, first raised in `Module 03 Design Review.md`'s F4 and resolved there by promoting it to this kind of explicit, named limitation rather than an implicit table note.
- **Only 2 of the 4 designed timestamp-source tiers are actually implemented in v1.** The four-tier hierarchy (embedded content metadata → EXIF → container/format metadata → filesystem timestamps, `Module 03 Design.md` §9A) is fully specified, but only tier 1 (Audio, via `mutagen`) and tier 2 (Image/Screenshot, via EXIF) have a real source in v1. Tier 3 (container/format metadata for categories like Video) has no implementation yet — no functional impact today since no category currently expects a tier-3 value, but a future category added to the taxonomy would need this tier built, not assumed to already exist. Tier 4 (filesystem timestamps) is deliberately never written by Module 03 at all — reserved for Module 05's own naming-time fallback, not a gap.
- **`MetadataExtractionProvider` is unused scaffolding for 4 of 11 non-Unknown categories.** Archive, Application, Video, and Audio are fully deterministic in v1 (`_DETERMINISTIC_ONLY_CATEGORIES`) — for these, the full Engine/Provider three-layer architecture exists but the Provider is never actually called. This is a pre-accepted, explicitly named trade-off (`Module 03 Design Review.md` F8, `Module 03 Design.md` §27): consistency and future extensibility (a category could move from deterministic to judgment-dependent later without an architecture change) outweigh the cost of an unused interface for categories that don't need it today. Mirrors the identical trade-off Module 02's own design accepted for its one-provider ratio.
- **Judgment quality has only been UAT-tested against one batch, and that validation was self-graded.** The single UAT run (`Runtime/UAT/Module03_UAT_2026-07-06_100928/`) covered 9 judgment-dependent extraction calls across a deliberately varied but still small set of categories, graded by the same agent that implemented the extractor and authored the test files' expected answers — not an independent rater. The run does meaningfully stress-test the redaction safety net specifically (a deliberately non-self-censoring "live" answer was fed for Bank Statement's account number, and the Engine's structural check caught it regardless) — but broader real-world Downloads folders (larger volume, more categories, more languages, more file conditions), ideally reviewed by someone other than the implementer, haven't been exercised yet. Same caveat standard as Module 02's own `KNOWN_LIMITATIONS.md`.
- **`save_file_record()`'s O(N×M) storage read/write cost is now also Module 03's problem, not just Module 02's.** `extract_metadata_batch()` calls `save_file_record()` once per file, same as `classify_batch()` — each call does a full read-modify-write of the entire cumulative `metadata_store.json`. Not a problem at current/expected v1 volumes (first flagged as a future concern in Module 01's own `KNOWN_LIMITATIONS.md`, restated as Module 02's problem in its release audit F9, and now genuinely a third module's cost too) — flagged here for the same reason: it will be the first thing worth optimizing (e.g. one load + one write per batch instead of per file) if store size or batch frequency grows materially.
- **`MetadataExtractionRequest.mime_type` is never populated by any Engine code path.** A pre-accepted, disclosed trade-off (`Module 03 Design.md` §27) — the same unused-field pattern already present in Module 02's `ClassificationRequest.mime_type`. Not new dead code; a future provider that needs MIME-type-aware behavior would need this wired through, not assumed already present.

## Intentional design decisions

- **Deliberately not code-shared with Module 02.** `MetadataExtractionEngine`/`MetadataExtractionProvider`/`ProviderResponse`/`ProviderMetadata`/`_sanitize_error()` are all Module 03's own classes/functions, structurally near-identical to Module 02's equivalents but never imported from `classification.py` — a deliberate convention-following choice (`Module 03 Design.md` §21), not duplication by oversight. Verified: zero cross-imports beyond the shared `Category` enum.
- **The required/optional/prohibited taxonomy (§7) was explicitly confirmed by the project owner before being treated as settled**, not silently inferred from `Rules/Naming Rules.md`'s templates the way the first design draft built it — the crux of `Module 03 Design Review.md`'s F1, resolved by making §7 a flagged, pending decision rather than a quiet assumption, and confirmed before implementation began.
- **The Bank Statement `account_last4` redaction rule is scoped to that one field, not generalized across every category's string fields.** `Module 03 Design.md` §18 gives a principled reason: `account_last4` is the only field in the entire taxonomy whose defined purpose is to be a truncated-safe view of a value that is itself prohibited. A blanket digit-pattern scanner across every field was considered and rejected as actively harmful (it would misfire on fields meant to hold long numbers, like `invoice_number`) — an over-engineering trap avoided deliberately, not an oversight (`Module 03 Design Review 2.md`'s verification of F3).
- **`category` stays a plain, already-validated `Category` enum at the point Module 03 reads it** — Module 03 trusts Module 02's own contract that `category` is never left as a raw string on a processed record, and never re-validates it itself; this is a deliberate reliance on an upstream contract already tested at that layer, not a gap in Module 03's own validation.
- **No retry, no secondary-provider chain in v1.** Any provider failure (unavailable, exception) falls straight to `null` for every judgment-dependent field of that call — simple, auditable, and consistent with Module 02's identical no-retry decision, rather than inventing new retry/confidence logic at this layer.
