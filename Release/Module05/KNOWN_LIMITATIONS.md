# Known Limitations — Module 05 (Naming & Destination)

## Deployment model — read this first

Like Module 04, there is no deployment-model caveat to state here. Module 05 has no provider, live or autonomous, of any kind (`Module 05 Design.md` §17) — every decision (template selection, field filling from already-extracted metadata, string sanitization, within-batch collision suffixing, category-to-folder lookup, override precedence) is a deterministic computation over already-structured data. Module 05 is identically functional whether invoked during a live Claude session or an unattended/scheduled run; there is no judgment-dependent field that could degrade between the two.

## Known limitations

- **Real-filesystem collision detection against the destination library is explicitly out of scope.** A deliberate, disclosed v1 boundary (`Module 05 Design.md` §9/§13), not a gap. Module 05 resolves collisions only *within* the current batch (`_2`/`_3` suffixing, keyed on `(destination, name)`); it never reads the real destination folder's directory listing and never learns the destination library's absolute root path. Module 07 is responsible for the authoritative, real-filesystem check at execution time, since only it has a guaranteed-current view of the destination at the moment a file is actually moved.
- **No `tier` awareness — by design, not oversight.** `resolve_destination()` takes no `tier` parameter; `tier` is Module 06's output, and Module 06 runs after Module 05 in the strict linear pipeline. Module 05 always computes a real `suggested_destination` for every eligible record, including records that will eventually turn out to be `review_required`. The "don't actually move a `review_required` file" rule (`Rules/Folder Rules.md`) is deferred to Module 07's execution-time gate, which will read both `suggested_destination` (this module's output) and `tier` (Module 06's output) together (`Module 05 Design.md` §8). Module 07's own design is responsible for implementing this gate; it is out of scope for Module 05's own release.
- **`_SIMPLE_TEMPLATES`'s type annotation doesn't accurately describe `"literal"` entries.** Found during the second Independent Implementation Audit (observation C1, Cosmetic). `_SIMPLE_TEMPLATES: Dict[Category, List[Tuple[str, Optional[Tuple[str, str]]]]]` states every spec entry's second element is `Optional[Tuple[str, str]]`, but a `("literal", "Statement")`-style entry's second element is a plain `str`. Zero runtime impact (Python doesn't enforce this), and the logic that consumes it handles both shapes correctly by branching on `kind` first. **Not fixed as part of this release** — widening the annotation (e.g. `Tuple[str, Union[str, Tuple[str, str]]]`) or introducing a small `TemplateSpec` type alias is the smallest fix, deferred as non-blocking, purely cosmetic technical debt.
- **`save_file_record()`'s O(N×M) storage read/write cost is now also Module 05's problem, not just Modules 02–04's.** `suggest_naming_and_destination_batch()` calls `save_file_record()` once per file processed — each call does a full read-modify-write of the entire cumulative `metadata_store.json`. Not a problem at current/expected v1 volumes (first flagged in Module 01's own `KNOWN_LIMITATIONS.md`, restated by each subsequent module as it became newly relevant) — flagged here for the same reason, per `Governance/ENGINEERING_STANDARD.md` §21's requirement that each additional module re-disclose a shared inherited cost.
- **Naming templates and folder routing are read from `Rules/Naming Rules.md`/`Rules/Folder Rules.md` as hardcoded literals in `naming.py`, not loaded dynamically at runtime.** A change to either rules document requires a corresponding code change and a fresh regression run — the same pattern every earlier module's rule-consuming code already follows (`Rules/Confidence Rules.md`'s deduction table is similarly hardcoded in anticipation of Module 06). Not a defect; a disclosed v1 architectural choice, consistent with the rest of the pipeline.
- **The UAT judgment-quality caveat applies to this module's upstream inputs, not to Module 05 itself.** The same session that implemented Module 05 also defined the "correct" expected Module 02/03 judgment answers used in UAT, which `Governance/ENGINEERING_STANDARD.md` §6.3 requires disclosing as a non-solved limitation. Module 05's own correctness, however, is verified by deterministic, re-derivable computation over whatever category/fields it's handed — a stronger form of evidence than judgment-quality sampling, and not itself subject to this caveat, mirroring the same distinction Module 04's own release record drew.

## Intentional design decisions

- **No Provider layer at all — the same architectural departure Module 04 established for itself.** Every decision Module 05 makes is a computation over already-extracted, already-structured data; no judgment call is required, confirmed through an explicit decision review that examined every candidate case (Invoice's `{DocSubtype}`, Resume's `{VersionOrDate}` priority, Archive/Video's missing date fields, Audio's broken fallback) and found each had a satisfying deterministic rule once examined, leaving no remaining case for a provider to fill (`Module 05 Design.md` §17).
- **Naive per-underscore-segment Title_Case with no exceptions list.** `str.capitalize()`'s exact behavior is applied uniformly — e.g. `"NDA"` → `"Nda"` — an explicitly accepted cosmetic cost (`Module 05 Design.md` §12), not a defect. A future version could add an acronym-preservation exceptions list if this proves to matter in practice; not built, and not currently planned, for v1.
- **Longest-segment truncation, not a fixed per-field character budget.** The ~80-character soft cap is enforced by iteratively truncating whichever segment is currently longest, preserving the template's overall structure and every other field intact, rather than allocating a fixed budget per field up front (`Module 05 Design.md` §12, corrected to be iterative by the first Implementation Audit's M2 finding). A deliberate, simple rule chosen over a more elaborate proportional-allocation scheme.
- **`naming_signals.fields_fell_back` records only the affected field names, never the fallback value itself.** Consistent with `classification_signals`/`duplicate_signals`'s existing pattern — the signal exists for Module 06's deduction arithmetic to consume, not as a full audit trail of what value was substituted. The substituted value is always recoverable from `suggested_name` itself if needed.
- **Collision-suffix scope is `(destination, name)`, not `name` alone.** A deliberate design choice, not an oversight (`Module 05 Design.md` §13) — two records that compute the identical base name but route to different destinations (e.g. one superseded, one not) are not really "the same slot," so neither should be forced into an unnecessary `_2` suffix. Confirmed as a genuine, emergent case during real Integration Testing (M05-COLL02), not merely a synthetic unit-test scenario.
