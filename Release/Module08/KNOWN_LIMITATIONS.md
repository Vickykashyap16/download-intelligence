# Known Limitations — Module 08 (Logging & Reporting)

## Deployment model — read this first

**Module 08 has no disclosed gap between interactive and autonomous/unattended operation — the only module in the pipeline of which this is true.** It has no Provider of any kind and no human-decision dependency anywhere in scope (§2); every report type is a pure, deterministic computation over already-structured data. `report()` runs identically whether invoked during a live Claude session or from an unattended scheduled task — there is no judgment-quality gap to disclose, unlike Modules 02/03 (no autonomous provider exists yet) or Module 07 (`approval_required`/`review_required` records need a human-supplied decision set from somewhere).

## Known limitations — release-blocking gaps, all resolved (2026-07-20)

- **Integration Testing** was performed against the real Module 01→08 chain end-to-end (functional, corrupted/partial action-log, mixed-batch, multi-day rollup, cross-module contracts, performance, regression) — zero findings. See `Tests/Module 08 Integration Test Plan.md`.
- **UAT** was performed against a real external Downloads-like folder and a real external destination-library-like folder, as `Module 08 Design.md` §0.3 requires — PASS WITH RECOMMENDATIONS, two disclosed non-blocking observations, zero defects. See `Tests/Module 08 UAT Plan.md`.
- **A measured performance number now exists**, taken fresh during the Release Audit: 2,000 records / 14,000 log lines, 0.5436s total across all four report types — no crash, no unreasonable slowdown, consistent with the design's own disclosed O(records)/O(log lines) cost model. See `TEST_RESULTS.md`.

## Known limitations — non-blocking, disclosed

- **Post-execution temp-directory cleanup could not be fully verified during UAT, in this specific execution environment.** A real `PermissionError` occurred during `execute_batch()`'s (Module 07's) post-run `shutil.rmtree()` cleanup call. Investigated directly: reproduced identically on a brand-new, completely unrelated directory (ruling out anything file- or batch-specific); the session's mount confirmed FUSE-backed via `mount` output; the identical `clear_batch_temp()` function succeeded without issue earlier in the same session against an isolated `/tmp`-based Integration Testing harness. The project owner reviewed this evidence directly and did not classify it as a production code defect — recorded as an environment-specific observation, not a Module 07 or Module 08 implementation gap. **No production code was modified as a result.** Listed here (rather than only under Module 07's own known limitations) because it was surfaced during Module 08's own UAT and directly affected that UAT's ability to fully verify post-run cleanup.
- **The Weekly Summary's top-line bullet summary shows all-zero totals on a day with real same-day activity**, because the current, still-open day is correctly excluded from week-to-date rollup (`ARCHITECTURE_DECISIONS.md` decision 27) — genuinely correct, designed behavior, not a defect. The "Not yet closed" status is disclosed in the Days table below the summary line but not reiterated next to the top-line zeros, which could read as confusing to a first-time user glancing only at the summary line. Recorded as a UAT observation/recommendation for a possible future documentation or UX pass — no code change indicated or authorized as part of this release.
- **Seven accepted Low findings across WP-1 through WP-5**, all non-blocking, all disclosed at the time and re-confirmed non-blocking during the Release Audit: private-attribute reach into `runtime_io._RUNTIME_REPORTS_PATH` (WP-2's closed-day check, WP-4's missing-day existence check) rather than a new public path accessor — a disclosed, deliberate choice to keep each work package's file-touch footprint minimal, not an oversight; a malformed-line disclosure line's wording differing slightly from the design's own illustrative example phrasing, same fact conveyed; the Duplicate Report's "Kept" category folding three distinct real states (still pending, explicitly rejected, reversed via undo) into one label, a necessary consequence of the frozen design's own fixed three-category vocabulary (`Runtime/Reports/README.md`'s "archived, kept, overridden by the user"), not a defect; the Markdown-bullet-parsing coupling between `generate_weekly_summary()` and `generate_daily_summary()`'s own exact rendering format — an intentional, design-level coupling per §9's own "avoid two parallel aggregation paths that could silently disagree" reasoning, not enforced at the type level, worth naming for future maintainers; one new-filter/marker function accessing `entry["timestamp"]` by direct indexing rather than `.get()` (WP-1) — unreachable under `append_action_log()`'s own real, unconditional behavior, a disclosed inconsistency against a sibling function's more defensive style, not a live risk.
- **Two Recommended and two Informational items from the WP-1–6 integration audit, deferred as maintenance:** a duplicated daily-summary path helper and a duplicated latest-sibling lookup helper (`_daily_summary_path()`/`_daily_summary_file_path()`, `_render_version_archived_detail()`/`_find_latest_sibling_name()`) — each deliberately kept separate across WP-2/WP-3/WP-4 to keep every work package's diff isolated from already-audited code, a real, disclosed duplication cost, not an oversight; stale wording in `Module 08 Design.md` §12 (a pre-implementation phrasing that no longer perfectly describes the shipped implementation's exact disambiguation mechanism, substance unaffected); accepted mypy technical debt (no new type-checking gap introduced by Module 08 beyond what the project already carries). None required a code change for this release; all remain candidates for a future housekeeping pass.

## Intentional design decisions (not limitations)

- **No dated snapshots for the Duplicate Report or Storage Report (decision 25).** Both are single, continuously-updated current-state files, overwritten in place on every call — a deliberate choice, since neither report type has a natural period concept the way Daily/Weekly Summary do, and no reader in this pipeline ever depends on a historical Duplicate/Storage Report snapshot.
- **No retroactive correction of an already-closed Daily Summary if underlying data changes afterward (NG2).** A closed day's report is treated as a historical record once written, consistent with this project's own "never rewrite history" documentation convention. Open Decision OD-2 remains deliberately deferred — a real, disclosed open question, not silently resolved.
- **Storage Report reflects file size at discovery time, never a live filesystem re-measurement (decision 29).** `size_bytes` is set once by Module 01 and never modified by any later module; no code path in this pipeline currently causes a filed file's real size to diverge from its recorded value, so this is not expected to matter in practice — a future, genuinely-motivated need for live measurement would be its own new, explicit decision.
- **No protection against `action_log.jsonl` growing unboundedly (NG5).** Log rotation/archival remains an explicitly deferred `Version 2` concern (`ARCHITECTURE_DECISIONS.md` decision 10) — Module 08 reads the log as it exists, however large it has grown, with a measured, disclosed cost for doing so (`TEST_RESULTS.md`'s performance section).
- **No learning or personalization from report content (NG6).** `Database/Learning/User Corrections.json` is never read by any report generator in v1 — a deliberate, disclosed scope boundary for a possible future feature, not an oversight (confirmed via grep, zero references anywhere in `reporting.py`).
