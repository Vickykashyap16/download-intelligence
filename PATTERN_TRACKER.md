## 1. How to read this document

**Classification** (per this phase's instruction) — exactly one of:
- **Confirmed Pattern** — observed consistently across 2 or more independent runs/datasets.
- **Single Observation** — observed in only one run so far, regardless of how many instances occurred within that run (9/9 files in one run is still one data point about the *pipeline*, not nine).
- **Environment Issue** — root cause is attributable to the validation execution environment or tooling, not the released pipeline's own code, rules, or data model.
- **User Expectation** — the observed behavior matches already-documented, designed, and previously-disclosed project behavior. Not an open question; recorded here only for cross-run traceability.

**Confidence** is a separate axis from Classification — it answers "how well-characterized is this finding's root cause," not "how many times has it recurred." A finding can be a Single Observation with High confidence (root cause directly confirmed against real code and real measured data, fully reproducible in principle) or a Confirmed Pattern with only Medium confidence (recurs consistently but the exact trigger isn't fully isolated yet). Scale: **High** (root cause directly verified against code/data, not inferred) · **Medium** (root cause plausible and partially verified, some ambiguity remains) · **Low** (root cause not yet investigated beyond the symptom).

**Evidence Count** is reported as *instances / opportunities* within each run, then summed — never a bare percentage, per `METRICS_DEFINITION.md`'s own standing rule that a raw count must always accompany any derived rate.

## 3. Finding detail

### PT-001 — Validation sandbox / real-folder access gap
**Root cause (directly confirmed, not inferred):** `mount` and `df` output from inside the code-execution sandbox showed `Desktop`, `Download Intelligence`, `outputs`, and `uploads` as real FUSE mounts; `Downloads` had no corresponding mount entry at all, confirmed by repeated `ls`/`find` returning empty against a folder the file-reading tools could see fully. **Why Confidence is High despite only 1 occurrence:** the mechanism was verified directly against system-level evidence (mount tables), not deduced from a symptom that could have another explanation. **Why Classification is Environment Issue, not a pipeline finding:** nothing in `src/` caused this — every module's contract already assumes direct filesystem access without specifying which access model, and this gap sits entirely in how this session's tooling grants folder access, not in Downloads Intelligence's own code.

### PT-002 — Screenshot misclassification of real non-screenshot images
**Root cause (directly confirmed, 2 independent runs):** `Rules/Classification Rules.md` Pass 1 routes every image-family extension through the Screenshot-vs-Image split only, with no route to `Document`/`Image` recovery once "no camera EXIF" fires. Real EXIF data measured directly (Pillow): Run 002, zero camera-metadata tags on all 9 files; Run 003, zero camera-metadata tags on 18 of 19 real images, with the 1 exception (genuine camera EXIF present) correctly classifying `Image` — a built-in contrast case that sharpens confidence the EXIF-absence signal, specifically, is the trigger. Fully deterministic — confirmed to not depend on live-provider availability, since `.jpg`/`.png` files never reach Module 02's provider-dependent deep pass at all. **Why Confidence is High:** every link in the causal chain (rule text → code implementation → real measured EXIF absence → observed output) was independently verified in both runs, not assumed, and Run 003 additionally visually confirmed 6 of its misclassified files really were non-screenshot content (a real photo, a product photo, a marketing banner, an AI-generated portrait, and the two files in PT-003's false version-pair). **Why Classification is now Confirmed Pattern:** 2 independent runs, 2 non-overlapping datasets, and — critically — Run 003's content diversity (personal photos, product/marketing graphics, AI-generated images) is materially wider than Run 002's single content shape (scanned academic documents), and the identical, precisely-understood mechanism reproduced across all of it. The real-world surface is now understood to be broader than "scanned documents saved as images": it also covers any real photo shared through a channel that strips EXIF (e.g. WhatsApp), which is a very common sharing pattern.

**CLOSED / IMPLEMENTED / VALIDATED / MERGED — 2026-07-23.** Full disposition:
- **Design:** `Build-out/02 Classification/Module 02 Post-Freeze Design Correction — PT-002.md` (Alternative A — remove the independent "no camera EXIF" trigger; Screenshot now requires filename-marker OR resolution-match only; no architectural change; disclosed trade-off named and bounded).
- **Implementation:** `src/pipeline/classification.py`'s `classify_screenshot_or_image()` corrected to the 2-condition form; `Rules/Classification Rules.md` updated to match. Scope held exactly to Module 02, no contract change, no other module touched (confirmed by diff).

### PT-003 — False-positive version-chain grouping
**Root cause — now isolated to a specific mechanism for the Run 003 instance (2026-07-20, root-cause investigation authorized under Project Phase 4, read-only code/data analysis only, no code modified):**

Direct empirical test against the real Run 003 files (`Runtime/Validation/RUN003_staging/image (2).png`, `image (42).png`, still present, unexecuted) and the real, unmodified `src/pipeline/duplicate_detector.py`/`src/core/hashing.py`/`src/storage/database.py` code:
- `normalize_filename()` on both files yields `"image (2)"` and `"image (42)"` — neither matches `_VERSION_TOKEN_SUFFIX` (which requires a literal `v`/`V` prefix before the digits, e.g. `_v2`), so no version-token stripping occurs.
- `rapidfuzz.fuzz.ratio("image (2)", "image (42)")` = **94.7**, directly measured — above `_NAME_SIMILARITY_THRESHOLD = 90` (`src/storage/database.py`). This alone is sufficient for `lookup_name_matches()` to return the pairing as a version-chain candidate inside `_check_version_chain()`.
- `parse_version_token()` returns `None` for both filenames — the generic `"(N)"` suffix is not a recognized version token at all; it plays no role in why these two were paired, only in what would happen afterward (rank assignment).
- Perceptual-hash distance, measured directly (`perceptual_hash()`/`hamming_distance()`): **36** — far above `_MAX_PHASH_DISTANCE = 5`. This independently confirms the near-duplicate/visual-similarity path (`_check_near_duplicate()`) correctly found no relationship and was **not** involved in the false pairing at all.

**Conclusion:** the Run 003 false positive has one single, fully identified cause: `_check_version_chain()`'s filename-similarity check (`fuzz.ratio()` against `_NAME_SIMILARITY_THRESHOLD = 90`) treats two filenames whose only shared structure is a generic, tool-assigned `"image (N)"` pattern as similar enough to be version-chain candidates, purely as edit-distance-close plain text — even though they share no version token, no date, and (confirmed by phash) no visual content. This is a distinct, now-precisely-specified mechanism from Run 002's own plausible-but-still-unconfirmed filename-token theory ("10th"/"12th" parsing) — both are filed under the same finding because both are false positives from Module 04's filename-similarity path broadly, not because they share one identical trigger.

**Why Confidence is now High for the Run 003 mechanism specifically:** every step of the causal chain (real filenames → `normalize_filename()` output → real `fuzz.ratio()` score → comparison against the real threshold constant → real phash distance ruling out the alternative path) was directly measured against real code and real data, not inferred from the observed outcome alone. Confidence for Run 002's own trigger remains at its prior, unconfirmed level — this update applies specifically to the newly-investigated Run 003 mechanism. **Classification remains Confirmed Pattern**, unchanged by this investigation (2 independent runs, 2 non-overlapping datasets, both real, incorrect version-chain groupings, zero true positives across either run) — root-cause isolation sharpens *why*, not *whether*, the pattern is real.

**Scope note, per explicit instruction:** this is root-cause investigation only. No design alternative, selected approach, or implementation is proposed for PT-003 here — `VERSION_091_IMPLEMENTATION_PLAN.md` §3 already named root-cause isolation as the precondition for PT-003's own future Design phase to begin; that phase has not been authorized.

**Update, 2026-07-23 — Run 002's mechanism now also directly isolated.** As part of PT-003's Design-phase evidence verification (`Build-out/04 Duplicate & Version Detection/Module 04 Post-Freeze Design Correction — PT-003.md`), the same direct-measurement method was applied to Run 002's 3 false-pairing instances (`Desktop/Validation Sample/`, files still present, unexecuted): `fuzz.ratio()` on the normalized filenames scored 93.3 (`"mark sheet 10th"` vs `"mark sheet 12th"`), 91.3 (`"...1st semester"` vs `"...4th semester"`), and 91.3 (`"...2nd semester"` vs `"...3rd semester"`) — all above the 90 threshold; `parse_version_token()` returned `None` for all 6 filenames; perceptual-hash distance measured 26, 8, and 6 respectively — all above `_MAX_PHASH_DISTANCE = 5` (the third pair's distance of 6 is a close call, but still confirmed on the correct side of the threshold), independently ruling out the near-duplicate path for all three. **Root cause for Run 002 is now identical in mechanism to Run 003's**: `_check_version_chain()`'s pure filename-similarity check, with no corroborating signal, scoring same-template/generic-suffix filenames above threshold despite no genuine version relationship. Confidence for Run 002's own trigger upgraded from unconfirmed to **High**, matching Run 003. Both runs' false positives are now confirmed to share one single, precisely-understood mechanism, not two separately-plausible ones.

**CLOSED / IMPLEMENTED / VALIDATED / MERGED — 2026-07-23.** Full disposition:

Selected design: `_check_version_chain()`'s candidacy requires, in addition to the existing `fuzz.ratio() >= 90` threshold, either an explicit version token on at least one side, or an identical normalized filename corroborated by a size-proximity check (`size_bytes`, already-captured metadata, no new extraction). No architectural change, no Module 04 contract change (PATCH-level); two disclosed, bounded residual risks carried forward (R1 — non-tokened, non-identical genuine renames not detected; R6 — a coincidental identical-generic-name-plus-similar-size false positive remains theoretically possible, not observed in either real dataset).

### PT-009 — Batch-cleanup failure on FUSE-mounted validation folder (environment)

Same category as PT-001 (validation-environment-specific, not a pipeline defect) — kept as a separate finding since the specific failing operation (temp-directory cleanup after successful execution) and its real-world implication (a fully-successful batch could still surface as an uncaught exception at the very end) are distinct from PT-001's blocking-before-any-work failure mode.

## 4. PT-002 and PT-003 are now Confirmed Patterns — what's needed next is characterization, not confirmation
