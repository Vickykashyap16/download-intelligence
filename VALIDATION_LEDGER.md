# Validation Ledger — Version 0.9.0 Real-World Validation

**Status:** Living document — appended after every validation run, never rewritten (mirrors `Release/VERSIONS.md`'s own append-only, never-rewrite-history convention).
**Governed by:** `REAL_WORLD_VALIDATION_PLAN.md` (process, error taxonomy, failure format), `METRICS_DEFINITION.md` (metrics), `DATASET_GUIDELINES.md` (privacy rules applied below), `VERSION_09_PLAN.md` (the milestone this ledger evidences).

## How to read this ledger

Each run gets one entry. An entry is never edited after the fact except to append a resolution note if a later run changes its status — corrections are new, dated additions, per this project's standing documentation convention. Findings are classified per `REAL_WORLD_VALIDATION_PLAN.md` §7 (error taxonomy) and, per this phase's explicit instruction, also tagged with one of six dispositions: **User expectation** / **Documentation issue** / **Rule improvement** / **Model improvement** / **Architecture defect** / **Implementation defect**. No fix is recommended from a single occurrence unless the finding is Critical-severity — recurring patterns across multiple runs are what earn a place in the eventual v0.9.1 roadmap.

## Run 001 — 2026-07-20 — Blocked before pipeline execution

**Dataset:** Real `~/Downloads` folder (`/Users/vicky/Downloads`), newly connected this session.
**Scope attempted:** A representative sample from `Downloads/Documents/Personal/` — mark sheets, degree certificate, and payslips (PAN/Aadhaar ID scans explicitly excluded from scope per project owner direction, `DATASET_GUIDELINES.md` §3).
**Outcome:** **Blocked — no pipeline stage executed. Zero metrics collected.**

### What was attempted

1. The real Downloads folder was connected and confirmed reachable by file-reading tools (Read/Grep/Glob operate directly on the host path `/Users/vicky/Downloads`).
2. A preliminary survey (Glob) found ~2,900 files, almost entirely nested 2+ levels deep inside large bulk folders — commercial contact-list spreadsheets (real estate, government employees, stock traders, etc.) and one ML course — plus one genuinely personal folder, `Documents/Personal/`, containing ID documents, academic records, and payslips.
3. Per project owner direction, scope was narrowed to exclude the bulk third-party contact databases and the two highest-sensitivity ID scans (PAN, Aadhaar), leaving 12 files (mark sheets, degree, 3 payslips) as the intended sample.
4. Attempting to execute the real pipeline (`src/main.py`) against this sample required the code-execution sandbox (`mcp__workspace__bash`) to see the same files the file-reading tools could see. It does not: `mount` confirms `Desktop`, `Download Intelligence`, `outputs`, and `uploads` are all FUSE-mounted into the bash sandbox; **`Downloads` is not** — it never received a working mount, and the mount point exists but is empty from the sandbox's perspective, confirmed directly (not inferred) via `mount`, `df -h`, and repeated `ls`/`find` calls all returning empty.
5. A binary-safe copy path was considered (read the file via the host-side Read tool, write it via the host-side Write tool into the `outputs` folder, which *is* bash-mounted) and rejected: the Read tool returns images/PDFs as rendered, multimodal content for me to interpret, not a raw byte stream — reconstructing a `.jpg`/`.pdf` through a text-oriented Write call would not reproduce the original file, and running the pipeline's real classification/extraction code against a corrupted reconstruction would not be a genuine test of that code.

### Why execution was not forced

Fabricating a workaround that doesn't actually exercise `src/main.py`'s real code paths would produce results indistinguishable from genuine evidence while being worthless as evidence — precisely the failure mode this entire validation phase exists to avoid (`REAL_WORLD_VALIDATION_PLAN.md` §2: "no code changes to pass validation... evidence over impression"). Stopping and reporting the blocker honestly was judged the correct call over presenting a simulated result.

### Finding VL-001-1

- **Error taxonomy category** (`REAL_WORLD_VALIDATION_PLAN.md` §7): Environment/infrastructure issue.
- **Severity:** High — not Critical (nothing unsafe happened; no file was touched, moved, or lost — this blocked *before* any pipeline action), but High because it fully prevents this milestone's core deliverable (real pipeline execution against real data) in this specific execution environment.
- **What was expected:** A folder connected for file access would be reachable by the same session's code-execution environment, allowing `src/main.py` to run against it directly, as `VALIDATION_CHECKLIST.md` Part A assumes.
- **What actually happened:** File-reading tools and the code-execution sandbox have independently granted filesystem access in this Cowork session; connecting a folder for one does not connect it for the other.
- **Disposition tag:** **Architecture defect** — but scoped to the *validation environment*, not the Downloads Intelligence pipeline's own architecture. Nothing in `src/` caused this; every module's `MODULE_CONTRACT.md` already assumes direct filesystem access without qualifying which access model. This is new information about *where* v1's "Manual mode, direct filesystem access" assumption (`Governance/ARCHITECTURE_DECISIONS.md`) does and doesn't hold — worth carrying into `TECHNICAL_DEBT_REGISTER.md` as a new item once a second occurrence confirms it isn't specific to this one session (per this phase's own "don't propose fixes after a single occurrence" instruction — recorded here, not yet promoted).
- **Suspected cause:** Cowork's per-session folder-connection model grants file-tool access and sandbox-mount access as separate steps; only the latter appears to have been skipped or failed silently for this particular folder connection.
- **Root-caused?** Partially — confirmed via direct `mount`/`df` inspection, not assumed. Not yet confirmed whether this reproduces on a fresh reconnect or a different folder (single occurrence — no fix recommended yet, per this phase's standing instruction).
- **Linked technical debt item:** None yet — new class of finding, specific to the validation environment rather than the pipeline itself. Candidate for a new register entry if a second run reproduces it.

### Secondary observation (not yet a scored finding — no pipeline execution occurred to score against)

The real Downloads folder's shape is independently informative even without running the pipeline: nearly everything discovered is nested 2+ folders deep, with the one genuinely personal-document folder (`Documents/Personal/`) itself two levels down (`Downloads/Documents/Personal/...`). Module 01 scans only the top level of Downloads by design (`TECHNICAL_DEBT_REGISTER.md` TD-09). If this folder's shape is at all typical, a real Manual-mode scan would find substantially less than what's actually sitting in the folder — worth specifically re-checking once a run actually executes, not concluding from this alone (one folder, zero pipeline runs, well below this framework's own evidence bar).

### Disposition

Open. Routed back to the project owner (this session) for a decision on how to restore real execution access before Run 002 can proceed. No fix implemented, no code touched, no metrics fabricated to fill the gap.

### Findings

**Finding VL-002-1 — Scanned document photos (`.jpg`) misclassified as Screenshot, 0/9 correct**
- Error taxonomy category: Misclassification. Severity: **Medium** (caught by `review_required`, not silently auto-filed — not an Auto-tier misfile).
- Root cause, confirmed directly against code and real EXIF data (not inferred): `Rules/Classification Rules.md`'s Pass 1 routes every image-family extension through the Screenshot-vs-Image split only — there is no path from a `.jpg`/`.png` to `Document`, unlike a scanned PDF (which can reach the vision deep pass and, from there, `Document`). All 9 real files independently measured via Pillow: zero EXIF tags on every file (`has_camera_metadata() == False` for all 9, confirmed by direct inspection, not assumed) — which alone is sufficient, per the current rule, to force `Screenshot` regardless of actual content. Filenames don't match screenshot markers and dimensions don't match any common screen resolution — those two signals correctly returned "not a screenshot," but the missing-EXIF signal overrode them.
- Disposition tag: **Rule improvement.** The implementation faithfully executes `Rules/Classification Rules.md` exactly as written — this is not an implementation defect. The gap is in the rule itself: it has no provision for an image-only file that is actually a scanned/photographed document rather than a photo or a screen capture.
- Linked technical debt: `TECHNICAL_DEBT_REGISTER.md` TD-16 (previously a theoretical/disclosed risk — "false-positives on real photos with no camera EXIF"). This run is the **first real, empirical confirmation** of that risk, and shows its reach is broader than TD-16's original framing: it captures scanned official documents, not just ordinary photos.
- Downstream compounding impact (not separately scored, but real): the misclassification cascades — naming lost all distinguishing detail (every file collapsed toward `Screenshot_Unknown_Context_Unknown_Date`), and Screenshot's extraction schema has no fields for academic-document content at all, so even a live provider couldn't have recovered the actual information (institution, semester, grade) once routed down this path.
- **No fix recommended yet** — first occurrence in real-world validation, per this phase's standing instruction. Tracked below for pattern confirmation.

**Finding VL-002-2 — Metadata extraction produced no usable fields (informational, not a defect in this run)**
- Both of Screenshot's fields (`context_description`, `capture_date`) came back `null` for all 9 files. `context_description` is provider-dependent and no live provider was supplied this run — expected, disclosed fallback behavior (`src/main.py`'s own docstring: "expected, safe behavior, not a defect"). `capture_date` relies on EXIF, and real EXIF was confirmed absent on all 9 files independent of provider availability.
- Disposition tag: **User expectation** (matches documented behavior exactly) for the fallback mechanism itself. A second, structural observation worth carrying forward: **Architecture defect** in the sense that Screenshot's extraction schema has no path to real document content even with a provider — that capability doesn't exist anywhere in the current taxonomy for image-only documents, only for text-bearing ones. Not scored as a standalone defect; recorded as context for VL-002-1's downstream impact.

**Finding VL-002-3 — False-positive "superseded version" relationships between distinct academic documents (3 of 9 files)**
- Error taxonomy category: Duplicate/version error (`incorrect_version_chain`). Severity: **Medium**, with a real-world impact note: the generated Duplicate Report states, in plain language, `MarK_Sheet_10th.jpg` — Superseded Version — Related To `Mark_Sheet_12th.jpg`, and similarly pairs 1st↔4th and 2nd↔3rd Semester mark sheets. These are not different versions of the same document — they are entirely distinct academic records (different grade levels / different semesters). Nothing was moved (safety held — see positive finding below), but a user who trusted this report's own framing and manually archived what it calls "superseded" could lose records that were never actually superseded. This is real, reproducible, and fully deterministic (Module 04 has no provider — confirmed via `Module 04 Design.md` §14) — not a one-off fluke of missing judgment.
- Disposition tag: **Rule improvement** (Module 04's version-chain heuristic needs reconsideration for same-template, same-institution document series that are legitimately distinct records, not sequential versions of one file) — possibly also **Model improvement** if perceptual-hash template similarity turns out to be the specific driver rather than filename-token parsing; root cause not yet isolated to that level of specificity, and per this phase's standing instruction, not pursued further from a single occurrence.
- Not yet linked to an existing `TECHNICAL_DEBT_REGISTER.md` item — this is a new class of finding (false-positive version-chain on structurally-similar-but-distinct real documents), distinct from TD-34 (which is about recording only one match per type, not about false positives). Candidate for a new register entry once a second occurrence confirms the pattern.
- **No fix recommended yet** — first occurrence, per this phase's standing instruction.

**Finding VL-002-4 (positive finding) — The safety gate held completely**

### Findings

**Finding VL-003-1 — Screenshot misclassification of real non-screenshot images: PT-002 recurs, promoted to Confirmed Pattern**
- Error taxonomy category: Misclassification. Severity: **Medium** (caught by `review_required`/staying unfiled for all-EXIF-less files scored below the auto/approval bar — no misfile reached execution).
- Root cause: identical mechanism to Run 002's VL-002-1 — `Rules/Classification Rules.md`'s Screenshot split treats "no camera EXIF data present" as sufficient on its own to force `Screenshot`, with no route back to `Image`/`Document` for image-family extensions. Confirmed again directly against real, independently measured EXIF (Pillow) on all 19 files this run.
- What's new this run: the content shape is far more diverse than Run 002's single scanned-document type. Six files were opened and visually confirmed as genuinely non-screenshot real content, all misclassified `Screenshot`: a real personal photo shared over WhatsApp (`IMG-20201219-WA0008.jpg`, EXIF stripped by the sharing app, not a scan), a product photograph (`chair.png`), a marketing banner graphic (`Professional_LinkedIn_Banner_1128x191.png`), and an AI-generated portrait image (`Untitled canvas.jpeg`) — plus the two files in the false version-chain (`image (2).png`, `image (42).png`), one a portrait and one a delivery-truck advertisement mockup, both clearly not screen captures. A further 12 EXIF-less images (5 explicitly filename-tagged `DALL·E`, 1 `ChatGPT Image`, 2 more `WhatsApp Image`-pattern files, 2 with long AI-image-generator-style descriptive filenames, 2 unverified) were also classified `Screenshot` but not individually opened — very likely the same pattern, disclosed as inferred, not confirmed.
- The one contrast case in this run's own sample, `IMG_20211002_001030_Bokeh.jpg` (genuine camera EXIF present), classified correctly as `Image` — sharpening confidence that EXIF-absence, specifically, is the trigger, exactly as VL-002-1 established.
- Disposition tag: **Rule improvement** (same as VL-002-1 — the code correctly implements the rule as written; the rule's own definition of "Screenshot" is what's over-broad). A secondary, newly visible dimension: WhatsApp-shared photos strip EXIF on export, meaning *any* real photo shared via a common messaging app — not just scanned documents — will hit this same gap. That is a materially larger real-world surface than Run 002's narrower framing suggested.
- **Classification change:** promoted from **Single Observation** to **Confirmed Pattern** — this is the second independent run, second independent (non-overlapping) dataset, and the widest content diversity tested so far, all showing the identical, precisely-understood mechanism. See `PATTERN_TRACKER.md` PT-002.
- Linked technical debt: `TECHNICAL_DEBT_REGISTER.md` TD-16.

**Finding VL-003-6 — Batch cleanup step failed on a FUSE-mounted validation folder, after all real file work had already completed correctly**
- Error taxonomy category: Environment/infrastructure issue. Severity: **Low** (no data loss, no incorrect filing — confirmed directly; the failure is confined to deleting an already-obsolete temp bookkeeping file).
- `execute_batch()`'s final step, `clear_batch_temp()`, calls `shutil.rmtree()` on `Runtime/Temp/<batch_id>/` to remove a `plan.json` staging file that is no longer needed once every file in the batch has been processed (each already succeeded or failed individually before this step runs, per `src/pipeline/execution.py`'s own per-record Layer 1/Layer 2 safety nets). On this run's validation folder — a FUSE mount (`mount` confirms `type fuse`) — the deletion raised `PermissionError: Operation not permitted`, reproduced independently via a direct, minimal `os.remove`/`os.rmdir` probe on the same mount outside the pipeline entirely, confirming the cause is the mount itself, not pipeline code.
- `clear_batch_temp()`'s own docstring discloses this is intentional, not a gap: `ARCHITECTURE_DECISIONS.md` decision 19 deliberately does not swallow a genuine deletion failure behind `ignore_errors=True`, so a real `OSError` is meant to propagate rather than be silently absorbed. This run confirms that "fail loudly" choice behaved exactly as designed — and, importantly, it fired only *after* every real file move for the batch had already completed and been logged correctly (confirmed directly: all 14 `move_rename` action-log entries and all 14 `processed_at` timestamps predate the failure).
- Disposition tag: **Architecture defect**, but scoped to the validation environment (same framing as PT-001), not the released pipeline — a normal user's own filesystem is very unlikely to reject `rmtree()` on its own disposable temp folder the way this sandbox's mounted validation folder does.
- Secondary, real observation worth carrying forward regardless of environment: a genuine `execute()` call on a real user's machine that hits *any* transient reason a temp-folder delete fails (a stray file lock from antivirus/cloud-sync software, for instance) would surface as an uncaught exception at the very end of an otherwise fully successful batch — which could read, to a real user watching the CLI output, as "the whole operation failed" even though every file was actually filed correctly. Not proposed as a fix (single occurrence, environment-attributable) — flagged as a discussion point for a future run if it recurs on a non-FUSE, real user filesystem.
- Linked technical debt: candidate for a new register entry if reproduced outside this validation sandbox.
