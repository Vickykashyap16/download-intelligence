# Project Review Board Report — Downloads Intelligence

**Date:** 2026-07-20
**Board role:** Independent Technical Review Board — final engineering review before authorizing the next development cycle.
**Inputs reviewed (sole evidentiary basis for this report):** `VALIDATION_LEDGER.md`, `PATTERN_TRACKER.md`, `VALIDATION_PROGRESS.md`, `VERSION_091_IMPLEMENTATION_PLAN.md`, `TECHNICAL_DEBT_REGISTER.md`, `PROJECT_RETROSPECTIVE.md`, `Governance/FROZEN_MODULE_CHANGE_POLICY.md`.
**Constraints observed while producing this report:** no production code was read for the purpose of forming new judgments, none was modified; no new implementation work was scoped; no architecture redesign or new feature is proposed anywhere below. Every conclusion below cites the specific source document and, where applicable, the specific finding ID (PT-/VL-/TD-) it rests on. Where the cited documents do not contain enough evidence to support a conclusion, that gap is stated directly rather than filled with inference.

## 1. Executive Summary

**Overall assessment.** Downloads Intelligence is a completed, internally-consistent eight-module engine (Pipeline v0.8.0) that has now been carried through an initial, honestly-scoped real-world validation phase. The engineering record behind v0.8.0 is unusually strong: 716/716 automated tests passing, zero Critical or High finding ever reached a released module unresolved, and every module passed through the same independently-audited nine-stage lifecycle (`PROJECT_RETROSPECTIVE.md` §1–2). Real-world validation (`VALIDATION_LEDGER.md` Runs 001–003) has, for the first time, tested this engine against genuine, uncurated content rather than purpose-built test fixtures — and the outcome is consistent with, not contradictory to, the engineering record: no safety property failed, but two real, previously-only-theorized quality gaps were confirmed with independent evidence across two separate runs.

**Current maturity level.** The project itself distinguishes "released" (independently verified against its own written contract) from "validated in real-world use" (`PROJECT_RETROSPECTIVE.md` §3: *"No module in this pipeline has ever processed a real, unstaged, everyday Downloads folder"* — true at the time that document was written). That gap has now been partially closed: three real validation runs have occurred, one of them (Run 003) exercising real file execution for the first time. But the evidence volume remains far short of the bar the project set for itself before any run began (`REAL_WORLD_VALIDATION_PLAN.md` §9, tracked in `VALIDATION_PROGRESS.md` §2): acceptance criterion A4 calls for ≥3 real-world sessions spanning ≥2 calendar weeks and ≥150 files; the project has 2 executed sessions, both on the same calendar day, and roughly 45 files cumulatively processed. This board's assessment: **engineering-complete, early-stage real-world-validated.** Not yet at the evidence volume the project's own plan requires for a confident production judgment.

**Major strengths** (each independently evidenced, not asserted):
- Reversibility and the confidence-tier safety gate held under real, uncurated, partly-wrong conditions in every run that reached execution — not just designed in, but empirically confirmed (`VALIDATION_LEDGER.md` VL-002-4, VL-003-3; `PATTERN_TRACKER.md` PT-004, now Confirmed Pattern).
- The one finding closest to a genuine stress test of core safety mechanics — Run 003's real execution against a batch with known-wrong upstream classification and duplicate-detection results — produced zero incorrect filings and zero content-integrity failures across all 14 executed files, independently SHA-256-verified (`VALIDATION_LEDGER.md` Run 003, Finding VL-003-3).
- The technical debt register anticipated one of the two confirmed defects before validation began: TD-16 disclosed "Screenshot classification heuristic false-positives on EXIF-stripped real photos" as a theoretical, disclosed risk prior to any real-world run. Validation converted that theoretical risk into directly-confirmed, broader-than-originally-scoped evidence (`TECHNICAL_DEBT_REGISTER.md` TD-16; `PATTERN_TRACKER.md` PT-002). This is a genuine sign of a project that discloses honestly rather than discovers by accident.
- Documentation discipline held under real validation pressure exactly as it held during original engineering: every run, every finding, every classification change is dated, append-only, and traceable (`VALIDATION_LEDGER.md`'s own "never rewrite history" convention, consistent with `CHANGELOG.md`'s established practice per `PROJECT_RETROSPECTIVE.md` §2).

**Major weaknesses** (each independently evidenced):
- Two Confirmed Pattern quality defects exist with real downstream consequences — misclassification (`PT-002`) and false version-chain grouping (`PT-003`) — both Medium severity, both never resulting in an unsafe automated action, but both real and reproducible across independent datasets.
- Acceptance criterion A2 (undo success rate) remains **entirely untested** against real executed content — zero undos have been attempted in any run, despite Run 003 producing 14 real, genuinely-filed files that were available for exactly this test (`VALIDATION_PROGRESS.md` §2, row A2).
- Acceptance criterion A4 (evidence volume/calendar time) is unmet, and by the project's own admission no amount of same-day work can close the calendar-time component (`VALIDATION_PROGRESS.md` §3–4).
- `TD-01` (no autonomous classification/extraction provider) means every real-world validation run to date depended on a live, in-session judgment substitute standing in for genuine unattended operation — a caveat on how far the current evidence generalizes to the "Scheduled" execution mode the project's own `README.md` already lists as supported (`TECHNICAL_DEBT_REGISTER.md` TD-01).
- One open High-severity finding, `PT-001` (validation-environment access gap), remains formally unresolved — it did not reproduce in Run 003, but the Pattern Tracker has not promoted this to "resolved" without a second independent confirmation (`PATTERN_TRACKER.md` §2, PT-001 row).

## 2. Architecture Review

**Is the architecture still sound after real-world validation?** Yes, on the evidence available. Across three runs and eleven consolidated findings, no finding identifies a violation of a module contract, an ownership-boundary breach, a cross-module data-integrity failure, or a structural flaw in the pipeline's stage sequencing. Every confirmed defect traces to a specific rule or heuristic operating exactly as designed: PT-002's own root-cause writeup states plainly, *"the implementation faithfully executes the rule as written; this is a rule-completeness gap, not an implementation defect"* (`VALIDATION_LEDGER.md` VL-002-1, reaffirmed in Run 003's VL-003-1). PT-003 is explicitly the same category — Module 04's deterministic matching logic, not its architecture, is the object of the finding (`VERSION_091_IMPLEMENTATION_PLAN.md` §3, "Root cause"). The one environment-attributable finding with real product-adjacent implications (`PT-009`, the FUSE-mount batch-cleanup failure) was traced directly to `ARCHITECTURE_DECISIONS.md` decision 19's deliberate "fail loudly, don't swallow a real `OSError`" choice — the finding confirms that design decision behaved exactly as intended once the failure's cause (a sandbox-specific filesystem limitation, independently reproduced outside the pipeline entirely) was isolated (`VALIDATION_LEDGER.md` Finding VL-003-6).

**Were any original design assumptions disproven?** None of the reviewed evidence disproves a core architectural assumption. One specific, already-disclosed limitation was shown to have broader real-world reach than its original framing suggested: `TD-16` framed the Screenshot-heuristic risk narrowly around scanned documents; Run 003's confirmed evidence extends it to messaging-app-shared personal photos, product/marketing graphics, and AI-generated images (`PATTERN_TRACKER.md` PT-002 detail). This board characterizes this as a **scope correction to an already-disclosed limitation**, not a disproven assumption — the underlying mechanism (no camera EXIF forces `Screenshot`) was correctly anticipated; only its real-world blast radius was underestimated at design time.

**Did validation reveal architectural flaws or only heuristic limitations?** Only heuristic limitations, on the evidence reviewed. This is stated with the appropriate caveat: the evidence volume gathered (§3 below) is not yet large enough to assert with high confidence that no architectural flaw exists anywhere in the pipeline — only that none has been found in the specific real-world content processed so far, across the specific stages exercised (classification, extraction, duplicate/version detection, naming, confidence scoring, real execution, reporting). Every one of Module 01 through Module 08's own contract guarantees relevant to the files processed in Run 002 and Run 003 held without exception. This board finds no basis in the reviewed documents to recommend any architectural change.

## 3. Validation Assessment

**Run-by-run review.**
- **Run 001 (2026-07-20):** Blocked before any pipeline stage executed. Zero files processed, zero metrics collected. Produced one finding, `PT-001` (validation-environment access gap), correctly classified as an Environment Issue rather than forced into a fabricated result (`VALIDATION_LEDGER.md` Run 001).
- **Run 002 (2026-07-20):** 9 real files, a single, narrow, hand-selected content shape (scanned academic documents). Executed the full pipeline chain through report generation; never reached execute() with anything eligible (all 9 landed `review_required`). Produced 5 findings (`PT-002` through `PT-006`).
- **Run 003 (2026-07-20):** 43 files, a systematic (not hand-picked) sample of the real Downloads folder's top level; 36 discovered/processed, 14 executed for real (8 `auto`, 6 `approval_required` with genuine approve/edit decisions). First run to exercise real file execution, real human-edit capture, and content-integrity verification. Produced 5 corroborating updates to existing findings (promoting `PT-002`, `PT-003`, `PT-004` to Confirmed Pattern) and 5 new findings (`PT-007` through `PT-011`).

**Quality of the evidence.** This board assesses the *methodology* as high-quality and the *volume* as low, and treats these as separate questions per `METRICS_DEFINITION.md`'s own standing rule that a rate must always be reported alongside its raw count. In both executed runs, ground truth was established by directly reading real file content (PDF text extraction, real EXIF measurement via Pillow, direct visual inspection of a subset of images) rather than assumed from filenames or extensions — `VALIDATION_LEDGER.md`'s Run 003 entry explicitly discloses which of its ground-truth judgments were directly confirmed (6 of 18 candidate misclassified images visually opened) versus inferred from strong-but-unconfirmed signals (the remaining 12), and excludes the inferred set from the formal accuracy metric. State isolation (a fully separate Database/Runtime/Filed library per run) means no run risked contaminating another run's evidence or the project's real state. Post-execution content-hash verification in Run 003 (14/14 files byte-identical to their pre-move hash) is direct, falsifiable evidence, not a self-report. This is a well-run validation process operating at low volume — the discipline of the methodology does not by itself compensate for the small sample.

**Statistically meaningful vs. inconclusive findings.**
- **Meaningful, with an important caveat:** `PT-004`/`PT-007` (the safety gate and real-execution correctness). Zero incorrect filings and zero content-integrity failures occurred across every opportunity in both executed runs (9 withheld correctly in Run 002; 14 correctly executed and 22 correctly withheld in Run 003). For a binary safety property specifically, a run of consecutive successes with zero counterexamples is meaningful evidence, even at modest N — a single counterexample would have been immediately disqualifying, and none occurred. This board still declines to call this "statistically proven" given the sample size; it calls it **the strongest evidence in the dataset**.
- **Meaningful for existence, not for rate:** `PT-002` and `PT-003`. Both are now Confirmed Pattern — the mechanism is real, reproduced independently across two non-overlapping datasets, and (for PT-002 especially) precisely root-caused. Neither run supports a reliable real-world *rate* estimate, a limitation the source documents state about themselves rather than one this board is inferring: `PATTERN_TRACKER.md` §4 states directly, *"neither run supports a reliable rate estimate."*
- **Explicitly inconclusive:** `PT-001` (did not reproduce in Run 003, but not confirmed resolved — one data point either way), `PT-009` (single occurrence, environment-attributable, not tested for reproduction on a non-FUSE filesystem), and `PT-008`/`PT-011` (each single-run evidence for an already-disclosed, designed behavior — informative but explicitly not treated as a new pattern by the project's own standing single-occurrence rule).
- **A2 (undo) and A5 (acceptance rate):** insufficient evidence to draw any conclusion. A2 has zero data points of any kind. A5 has exactly 6 real approval decisions in the entire validation history (`VALIDATION_LEDGER.md` Run 003 metrics table) — this board states plainly that six data points cannot support a production acceptance-rate judgment, even though the *cause* of all 4 edits in that sample is well understood (`PT-008`).

## 4. Release Readiness

**Recommendation: Ready for Limited Production.**

**Justification.** This board considered all four options and rejects the other three on the following grounds:

- **Not "Ready for Production"** — the project's own pre-declared acceptance bar (`REAL_WORLD_VALIDATION_PLAN.md` §9, tracked in `VALIDATION_PROGRESS.md` §2) is not met: A4 (evidence volume/calendar time) and A2 (undo, entirely untested) are the two clearest gaps, and A7 carries one still-open High finding. Recommending full production readiness would mean overriding the project's own, already-agreed bar without new evidence to justify doing so — this board declines to do that.
- **Not "Not Ready"** — this would misrepresent the evidence. Every safety-relevant property tested (data-loss prevention, correct execution/withholding discrimination, content integrity, human-edit honoring, malformed-input handling) held without exception across all real testing performed. The confirmed defects (`PT-002`, `PT-003`) are quality-of-classification issues that were, in every single instance across both runs, caught by the existing safety net before any incorrect action occurred. "Not Ready" is not supported by evidence of unsafe behavior, because none exists in the reviewed record.
- **Not "Ready with Known Limitations"** (unqualified) — this framing, as used elsewhere in the project's own vocabulary (`TECHNICAL_DEBT_REGISTER.md`'s "accepted trade-offs, not defects"), implies a green light for ordinary use with disclosed caveats. This board finds one gap too consequential to leave as a mere disclosed caveat: reversibility is the project's own stated non-negotiable (`CLAUDE.md`: *"every action must be reversible"*), and it has never been tested against real executed content. An untested non-negotiable is a scope boundary, not a footnote.
- **"Ready for Limited Production"** is the recommendation this board finds proportionate to the evidence: continued real use is warranted, under supervision (attended sessions, with the human-approval workflow this validation phase itself relied on remaining in place, not the "Scheduled"/unattended mode `TD-01` already flags as unproven), while the two specific, well-scoped gaps below are closed. This is a scope restriction on *how* the engine is used next, not a statement that the engine is unsafe.

**Conditions this board attaches to the "Limited" scope**, each tied to a specific open item above: (1) continue attended/interactive operation only, consistent with `TD-01`'s own disclosure that unattended operation has never been validated end-to-end; (2) treat any file that reaches `auto` tier with continued spot-checking, consistent with A3's own "first real evidence" status (8/8 correct is a promising but small sample); (3) do not rely on this validation evidence to certify Scheduled/Watch-Folder execution modes, which remain outside everything tested to date.

## 6. Review of the v0.9.1 Implementation Plan

**Appropriately scoped: Yes.** The plan's own selection filter (`VERSION_091_IMPLEMENTATION_PLAN.md` §1) admits a finding only if it is Confirmed Pattern, a genuine defect (not User Expectation/Environment Issue/positive finding), and directly root-caused against real code. Applied to the current 11-item Pattern Tracker, this yields exactly the two candidates this board also independently identifies as the only defensible engineering candidates in §5 above (`PT-002`, `PT-003`). The plan explicitly accounts for and excludes the other nine findings with a stated, verifiable reason each (`VERSION_091_IMPLEMENTATION_PLAN.md` §4) — this board checked each exclusion against the Pattern Tracker's own classification and found no exclusion that misreads the underlying evidence.

**Evidence-based: Yes.** Every acceptance criterion in the plan cites a specific prior measurement (e.g., the existing Screenshot-positive-control test names, the specific Run 003 file counts) rather than an invented target. The plan is explicit where its own proposed numeric thresholds (e.g., "≥85% accuracy on a fresh ≥30-file held-out sample") are its own proposal, not something independently validated elsewhere — this board notes that distinction is disclosed by the plan itself, which this board regards as appropriate rigor rather than a defect in the plan. For `PT-003` specifically, the plan does not overstate its own evidence: it states root-cause isolation as a **precondition** to be completed before implementation design, rather than assuming the Medium-confidence root cause is already sufficient to design against (`VERSION_091_IMPLEMENTATION_PLAN.md` §3, "Root cause").

**Free from unnecessary feature creep: Yes.** The plan presents candidate directions without selecting one, explicitly defers that choice to each candidate's own future Design phase, and flags the one option (a live-judgment provider for Module 04) that would constitute a genuine architectural departure as requiring the full breaking-change approval process rather than folding it in as an ordinary patch (`VERSION_091_IMPLEMENTATION_PLAN.md` §3, "Risk estimate"). The plan explicitly states it does not authorize implementation and does not select an approach — this board finds no instance of scope expansion beyond the two evidenced candidates anywhere in the document.

**One gap this board notes, not a defect in the plan itself:** the plan does not address sequencing against the still-open evidence-volume gap (A4) — i.e., whether implementation-track work should proceed in parallel with continued validation, or wait for it. This is a legitimate silence (sequencing across documents is this board's remit, not the plan's own), addressed in §7 below.

## 7. Recommendations

In priority order:

1. **Continue validation — highest priority, and the only item on this list with no substitute.** Specifically: a validation session on a calendar day distinct from 2026-07-20 (the single largest, least-substitutable gap against A4); a real undo test against Run 003's 14 already-filed files (the only currently-available way to produce any A2 evidence at all, and does not require a new dataset); a larger or more representative image-heavy sample to move `PT-002` from confirmed-to-exist toward a defensible real-world rate. All three are explicitly recommended in `VALIDATION_PROGRESS.md` §5 and are not new proposals from this board.
2. **Begin Design-phase work for `PT-002` now — evidence is already sufficient.** Root-cause confidence is High and has been since Run 002, reaffirmed in Run 003 across a materially more diverse dataset. This board sees no evidentiary reason to wait for more validation runs before a Design phase (per `Governance/ENGINEERING_STANDARD.md`) begins evaluating the candidate directions the implementation plan already lays out — subject to the explicit per-finding approval `FROZEN_MODULE_CHANGE_POLICY.md` §1 requires before any change to a Frozen module (see §8 below).
3. **Do not begin Design-phase work for `PT-003` until its root cause is isolated to a specific comparison in Module 04's matching logic.** The implementation plan itself states this as a precondition (`VERSION_091_IMPLEMENTATION_PLAN.md` §3); this board concurs and treats it as a hard gate, not a preference — Medium confidence in the mechanism is not sufficient grounding for a Design phase whose job is to select among specific engineering approaches.
4. **Do not begin implementation of either candidate yet.** Both require Design → independent Review → Freeze before implementation under the project's own standard lifecycle, and both require explicit project-owner approval under `FROZEN_MODULE_CHANGE_POLICY.md` §1 before that lifecycle may even begin, since both Module 02 and Module 04 are Frozen, released modules.
5. **Other — resolve `PT-001`'s open status.** Confirm on a future session whether the validation-environment access gap is durably resolved or was a one-time condition, so `PATTERN_TRACKER.md`'s one remaining open High finding can be closed or re-confirmed rather than left ambiguous.

## 8. Final Decision

**Should v0.8.0 now be frozen as the validated baseline?** **Yes, with an explicit scope caveat.** Every module in v0.8.0 is already Frozen at the engineering-contract level (`PROJECT_RETROSPECTIVE.md` §2). This board recommends v0.8.0 also be treated as the **validated baseline** for future comparison (in the sense `BENCHMARK_SPECIFICATION.md`'s cross-release comparison design anticipates) — but "validated" must be read as "validated against the evidence gathered in Runs 001–003," not "exhaustively validated." No evidence reviewed suggests v0.8.0's code should be held back from this designation; the caveat is about the confidence level attached to the label, not about withholding the label itself.

**Should development of v0.9.1 begin?** **Design-phase work: yes, for `PT-002` only, subject to the blocker below. Implementation: no, not yet, for either candidate.** This board finds the evidence sufficient to begin the Design phase for the Screenshot-misclassification candidate specifically, and insufficient — by the implementation plan's own stated precondition — to begin Design work for the version-chain candidate until its root cause is isolated.

**Blockers that must be resolved first:**
1. **Explicit, per-finding project-owner approval under `Governance/FROZEN_MODULE_CHANGE_POLICY.md` §1.** Nothing in the reviewed documents records this approval as having been granted for either `PT-002` or `PT-003`. This is a procedural blocker, not an evidentiary one — the policy is unambiguous that "the project owner is asked to approve before any change is made... exactly as if it were a fresh finding," and this board finds no record that step has occurred.
2. **Root-cause isolation for `PT-003`**, as stated in the implementation plan itself and reaffirmed in §7 above — a precondition to that candidate's Design phase, not yet satisfied.
3. **A real undo test against genuinely executed content**, before any broader (beyond "Limited Production," per §4) release-readiness judgment is revisited. This is not a blocker to beginning `PT-002`'s Design phase, which is independent of A2 — but it is a blocker to any future upgrade of this board's §4 recommendation.

No other blocker is identified in the reviewed evidence. This board finds no basis to recommend against the project's own proposed next steps once the three items above are addressed.
