# PT-003 Postmortem — False-Positive Version-Chain Grouping

**Date:** 2026-07-23
**Status:** PT-003 is CLOSED / IMPLEMENTED / VALIDATED / MERGED. This document is the full narrative record of its lifecycle, from first observation through closure.

---

## 1. Original Observation

Validation Run 002 (2026-07-20, `Desktop/Validation Sample/`, 9 real academic mark-sheet photos) produced 3 instances where Module 04 flagged two distinct, unrelated files as a "version chain" — one marked `superseded`, the other `latest` — despite the files being genuinely different documents (10th vs 12th grade mark sheets, 1st vs 4th semester, 2nd vs 3rd semester). Validation Run 003 (same day, a systematic 43-file sample of a real Downloads folder) produced a fourth instance: `image (2).png` (a portrait photograph) and `image (42).png` (an unrelated delivery-truck advertisement mockup) were flagged as a version chain despite sharing no subject matter, no visual similarity, and no genuine relationship of any kind. Both runs recorded this as `PATTERN_TRACKER.md` PT-003. Because it recurred across two independent runs and two non-overlapping real datasets, it was classified Confirmed Pattern from the outset, Medium severity — never resulted in an incorrect automated filing in either run (both instances' files happened to also be `review_required` for unrelated reasons), but a real, reproducible false signal in the Duplicate Report's "Superseded Version" language.

## 2. Evidence Chain

The evidence supporting PT-003's diagnosis and fix forms one unbroken, independently-verifiable chain, each link measured directly against real code and real data rather than inferred from the observed symptom alone:

Real files (`Desktop/Validation Sample/`, `Runtime/Validation/RUN003_staging/`) → `normalize_filename()`'s real output on each pair → `rapidfuzz.fuzz.ratio()`'s real measured score on the normalized names (93.3, 91.3, 91.3 for Run 002's three pairs; 94.7 for Run 003's pair — all above `_NAME_SIMILARITY_THRESHOLD = 90`) → `parse_version_token()`'s real output (`None` for all 8 filenames, ruling out the token path) → perceptual-hash distance, measured directly (26, 8, 6, and 36 respectively — all far above `_MAX_PHASH_DISTANCE = 5`, ruling out the near-duplicate path) → the design package's synthetic regression tests reproducing both confirmed shapes and asserting no group forms under the corrected logic → the post-implementation validation gate re-running the real, corrected code against the real, hash-verified original datasets and confirming all 4 instances resolve with zero unintended side effects. Every link in this chain is traceable to a specific command run against specific real data, recorded in `PATTERN_TRACKER.md`, `VALIDATION_LEDGER.md`, `PT003_IMPLEMENTATION_REPORT.md`, and `PT003_VALIDATION_REPORT.md` — nothing in this diagnosis rests on an untested assumption.

## 3. Root Cause

`_check_version_chain()` (`src/pipeline/duplicate_detector.py`) accepted any same-category candidate whose normalized filename scored `fuzz.ratio() >= 90` against the record being processed as sufficient, on its own, to form a version-chain relationship. `normalize_filename()` strips only the extension, a recognized explicit version token, and separator runs — it does not strip generic trailing content like ordinals ("10th"/"12th") or tool-assigned incrementing suffixes ("(2)"/"(42)"). Two filenames sharing a long common template prefix or a generic base name, differing only in that trailing content, routinely clear a 90% edit-distance similarity score even though the differing suffix is exactly what distinguishes two genuinely unrelated files. No second, corroborating signal — an explicit version token, an identical name, file size, or anything else — was required before this similarity score alone produced a version-chain grouping. This is a rule-completeness gap: the code faithfully implemented the design as originally specified; the design itself never required a second signal.

A related, second instance of the same underlying gap was found later, during the design's own independent review (Round 1, Finding E1): an *identical* normalized filename trivially scores 100 on the same similarity metric, and the original selected design's identical-name branch treated that as unconditionally sufficient too — which is equally vulnerable, since an identical name is just as often a symptom of a generic, uncustomized filename (`invoice.pdf`, `document.pdf`) as it is evidence of a genuine version relationship. Both instances of the gap were closed together in the design's final revision.

## 4. Design Evolution

The design went through two substantive revisions before reaching its approved final form:

- **Revision 1** proposed the core fix: require `fuzz.ratio() >= 90` (unchanged) plus a corroborating signal — either an explicit version token on at least one side, or an identical normalized filename. Evaluated and rejected three alternatives on the record: a phash-based corroboration (rejected — the confirmed evidence showed the near-duplicate path was never involved in any instance, so it wouldn't have helped); a live-judgment provider (rejected — a genuine architectural/contract change, pre-disclosed as a deferred future option in `Release/Module04/KNOWN_LIMITATIONS.md`, disproportionate to two small, well-scoped, deterministically-fixable instances); and a Duplicate Report language softening (rejected as in-scope — it's Module 08's code, not Module 04's, and this project's own scope discipline keeps one correction to one module).
- **Revision 2** (produced in response to the Round 1 independent review, see §5) added a size-proximity requirement to the identical-name branch (closing Finding E1), and added a dedicated "Rejected Alternatives" section explicitly evaluating and rejecting a pure similarity-threshold increase using the real confirmed scores (closing Finding S1). During this same revision, the round-2 reviewer caught and the design closed, within the same pass, a division-by-zero edge case in the new size-proximity formula for two zero-byte files.

The final selected design: a candidate qualifies only if it clears the existing similarity threshold **and** either carries an explicit version token on at least one side, or shares an identical normalized name corroborated by a size-proximity check (smaller file at least half the larger's size, with zero-byte files special-cased as unambiguously equal and missing size data conservatively treated as disqualifying).

## 5. Review Findings

Two independent review rounds were performed, both genuinely adversarial rather than confirmatory:

- **Round 1** returned **APPROVE WITH CHANGES**, with two Medium findings: **E1**, that the identical-name branch added no real corroborating value on its own and was itself exposed to a generic-filename false-positive shape (the same class of problem the fix was meant to solve, relocated rather than eliminated); and **S1**, that the design never explicitly evaluated and rejected the simplest alternative — raising the similarity threshold — leaving that comparison implicit rather than on the record. No Critical or High finding; no architectural, contract, or performance concern.
- **Round 2**, against the revised design, returned **APPROVE**. It independently re-verified that E1 and S1 were substantively closed (not just superficially addressed), and its own adversarial pass caught one new, real issue in the revision itself — the zero-byte division-by-zero — which was folded into the same revision rather than triggering a third cycle. It also recorded, as non-blocking, that the new `_VERSION_SIZE_PROXIMITY_RATIO = 0.5` constant is an engineering judgment call not yet calibrated against real data (no real instance of the identical-name branch existed in either validation dataset to calibrate against).

## 6. Implementation Summary

Implemented entirely within `src/pipeline/duplicate_detector.py`: one new constant (`_VERSION_SIZE_PROXIMITY_RATIO`), one new private method on `DuplicateDetectionEngine` (`_has_corroborating_version_signal()`), and a three-line filter added to `_check_version_chain()`. `src/storage/database.py` and `lookup_name_matches()` were left completely untouched — the corroboration check turned out to be implementable entirely from data `_check_version_chain()` already held (`records_by_id`'s full `FileRecord` objects), a smaller footprint than the design document itself anticipated might be needed. Four existing tests, which had relied on identical-name-only matching to exercise unrelated logic (cross-group-conflict flagging, M1's null-group handling, batch re-examination idempotency), were updated with matching `size_bytes` fixtures to keep exercising that logic under the new rule — exactly the follow-up the design's own Regression Impact analysis had predicted. Nine new tests were added covering both confirmed real-world shapes, the identical-name positive/negative controls, the two disclosed residual risks, the missing-size-data case, and the zero-byte edge case. Full regression suite: 720 → 729, all passing. No module contract changed, no module outside Module 04 was touched, no scope expansion occurred, and no stop condition was triggered.

## 7. Validation Summary

Rather than reconstructing the original datasets a third time, the post-implementation validation gate reused the already byte-for-byte-reconstructed, already hash-verified Run 002 and Run 003 datasets from PT-002's own validation cycle — sound because PT-003 touches nothing upstream of Module 04, and independently re-confirmed by re-hashing all 45 real files immediately before this validation (0 missing, 0 mismatches). The four Module-04-owned fields were reset on a fresh copy of every record, and the real, corrected `detect_duplicates_batch()` was re-run against isolated storage. Result: all 4 confirmed false-positive version-chain groupings eliminated, zero replacement pairings formed, zero unintended change to any other field on any of the 45 records, `tier`/`confidence_score` byte-identical before and after for every record, and the full regression suite passing at 729/729. Verdict: **PASS WITH NOTES** — the notes cover two honest, pre-existing gaps neither dataset can positively resolve (neither contains a genuine version chain to test recall against, nor a real generic-identical-name pair to test the new size-proximity branch against), not defects found in the correction itself.

## 8. Residual Risks

Two risks are carried forward by design, disclosed rather than eliminated:

- **R1** — a genuine version chain using neither an explicit token nor an identical filename (e.g. an organic rename like "Meeting Notes.docx" → "Meeting Notes Updated.docx") is not detected. Confirmed present in synthetic testing; not yet exercised by any real dataset in either direction.
- **R6** — two unrelated files that coincidentally share both a generic identical filename and a similar size can still form a false version-chain group. Narrower than the pre-fix exposure (which required only an identical name, no size check at all), not eliminated by it. Not observed in either real validation dataset, not proven absent from a broader real Downloads folder.

A third item, **not this fix's scope but surfaced during its design work**, remains open and unresolved: `resolve_precedence()` (`src/pipeline/execution.py`) does not exempt non-`review_required`-tier records from automatic archival on a `superseded_version` outcome. Neither PT-003 instance ever reached this exposure (both were `review_required` for unrelated reasons), so it remains a real, disclosed, entirely untested risk independent of PT-003's own disposition.

## 9. Lessons Learned

1. **A fix for one false-positive mechanism can relocate the same reasoning error rather than eliminate it, if the fix's own new branch isn't held to the same evidentiary standard as the one it replaces.** Finding E1 exists because the identical-name branch, added to fix a filename-similarity problem, itself relied on filename similarity (in its most extreme, 100%-score form) as sole evidence — an adversarial review pass, not the original design author, caught this. A second independent review round for a design revision is not procedural overhead; it found a real, distinct issue (E1/S1) the first pass could not have anticipated because the revision didn't exist yet.
2. **Reusing an already-verified dataset reconstruction, rather than rebuilding one from scratch, is legitimate when the new change provably doesn't touch anything upstream of the reused state** — but that reasoning has to be stated explicitly and checked (PT-003's validation confirmed this in writing, §7, rather than silently reusing PT-002's artifacts without justification).
3. **A synthetic-test-only edge case (the zero-byte division-by-zero) is exactly the kind of defect an adversarial design review, not a design author's own re-reading, is positioned to catch** — it required reasoning about a boundary condition (`0/0`) the design's own worked examples never exercised.
4. **This project's own process document (`ENGINEERING_CHANGE_PLAYBOOK.md`), written in direct response to PT-002's gate-skip, was itself skipped again for PT-003** — the 13-check PCV gate was not re-run at this Medium-severity change's closure, the exact gap the playbook exists to close. This is the most important lesson from this cycle: writing a process document does not, by itself, guarantee the next change follows it — the gap recurred on the very next Medium-severity change after the playbook was created. See §10.

## 10. Process Improvements

1. **The PCV gate gap needs an actual enforcement mechanism, not just a documented requirement.** `ENGINEERING_CHANGE_PLAYBOOK.md` §6 already states the requirement in writing; that alone was not sufficient to make it happen for PT-003. A future change to the playbook or the merge-checklist habit itself (e.g. treating the PCV gate as a literal, un-skippable step in the closure instructions given to whoever executes Merge and Close, not just a documented expectation) is worth the project owner's explicit attention before a third Medium-severity change repeats this same gap a second time.
2. **The "Rejected Alternatives" section format introduced for S1 is worth keeping as a standard design-package section going forward**, not just a one-off fix for this review finding — it made the threshold-raise-vs-corroboration-signal tradeoff auditable in a way the original Alternatives Considered section's more general comparison did not.
3. **Reviewing a revised design for new issues introduced by the revision itself (not just whether the original findings were closed) is worth keeping as an explicit, named step of any second review round** — Round 2's catch of the zero-byte edge case only happened because the review scope explicitly included "does the fix introduce anything new," not just "were E1/S1 addressed."

## 11. Why the Change Succeeded

- **Every claim was graded honestly by its actual evidentiary basis** (Confirmed / Suspected / Unknown) from the first evidence-verification step through the final validation report — nothing was asserted at higher confidence than the underlying measurement supported, and gaps (U1, U3, U4/R6) were named explicitly rather than implied away.
- **The fix stayed exactly as small as the evidence justified.** Two confirmed real-world instances, fully explained by one precise mechanism, received a three-line, single-file, single-condition correction — not a broader redesign, not a new architectural layer, not a scope expansion to "fix duplicate detection in general."
- **Two genuinely independent adversarial review rounds found real, non-trivial issues** (E1, S1, and the zero-byte edge case) that materially improved the design before any code was written — the review process was not a rubber stamp at either round.
- **Validation reused, rather than duplicated, already-verified evidence where it was legitimate to do so, and said so explicitly** — efficient without sacrificing rigor, because the reasoning for reuse was stated and checked rather than assumed.
- **What remained unresolved was disclosed as unresolved, at every stage** — the two residual risks (R1, R6), the untested `resolve_precedence()` exposure, and the recurring PCV-gate process gap are all recorded plainly in this document and the release documentation, not smoothed over to present a cleaner-looking closure than the evidence actually supports.
