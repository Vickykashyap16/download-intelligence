# Module 07 (Preview, Approval & Execution) — Implementation Plan

**Status: PLANNING ONLY. No implementation code has been written.** This document decomposes the frozen `Module 07 Design.md` (2026-07-12, see `Module 07 Design Review.md`'s "Freeze Record") into small, independently reviewable implementation work packages, per the project owner's explicit instruction. Every work package below is derived strictly from an existing, frozen design section — no architectural redesign, no feature addition, no scope expansion beyond what §0–§28 of the frozen design already specify. Where a work package's scope is ambiguous, the ambiguity is resolved by quoting the exact design section it traces to, not by inventing new behavior.

Per the project owner's explicit instruction, this document stops after planning: no code is written, no work package is begun, and Modules 01–06 remain untouched.

## 0. How these work packages were derived

Each package maps to one design section or one tight cluster of design sections (§9's `preview_batch()`/`ExecutionEngine`/`execute_batch()` three-part architecture is the primary organizing spine — §9 itself decomposes into WP-3/WP-7/WP-9). Packages are ordered so that each one's dependencies are already implemented, independently audited, regression-tested, and merged before the next begins, exactly as the project owner's requirement specifies. No package requires touching a file owned by an already-frozen module (01–06); every package only adds new files or new functions within Module 07's own scope (`src/pipeline/execution.py`, `src/models/execution.py`, and the already-scaffolded `NotImplementedError` stubs in `src/storage/database.py`/`src/storage/runtime_io.py` explicitly marked "Module 07 territory").

**Relationship to `Governance/ENGINEERING_STANDARD.md`'s Implementation Audit (§7):** the per-package review/regression/merge cadence below is a finer-grained internal implementation discipline operating *within* lifecycle stage 4 ("Implementation," `ENGINEERING_STANDARD.md` §1). It does not replace, and is not a substitute for, the single, formal, mandatory Independent Implementation Audit (§7) that still occurs exactly once, after WP-13 is complete, before Integration Testing (stage 6) begins — identical to the process every one of Modules 01–06 actually followed. This plan adds granularity inside stage 4; it does not add, remove, reorder, or merge any of the nine lifecycle stages themselves.

### WP-9 — `execute_batch()` (batch orchestration & Layer 2 safety net)

- **Objective:** implement the outer batch loop (§9's `execute_batch()`), the fixed processing order (§7: `discovered_at` ascending, `file_id` lexicographic tie-break), the Layer 2 outer try/except safety net (§14), and the distinct, batch-level (not per-file) failure class for an unset/unreadable destination library root (§14's closing paragraph).
