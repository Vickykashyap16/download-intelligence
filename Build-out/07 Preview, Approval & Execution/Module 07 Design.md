## 0. Acceptance criteria (defined before the design below; the rest of this document must satisfy these)

### 0.1 What Module 07 guarantees

- **G1 — Never delete.** No file is ever permanently deleted by Module 07, under any code path, including failure paths. Duplicates and superseded versions are moved to `~ARCHIVE~/Duplicates/` / `~ARCHIVE~/Old Versions/` (`Rules/Folder Rules.md`), never removed. (`CLAUDE.md` non-negotiable.)
- **G2 — Log immediately, recover honestly, never silently lose an action. Resolves Design Review finding M4 — this guarantee is now stated in three explicitly distinguished parts, corrected to claim only what can actually be promised:**
  - **Guaranteed design behavior (unconditional):** every filesystem-mutating action (move, rename, archived move, destination-folder creation) is *by design* always immediately followed by its log entry — `batch_id`, `file_id`, `action`, `from`, `to`, `timestamp`, `approved_by` — written to `Runtime/Logs/action_log.jsonl` with no intentional deferral, batching, or best-effort skipping. Nothing in Module 07's own code path ever performs a move and deliberately postpones or omits its log entry.
  - **Best-effort crash recovery (a mechanism, not a promise of prevention):** if a hard crash genuinely interrupts this sequence — between the move and the log write, or between the log write and the `FileRecord` save — §13A's reconciliation procedure detects the inconsistency on the next run and repairs it, using the real, on-disk file location as the tie-breaker whenever the log and `FileRecord` disagree. This is what makes the guarantee hold *in practice*, not a claim that the interruption itself cannot occur.
  - **Unavoidable OS-level limitation (disclosed, not hidden):** true atomicity across "move the file, log the action, save the record" — three separate operations — is not achievable at the operating-system level by any application-level design. No amount of careful code ordering closes this gap by construction alone; only detection-and-repair after the fact (§13A) can.
  
  **Net guarantee, precisely stated:** no action is ever *deliberately* left unlogged, and no crash-interrupted action is ever *silently* left in an inconsistent state — every such case is detected and resolved (repaired or explicitly flagged as `error`, never guessed at) the next time Module 07 runs. This is a strictly weaker literal claim than the original wording ("never lost on a crash") but a strictly more accurate one — no guarantee this design can actually deliver has been removed; only a guarantee it could never actually have delivered has been stated honestly.
- **G3 — `review_required` is never executed, unconditionally.** A record with `tier == "review_required"` is never moved or renamed by Module 07, even if some upstream input incorrectly marks it "approved." This is a structural check inside Module 07's own execution gate, independent of whatever produced the approval decision — the same "verify at the trust boundary, don't just trust the input" philosophy `ARCHITECTURE_DECISIONS.md` decisions 8/9 already established for privacy/redaction. Directly required by `Rules/Folder Rules.md`: *"`review_required` tier → not moved at all, regardless of category."*
- **G4 — `auto` tier requires no live human decision; `approval_required` always does.** A record with `tier == "auto"` is filed with `approved_by = "auto"` and no human step. A record with `tier == "approval_required"` is filed only after an explicit decision is recorded (individual confirmation, or an explicit "approve all pre-checked" batch action) with `approved_by = "user"`. This is not invented by this design — `FileRecord.approved_by`'s own docstring already states the two-value enum `"auto" | "user"` (`src/models/file_record.py` line 89), which only makes sense if `auto`-tier filing was always intended to proceed without a human step.
- **G5 — Fully reversible via log replay, no separate backup.** Every executed batch can be undone by replaying that batch's `move_rename`/`archive_duplicate`/`archive_superseded_version` log lines with `from`/`to` swapped. No trash folder, no backup copy — the log line *is* the undo mechanism (`ARCHITECTURE_DECISIONS.md` decision 10; `Build-out/08 .../Metadata & Log Schema.md`: *"An undo replays the matching line(s) for a batch_id with from/to swapped"*).
- **G6 — One file's failure never aborts the batch.** A per-file execution failure is caught, logged (`error` action), and processing continues with the rest of the batch — the two-layer error-handling philosophy every module since Module 02 has followed (`ARCHITECTURE_DECISIONS.md` decision 18).
- **G7 — Every user correction is captured.** Any edit or rejection of a suggested category/filename/destination during approval is written to `Database/Learning/User Corrections.json`, even though v1 takes no automatic action on it (`Database/Learning/README.md`).
- **G8 — Ownership boundary respected, with one disclosed exception.** Module 07 never writes any field owned by Modules 01–06 (see §8's exhaustive list). The sole disclosed exception is `current_path`: Module 01 initializes it at discovery, and Module 07 updates it after a real, successful move — this is not an oversight, it is stated directly in `FileRecord`'s own field comment (`src/models/file_record.py` line 29–31) and in `Metadata & Log Schema.md`'s identity-fields section.
- **G9 — Never writes `Runtime/Reports/*`.** That remains exclusively Module 08's, exactly as every earlier module's contract already states for itself.
- **G10 — Deterministic given the same inputs.** Module 07 always performs the same filesystem operations and produces the same log entries (timestamps excepted) when all five of the following are held identical: (1) input metadata (the input `FileRecord` batch), (2) `Rules/*.md` business rules (in particular `Rules/Folder Rules.md`'s destination/override mapping), (3) destination library contents (the real folder state the execution-time collision check observes), (4) configuration (the resolved destination library root, §11), and (5) filesystem state relevant to collision detection (identical to (3), restated because it is the one variable most likely to legitimately differ between two runs of *this specific module* — see §7 for the full, precise statement). **If any of these five differ, a different execution plan is expected and correct — not a violation of this guarantee.** This clarification only makes explicit what was always true; it does not weaken determinism, it precisely defines the variables it is measured against, the same discipline `ARCHITECTURE_DECISIONS.md` decision 13's own module-contract precision already requires (Design Review finding M1, resolved).

### 0.2 What Module 07 explicitly does not guarantee

- **NG1 — Does not guarantee the destination library's folder structure exists in advance.** It creates a missing destination folder as a safe, idempotent side effect of execution (never destructive) — but it does not guarantee the *library root itself* is valid or reachable; an invalid/unreachable root blocks the whole batch as a configuration failure, not a per-file one (§14).
- **NG2 — Does not guarantee a `review_required` file is ever revisited automatically.** It is left in place, flagged in the preview and (per `Rules/Folder Rules.md`) in that day's `Daily Summary` — no automatic re-queue exists in v1.
- **NG3 — Does not guarantee protection against concurrent external changes to the destination library** beyond one synchronous re-check performed immediately before each individual move (§12). No distributed lock, no whole-batch transaction.
- **NG4 — Does not guarantee real-time/background operation.** v1 execution is invoked the same way every other module's batch function is — Manual or Scheduled (`README.md`'s Execution Modes table) — never a live filesystem watcher.
- **NG5 — Does not guarantee, or even define, the human-facing interaction mechanism.** *How* a person actually reviews and approves a preview (an interactive chat table vs. a generated review file) is explicitly out of this design's architectural scope — flagged as Open Decision OD-3 (§26), exactly the question the pre-build spec note itself left open ("Preferred approval interface... worth deciding once we're ready to build this step").
- **NG6 — Does not guarantee learning/auto-application of corrections.** `Database/Learning/User Corrections.json` is passive capture only in v1 (`Database/Learning/README.md`).
- **NG7 — Does not guarantee a successful undo in every case.** Undo mechanically replays the log; it cannot restore an original location that something *outside this pipeline* has since altered (the user manually deleted/moved the folder afterward), and cannot undo an action whose `reversible` field was recorded `false` (§15).

## 1. Purpose

The human checkpoint and the pipeline's only filesystem-mutating stage: show the whole batch's proposed outcome, obtain (or, for `auto` tier, forgo) a human decision, then execute — reversibly, auditable, never destructively. Module 07 is the first module in this pipeline that changes anything about the real world outside `Database/`/`Runtime/` (`Release/DEPENDENCY_DIAGRAM.md`; every module 01–06 either reads real files without mutating them, Module 01, or operates purely on already-collected `FileRecord` data, Modules 02–06).

## 2. Architectural decision: human approval is not a Provider — the three-layer Engine/Provider pattern does not apply here

`ARCHITECTURE_DECISIONS.md` decisions 4–6 define the Engine → Provider three-layer pattern specifically for **judgment-dependent** logic — a decision that requires Claude's live understanding of file *content*. Module 07's human-approval step is categorically different: it is not a judgment call about what a file *is*, it is a **decision by the file's owner** about what should happen to it. Reusing the Provider abstraction here would misrepresent a human decision as an AI judgment call, and would wrongly imply Module 07 needs a "fake human" test double in the same sense Module 02/03 need a `FakeClassificationProvider` (a stand-in for judgment quality) — what Module 07 actually needs is a stand-in for a **decision channel**, which is a strictly simpler thing to fake (a decision either arrives or it doesn't; there is no "judgment quality" gradient to fake honestly).

**Decision:** Module 07 does not have a Provider. It has an **ApprovalDecision input** — a plain data structure (not a class hierarchy, not an ABC) describing, per file, what a human (or the `auto`-tier default) decided: file as-suggested, file with an edited name/destination, or skip/reject. Module 07's own code never *asks* for this decision interactively; it *consumes* an already-produced decision set, exactly the way Module 02/03's Engine consumes a Provider's *response* without knowing how that response was obtained. **The mechanism that produces the decision set — an interactive chat table, a generated preview file the user marks up and returns — is explicitly out of this design's scope** (Open Decision OD-3, §26), matching the pre-build spec note's own unresolved "Open questions" entry. This keeps Module 07's core batch logic (preview construction, gating, execution, logging) fully unit-testable with a plain in-memory decision set, with no interactive UI dependency of any kind — the same testability goal the Provider pattern serves for judgment-dependent modules, achieved by a simpler mechanism appropriate to a fundamentally different kind of dependency.

## 3. Responsibilities

1. **Preview generation** — for every eligible record (§5), produce a structured preview row: original name/location, suggested name/destination, category, confidence score, tier, and which override (if any, per `Rules/Folder Rules.md`) applied.
2. **Approval gating** — apply the tier-based execution gate (§13): `auto` proceeds without a human decision; `approval_required` proceeds only with a recorded human decision; `review_required` never proceeds, unconditionally.
3. **Execution planning** — for every record cleared to execute, resolve its final absolute destination path (§11/§12), re-check for a real collision against the actual destination library state (not the batch-only check Module 05 already performed), and determine the exact move/rename operation to perform.
4. **Safe filesystem execution** — perform the move/rename; create a missing destination folder as a safe, idempotent step; never delete anything (G1).
5. **Progress reporting** — surface, at minimum, per-file success/failure and a batch-level summary (counts by tier, count executed, count failed, count skipped/rejected) — the same shape every earlier module's CLI summary already follows (`src/main.py`).
6. **Execution logging** — append a fully reconstructable log line for every action (§17).
7. **Rollback metadata** — ensure every executed action's log line carries everything `undo_batch()` needs to reverse it (§15).
8. **User-correction capture** — write every inline edit/rejection to `Database/Learning/User Corrections.json` (G7).
9. **`current_path`/`processed_at`/`approved_by`/`approved_at`/`reversible` maintenance** — the five fields this module owns or (for `current_path`) is the sole disclosed exception for (§8).

## 4. Responsibilities explicitly reserved for later modules (per `ARCHITECTURE_DECISIONS.md` decision 3)

Named here so a future Module 08 reviewer can verify Module 07 didn't quietly do part of Module 08's job, mirroring the discipline every earlier module's design applied to itself:

- **Report generation** (`Daily Summary`, `Weekly Summary`, `Duplicate Report`, `Storage Report`, `Runtime/Reports/*`) — Module 08's job entirely. Module 07 supplies the raw material (the action log, the final `FileRecord` states) but never writes to `Reports/` itself.
- **Any classification/metadata/duplicate/naming/confidence re-computation.** Module 07 never re-derives or second-guesses `category`, `extracted_metadata`, `duplicate_of`/`version_group_id`/`version_rank`, `suggested_name`/`suggested_destination`, or `confidence_score`/`tier` — it only *acts on* them. If a value looks wrong, that is a defect in the owning, already-frozen module, handled under `Governance/FROZEN_MODULE_CHANGE_POLICY.md`, never silently patched by Module 07.
- **Active learning from `Database/Learning/User Corrections.json`.** Passive capture only (G7/NG6) — acting on this data is `ROADMAP.md`'s later-version scope.
- **Multi-source fan-in, real-time watching.** Both remain `ROADMAP.md` Version 3 items; Module 07 operates on whatever batch it's handed, regardless of how many Sources eventually feed Module 01.

## 5. Inputs

**Receives:** `List[FileRecord]` from Module 06 — specifically, every record with `status == "discovered"`, `category is not None`, `suggested_name is not None`, and `confidence_score is not None` (i.e., `tier` is populated) — confirming the full Module 01→06 chain has already run on this record. This traces directly to Module 06's own contract: *"Module 07 reads `tier` directly off the `FileRecord` it's already holding"* (`Release/Module06/MODULE_CONTRACT.md`) and to Module 05's explicit deferral: *"the 'don't actually move a `review_required` file' behavior is Module 07's execution-time gate... Module 07's own design is responsible for actually implementing this gate"* (`Build-out/05 Naming & Destination/Module 05 Design.md` §8).

**CLI-level eligibility additionally requires `processed_at is None`** (mirroring `main.py`'s established per-module idempotency pattern, e.g. `score_confidence()`'s `confidence_score is None` check), so a record already successfully executed in a prior run is never re-selected in the first place — see §13A for the full idempotency mechanism this resolves (Design Review finding M2).

**Also receives:**
- An `ApprovalDecision` set (§2) — one entry per `approval_required`-tier record the human has actually decided on; absent entries are treated as "not yet decided" (left un-executed, not defaulted to either approve or reject — an explicit decision is required, silence is never consent, consistent with `review_required`'s own "never move without an explicit signal" posture).
- A resolved **destination library root** — an absolute filesystem path Module 07 needs to turn a `suggested_destination` (a root-relative folder string, e.g. `Finance/`) into a real, absolute target. **No configuration for this currently exists anywhere in the repository** (§11 / Open Decision OD-1) — Module 05's design explicitly confirmed `src/config/sources.yaml` configures only the *source*, never a destination root, and deferred resolving this entirely to Module 07 (`Module 05 Design.md` §9).

## 6. Outputs

**Produces:**
- The same `List[FileRecord]` handed in, enriched in place for every record that was actually executed: `processed_at`, `approved_by`, `approved_at` populated; `current_path` updated to the real, final absolute path; `reversible` set per §15's rules. Records left un-executed (`review_required`, or `approval_required` with no decision yet recorded) are returned completely unchanged — Module 07 never partially mutates a record it didn't actually move.
- One or more action-log entries per file actually touched (§17): `move_rename` or, for a duplicate/superseded-version override, `archive_duplicate`/`archive_superseded_version`; `skip`/a new value for a user-declined file (Open Decision OD-2, §26); `error` for a caught, unanticipated per-file failure.
- Real filesystem side effects: the file physically moved/renamed at its resolved destination; any newly-required destination folder created.
- New entries in `Database/Learning/User Corrections.json` for every inline edit/rejection (G7).
- No new `Database/Metadata`/`Database/FileIndex`/`Database/History` structures — Module 07 reads and writes `FileRecord`s the same way every earlier module does, and never touches `FileIndex/`/`History/` (those remain exclusively Module 04's, per its own contract).

## 7. Determinism guarantee (part of the external contract, matching every module since Module 04)

**Resolves Design Review finding M1.** Module 07's determinism guarantee is held against five explicitly controlled variables — not "the input batch" alone, because unlike every earlier module, Module 07 is the first whose output depends on live, external, mutable state (the real destination library) as well as its own prior runs' effects on that state:

1. **Identical input metadata** — the same `List[FileRecord]` batch (§5), byte-identical field values.
2. **Identical `Rules/*.md`** — in particular `Rules/Folder Rules.md`'s category → destination mapping and override table; a rule change is a business-rule change, not a code change, and is expected to change output exactly as intended when it happens.
3. **Identical destination library contents** — the real folder/file listing under the resolved library root, as it exists at the moment execution begins.
4. **Identical configuration** — the resolved destination library root (§11/OD-1) and any other run-time configuration Module 07 reads.
5. **Identical filesystem state relevant to collision detection** — restated distinctly from (3) because it is specifically what the execution-time collision re-check (§12) observes; in practice the same real-world state as (3), named separately here because it is the one variable this module's *own prior runs* directly change, making it the most operationally important of the five to hold in mind.

Given all five held constant, Module 07 always performs the same filesystem operations, in the same fixed processing order (`discovered_at` ascending, `file_id` lexicographic tie-break — the same convention `ARCHITECTURE_DECISIONS.md` has held since Module 04, continued here for log-ordering consistency and for deterministic collision-suffix assignment when two records in the same batch would otherwise land on an identical final path), and produces byte-identical log entries except for `timestamp`. **If any of the five differ between two runs, a different execution plan is expected and is not a violation of this guarantee** — e.g. re-running the same batch after a file has since been manually added to the destination folder correctly produces a different collision-suffix outcome; that is (3)/(5) changing, not nondeterminism. No record's *execution outcome* depends on any other record's outcome except the shared, deterministic collision-suffix sequence within the same destination folder in the same batch (the same kind of dependency Module 05's own within-batch collision handling already has, extended here to the *real*, execution-time collision check against the live destination folder, not just other records in the same batch). This section, together with G10 (§0.1), is the complete, precise statement of the guarantee — it is not weakened by this precision, only scoped to what it was always actually true of.

## 8. DOES NOT MODIFY (exhaustive ownership boundary)

Module 07 never sets or touches any of the following on any record — every one is left exactly as Modules 01–06 left it:

- **Module 01's own fields** — `file_id`, `source_id`, `original_name`, `original_path`, `extension`, `mime_type`, `size_bytes`, `created_at`, `modified_at`, `content_hash`, `discovered_at`, `status`, `error`, `batch_id` are all read, never rewritten.
- **Module 02's own fields** — `category`, `classification_signals` are read, never rewritten.
- **Module 03's own fields** — `extracted_metadata` is read, never rewritten.
- **Module 04's own fields** — `duplicate_of`, `version_group_id`, `version_rank`, `duplicate_signals` are read (to decide the `~ARCHIVE~/` override, §12), never rewritten. Module 07 also never writes to `Database/FileIndex/*` or `Database/History/version_history.json` — those remain exclusively Module 04's.
- **Module 05's own fields** — `suggested_name`, `suggested_destination`, `naming_signals` are read, never rewritten — including when the user edits the name/destination inline during approval (§2.1 below explains why the edited value is *not* written back onto these fields).
- **Module 06's own fields** — `confidence_score`, `confidence_breakdown`, `tier` are read, never rewritten.
- **Logging & Reporting** — `Runtime/Reports/*`. Module 07 only ever appends to `Runtime/Logs/action_log.jsonl` (G9).

**The one disclosed exception: `current_path`.** Module 01 initializes it at first discovery (equal to `original_path` at that point). Module 07 updates it — and only it, among Module 01's own field group — after a real, successful move, to the file's new absolute path. This is not a new decision introduced by this design: it is already stated in the current, frozen source (`src/models/file_record.py`'s own field comment: *"Module 01 sets it at discovery, Module 07 updates it after every move/rename"*) and in `Metadata & Log Schema.md`'s identity-fields section, which predates this design document. This design's contribution is making the exception explicit and contract-documented, mirroring exactly how Module 04's own MODULE_CONTRACT.md discloses its one side-effect exception rather than leaving it implicit in code comments alone.

**§8.1 — Why an edited suggestion is never written back onto `suggested_name`/`suggested_destination`.** If a user edits Module 05's suggested name or destination during approval, Module 07 executes the *edited* value but does not write it back onto `suggested_name`/`suggested_destination` — those fields remain Module 05's permanent, historical record of what was originally suggested (consistent with every other module's fields being a permanent record of that module's own output, e.g. `original_name`/`original_path` staying fixed forever). The *actual, final* outcome is fully captured by two other, already-existing mechanisms: `current_path` (Module 07's own field, showing exactly where the file ended up) and the action log's `to` value for that file's `move_rename` entry (showing exactly what happened, including any deviation from the suggestion). `Database/Learning/User Corrections.json` separately captures the fact that a correction occurred, its field, and both values (`Database/Learning/README.md`'s own schema already has `suggested_value`/`corrected_value` fields for exactly this). No new `FileRecord` field is required to represent "the executed name/destination" — this is a confirmed architectural decision (§27), not an open question, because the evidence for it already exists in three independent, pre-existing places.

## 9. Internal architecture

Two stages, not the three-layer Engine/Provider shape (§2 explains why that pattern doesn't fit):

```
preview_batch(records) -> List[PreviewRow]
        │  (pure, read-only — no filesystem/log/Database writes)
        ▼
execute_batch(records, decisions, library_root) -> List[FileRecord]
        │  (the only stage that mutates the filesystem, the action log, or FileRecord)
        ▼
  ExecutionEngine (per-file): gate → resolve destination → collision re-check →
                                move → log → update record
```

- **`preview_batch()`** — pure function, no side effects. Builds one `PreviewRow` per eligible record: everything a reviewer needs to decide (old/new name, old/new destination, category, confidence, tier, which `Folder Rules.md` override applies). Safe to call repeatedly, safe to call without ever executing anything.
- **`ExecutionEngine`** — the per-file decision-and-action unit, mirroring the shape of Module 04's Engine (no Provider, direct calls to storage functions) rather than Modules 02/03's Engine→Provider shape. Its steps, in fixed order, per file: (1) apply the tier gate (§13); (2) resolve the real, absolute destination path, applying the rule precedence order (§11A) and then §11's root-resolution mechanics; (3) re-check the resolved path for a real collision (§12) and apply the same collision-suffix strategy Module 05 already established for within-batch collisions, now applied against the real destination folder's actual current contents; (4) perform the move; (5) immediately append the log entry, by design never deferred or batched (G2's guaranteed-design-behavior component); (6) update the record's `current_path`/`processed_at`/`approved_by`/`approved_at`/`reversible`. A record is only ever handed to `ExecutionEngine` at all if `needs_execution(record)` (§13A) is `True` — already-executed records never reach step 1.
- **`execute_batch()`** — batch orchestration, mirroring every earlier module's `*_batch()` shape: loops `ExecutionEngine` over every eligible record in the fixed processing order (§7), wrapped in the outer per-file safety net (§14), and returns the same list handed in, enriched in place.

## 10. Execution lifecycle

1. **Preview.** `preview_batch()` runs over every eligible record (§5). Rows are grouped by tier: `auto` pre-checked (will execute without further input), `approval_required` unchecked (needs a recorded decision), `review_required` shown separately as "needs your attention" — never pre-filed, no dedicated folder (`Rules/Folder Rules.md`).
2. **Approval.** Whatever interaction mechanism is eventually chosen (Open Decision OD-3) produces an `ApprovalDecision` set: approve-as-suggested, approve-with-edit (new name and/or destination), or reject/skip, for each `approval_required` record the human actually acted on. Any edit or rejection is captured in `Database/Learning/User Corrections.json` (G7) at this step, before execution.
3. **Execution.** `execute_batch()` runs. For every record cleared by the tier gate (§13): resolve destination, re-check collision, move, log, update record (§9's `ExecutionEngine` steps).
4. **Reporting.** A batch-level summary is produced (counts by tier, executed/failed/skipped) — printed by the CLI wiring the same way every earlier module's `*()` CLI function already does (`src/main.py`), not written to `Runtime/Reports/` (§4/G9).

Records not cleared by the tier gate — every `review_required` record, and every `approval_required` record with no recorded decision yet — are returned completely unchanged (§6) and remain eligible for a future run once a decision exists.

## 11. Destination resolution

`suggested_destination` is a root-relative folder string (e.g. `Finance/`, `~ARCHIVE~/Duplicates/`) — Module 05 deliberately never resolves it to an absolute path (`Module 05 Design.md` §9, confirmed by direct inspection of `resolve_destination()` in `src/pipeline/naming.py`, which returns literal relative strings). Module 07 is where this must finally become a real filesystem path: `absolute_destination = library_root / suggested_destination / suggested_name` (or the edited equivalents, §8.1).

**Open Decision OD-1 (§26): no configuration for the destination library root currently exists anywhere in the repository.** `src/config/sources.yaml` configures only the Downloads *source* being scanned (`type: local_folder`, `path`, `source_id`) — there is no equivalent entry for where filed output should land. `Rules/Folder Rules.md` itself notes destination folders "live in the user's real, organized library... not inside this build vault," confirming the root is expected to be a real, user-specific, externally-configured path, not something Module 07 can derive or default. This design proposes (not confirms — this is a business-rule-adjacent decision, per `ENGINEERING_STANDARD.md` §2's "architectural vs. business-rule judgment call" distinction, requiring the project owner's own confirmation the same way Module 03's taxonomy did) adding a `destination_root` entry alongside the existing `sources:` list in `src/config/sources.yaml`, read once per run the same way `load_source_config()` already reads the source path — but the project owner should confirm this location/shape before implementation, not have it silently decided here.

**Note on `~ARCHIVE~/` name collision.** `suggested_destination` values of `~ARCHIVE~/Duplicates/` and `~ARCHIVE~/Old Versions/` (literal strings already produced by `src/pipeline/naming.py`) are relative to the **destination library root**, resolving to an `~ARCHIVE~/` folder *inside the user's real filed library* — a deliberately identical name to, but a physically distinct folder from, this build vault's own top-level `~ARCHIVE~/` (`CLAUDE.md`: "old versions, dead ends, replaced files" — this project's own housekeeping archive). This is not a contradiction requiring the design to stop; it is a naming coincidence between two independently-reasonable uses of the same convention in two different roots, and is called out explicitly here so no future reader conflates the two.

## 11A. Rule precedence order when multiple `Rules/Folder Rules.md` overrides could apply to the same record (resolves Design Review finding M3)

`Rules/Folder Rules.md`'s override table lists three rules — exact duplicate, superseded version, and `review_required` tier — without stating which wins when more than one applies to the same record at once. This is a real, possible case, not a hypothetical one: Module 06 reads `duplicate_signals` as part of its own scoring (`Release/Module06/MODULE_CONTRACT.md`'s DOES NOT MODIFY section: *"Module 04's own fields... `duplicate_signals` (`duplicate_signals` only)... are read"*), so a record can plausibly be both an exact duplicate or version-chain member *and* score `review_required` for an unrelated reason (e.g. a `Category.UNKNOWN` hard floor on a file Module 04 also identified as a hash match). No earlier module needed to resolve this, because none of Modules 04, 05, or 06 physically execute anything — Module 07 is the first module for which the answer has a real, physical consequence.

**Confirmed precedence order, checked in this fixed sequence, before any destination is resolved for a given record:**

1. **`tier == "review_required"` → leave the record completely unchanged.** Checked first, absolute, no exception. This is already how §13's execution gate is structured — `tier` is evaluated before `ExecutionEngine` ever reaches destination resolution — and this section makes that ordering an explicit, binding precedence rule rather than an implicit side effect of the pseudocode's own structure.
2. *(only reached if tier != "review_required")* **exact duplicate** (`duplicate_of is not None`) → archive to `~ARCHIVE~/Duplicates/`.
3. *(only reached if step 2 did not apply)* **superseded version** (`version_rank == "superseded"`) → archive to `~ARCHIVE~/Old Versions/`.
4. *(only reached if none of the above apply)* **normal category → destination mapping** (`Rules/Folder Rules.md`'s table).

**Why this order, and why it preserves the pipeline's safety guarantees:**

- `review_required` is checked first and absolutely because it is the pipeline's only signal that a human explicitly needs to look at this record before anything physical happens to it — and that signal's entire purpose (`Rules/Confidence Rules.md`, `Module 06 Design.md`) is to catch cases where some upstream module's confidence in its own conclusion — about *any* aspect of the record, including its duplicate/version status — is low enough to warrant a second opinion. Treating exact-duplicate detection as an exception that bypasses `review_required` would silently assume Module 04's own signal is always trustworthy enough to override the very mechanism designed to catch cases where trust is insufficient — an assumption this design deliberately declines to make on the project's behalf.
- This ordering costs nothing in the case that actually matters (a genuine, uncontested exact duplicate): the correct disposition is **delayed, never lost**. A record correctly identified as a duplicate that also happens to be `review_required` for an unrelated reason is left in place, visible in the preview's "needs your attention" section (§10) and (once Module 08 exists) that day's Daily Summary. Once a human clears the review — the record's `tier` is not something Module 07 recomputes, so this requires either a future Module 06 re-score after some other change, or a manual override mechanism outside this design's v1 scope — a normal future run then applies the duplicate override exactly as it would have originally. Nothing about the record's eventual, correct disposition is lost by deferring it; only the timing changes.
- This is the same "prefer inaction over an autonomous guess" posture the project's own stated goal already commits to (`README.md`: *"Never act with full autonomy on uncertain calls"*) — applied here to the one case where two rules could otherwise disagree, resolved in favor of the rule that leaves the file exactly where a human can find it, rather than the rule that moves it (even reversibly) without them having looked first.

This precedence order is a confirmed decision (§27), not an open one — the project owner has directly resolved it as part of approving this design correction, rather than leaving it as a further open decision for later.

**`Rules/Folder Rules.md` itself is a living business-rules document maintained outside this design's authority (`CLAUDE.md`) and is not edited by this document** — updating it to state this precedence explicitly is a required follow-up action at implementation time, mirroring `Module 05 Design.md` §10's identical treatment of its own naming-template mismatches ("`Rules/Naming Rules.md` itself is a living business-rules document... not edited by this document — updating it... is a required follow-up action at implementation time, not performed here"). The same documentation discipline applies here: this design resolves the ambiguity architecturally; `Rules/Folder Rules.md`'s own override table should be updated to match at Module 07's implementation time, not left silently stale.

## 12. Filesystem execution mechanics

- **Move + rename in one step**, per file, per the pre-build spec note's own original intent (`07 Preview, Approval & Execution.md`) — never a copy-then-delete (a copy-then-delete window would violate G1's spirit if the process crashed between the two).
- **Destination folder creation is safe and idempotent**, performed as part of executing the first file that needs it — never treated as a destructive or notable action requiring its own approval step (matches the pre-build note's "Edge cases" section verbatim).
- **Real, execution-time collision re-check**, immediately before each individual move. Module 05's own collision detection is explicitly scoped to *within the current batch only* (`Module 05 Design.md` §9) — the *authoritative* check against the real, current destination-folder contents is Module 07's responsibility, performed synchronously right before that specific move (not pre-computed at preview time, since the destination library's true state can only be trusted at the moment of the actual write — the same reasoning Module 05's design already used to justify deferring this check). On a real collision, the same collision-suffix strategy Module 05 already established (`_2`, `_3`, ...) is applied, extended to check against the real folder rather than only sibling batch records.
- **A partial batch failure never aborts the rest of the batch** (G6/I4) — a per-file failure (e.g. a permissions error) is caught by the outer safety net (§14), logged as an `error` action, and the batch continues with the next file.

## 13. Invariants (execution gate, restated precisely from §0.4)

```
for record in batch (fixed order, §7):
    if record.tier == "review_required":
        leave record completely unchanged            # I2 — unconditional, no exception
    elif record.tier == "auto":
        execute with approved_by = "auto"             # G4
    elif record.tier == "approval_required":
        if a recorded ApprovalDecision exists for this record:
            if decision == reject/skip:
                log the decline (Open Decision OD-2), leave record unchanged
            else:
                execute with approved_by = "user"      # G4, using edited name/destination if provided
        else:
            leave record completely unchanged          # absent decision is never treated as consent
```

This gate is evaluated **inside `ExecutionEngine`, from `FileRecord.tier` directly**, never from a cached/precomputed "is this row checked" boolean handed in from the preview stage — the same "verify at the trust boundary" discipline `ARCHITECTURE_DECISIONS.md` decisions 8/9 apply to privacy/redaction, now applied to the one condition (G3/I2) that must never be bypassable by any upstream mistake, including a hypothetical bug in whatever UI eventually implements OD-3.

## 13A. Idempotency mechanism (resolves Design Review finding M2)

**Module 07 must never execute the same approved operation twice after a successful execution.** This section defines the mechanism, mirroring Module 04's own established precedent for exactly this class of gap: Module 04's original idempotency check (a bare field-null check) was insufficient in practice and required a dedicated `needs_duplicate_detection()` function (post-freeze correction #2) — Module 07 adopts a dedicated function from the start rather than repeating that lesson a second time.

**How an already-executed record is recognized.** A record is "already executed" for a given batch iff `record.processed_at is not None`. This is checked by a dedicated function, `needs_execution(record)` (mirroring `needs_duplicate_detection()`'s naming and purpose exactly), not a bare inline check scattered across call sites — `needs_execution(record)` returns `False` when `processed_at is not None`, regardless of tier, and is applied at the CLI-level eligibility filter (§5) *before* a record ever reaches `preview_batch()` or `ExecutionEngine`. An already-executed record therefore never re-enters the gate (§13) at all — it is filtered out one layer earlier, the same way Module 06's own `confidence_score is None` CLI check keeps an already-scored record from re-entering `score_confidence_batch()`.

**Fields checked:** `processed_at` alone is the recognition signal (mirroring the precedent that a single, unambiguous "has this module's own terminal output been set" field is sufficient when — unlike Module 04's `duplicate_of`/`version_group_id`/`version_rank`, which can legitimately stay `null` forever for a record with no duplicate — `processed_at` has no legitimate permanent-`null`-after-processing case: every record Module 07 successfully executes gets a real timestamp, and every record it doesn't execute keeps `processed_at == None` indefinitely until it is executed). `approved_by`/`approved_at`/`current_path` are set atomically alongside `processed_at` (§9's `ExecutionEngine` step 6) and are never checked independently for idempotency purposes — `processed_at` alone is authoritative.

**Expected behavior on a second `execute_batch()` call (no interruption):** already-executed records are excluded before selection (see above) — the second call simply has a smaller (or empty) eligible set. No re-move, no re-log, no re-write of `FileRecord`. This is true whether the second call happens moments later or after an arbitrary delay.

**Expected behavior after interruption or partial execution (a crash mid-batch):** true atomicity of "move the file, log the action, save the record" across all three operations is not achievable at the OS level — a crash can occur between any two of them. This is handled, not assumed away, via `Runtime/Temp/<batch_id>/plan.json` (§18) as a reconciliation record, resolved in this order on the next run before any new execution is attempted:

1. For each file listed in a leftover `plan.json` (a prior run's batch that never reached a clean `clear_batch_temp()`), check whether `Runtime/Logs/action_log.jsonl` has a completed `move_rename`/`archive_duplicate`/`archive_superseded_version` entry for that `file_id` matching the plan's intended `to` path.
2. If a matching log entry exists **and** `FileRecord.processed_at` is already set → the operation completed fully before the crash; nothing to reconcile, the plan entry is discarded as already-terminal.
3. If **no** matching log entry exists **and** `processed_at` is still `None` → the operation was never attempted (or the plan was staged but execution hadn't reached this file yet); safe to retry from scratch on the next `execute_batch()` call, no special handling needed.
4. If a matching log entry exists **but** `processed_at` is still `None` → the crash happened between the log write and the `save_file_record()` call — exactly G2's disclosed OS-level-limitation case, resolved here by G2's best-effort-crash-recovery component. This is treated as a Layer-1 anticipated recovery case, never guessed at silently: the actual on-disk file location is checked against the log's `to` path. If the file is really there, `FileRecord` is repaired to match reality: `processed_at`/`approved_by`/`approved_at`/`current_path` are set from the log entry's own recorded values (the log is authoritative here, since it's the artifact closest in time to the actual filesystem write); **`reversible` is conservatively repaired to `false` unless the log entry's own `details` positively indicate the move was a clean, non-collision-suffixed, non-`~ARCHIVE~/` operation matching the original suggestion (§15's two `reversible = false` trigger conditions) — never silently defaulted to `true`, since `true` is the unsafe direction to guess wrong in (it would suppress §15's "a human should double-check before undo" safeguard exactly in the case most likely to need it).** The record is now correctly recognized as executed. If the file is *not* there (the log line was written but the move itself failed or didn't complete), this is logged as an `error` with an explicit "inconsistent state, log claims completion but disk disagrees" diagnostic, and the record is left with `processed_at == None` so it remains eligible for a clean retry — never silently assumed complete when the disk disagrees with the log.
5. Once every entry in a leftover `plan.json` is reconciled to one of the above terminal states, `clear_batch_temp()` runs and the batch proceeds normally with whatever remains genuinely unexecuted.

**Interaction with rollback metadata:** `undo_batch()`/single-action undo (§15) is the idempotency mechanism's exact inverse — after a successful undo of a given record's execution, `processed_at`, `approved_by`, and `approved_at` are reset to `None`, `current_path` is restored to its pre-move value, and `reversible` is reset to its dataclass default (`true`). This is what makes `needs_execution(record)` correctly recognize the record as eligible again on a future run — without this reset, an undone record would be permanently, incorrectly treated as "already executed" and could never be re-previewed, re-approved, or re-filed. The `undo` log entry itself is never retracted (append-only, per §15) — only the `FileRecord`'s own state is rolled back, exactly mirroring how a fresh execution and an undone-then-re-executed record are indistinguishable in the log's own append-only history, which is the intended, auditable behavior.

## 14. Failure model

Two layers, per the established project-wide philosophy (`ARCHITECTURE_DECISIONS.md` decision 18), applied identically to Module 07's own new failure classes:

**Layer 1 — anticipated failures, handled inside `ExecutionEngine` with a named, honest result (never a silent guess, never fabricated success):**
- Destination-folder creation fails (permissions) → the specific file's move is not attempted; logged as `error` with a sanitized `error_detail`; record left unchanged; batch continues.
- A real, execution-time collision is detected → resolved via the collision-suffix strategy (§12), not treated as a failure at all unless the suffixed name *also* collides after a bounded number of attempts, in which case it degrades to a logged `error` (never an infinite loop, never a silent overwrite).
- The move itself fails partway (an OS-level error mid-operation, e.g. a full destination volume) → logged as `error` with whatever diagnostic is available; the record's `current_path` is only ever updated *after* a move is confirmed complete, so a failed/partial move never leaves `FileRecord` claiming a location the file isn't actually at.
- **A process crash between the log write and the `FileRecord` save** (G2's disclosed OS-level limitation) → reconciled on the next run via §13A's five-step procedure, never silently assumed complete and never silently retried without first checking the real disk state against the log.

**Layer 2 — the outer, batch-orchestration safety net** (`execute_batch()`'s own try/except around each `ExecutionEngine` call): catches anything genuinely unanticipated, logs an `error` action, and continues to the next file — never lets one file's unexpected exception abort the whole batch (G6/I4).

**A distinct, batch-level (not per-file) failure class:** the destination library root itself (§11) being unset, non-existent, or unreadable. Unlike a per-file destination-folder-creation failure, this blocks the *entire* batch before any file is attempted — there is no meaningful per-file fallback for "the whole library is unreachable," and attempting one anyway risks silently filing files into the wrong place. This is reported clearly and the batch is not partially executed against a root that can't be trusted.

## 15. Rollback strategy

**No separate trash/backup mechanism.** The action log line for each executed action *is* the complete undo record (G5), per the architecture this project already committed to (`ARCHITECTURE_DECISIONS.md` decision 10) and the pre-existing, already-scaffolded `undo_batch(batch_id)` stub (`src/storage/runtime_io.py`, currently `raise NotImplementedError("Module 07 ... territory")`).

- **Single-action undo:** replay one `move_rename`/`archive_duplicate`/`archive_superseded_version` log line with `from`/`to` swapped — move the file back, then update that record's `current_path` back to the original value.
- **Batch undo:** replay every action-log line for a given `batch_id`, `from`/`to` swapped, in **reverse chronological order** (undoing the most recent action first) — necessary because a version-chain archive and a subsequent rename of the "new latest" file could otherwise be replayed out of order and produce a different final state than simply reversing time.
- **`reversible` field semantics (confirmed, §27):** defaults to `true` (`FileRecord`'s own dataclass default). Module 07 sets it to `false` on a specific, executed action only when undo is known, at execution time, to be unsafe to attempt automatically — concretely: (a) the move landed the file at a path that already existed as something *other* than the pristine original location, requiring a collision-suffix that means "moving it back" isn't a pure reverse of one clean operation, or (b) the original location was itself inside `~ARCHIVE~/` (undoing an archive-of-a-duplicate could resurrect a name collision with the file that superseded it). `reversible = false` never means "not logged" — the log entry always exists in full (G2); it means "replaying this specific entry mechanically is not guaranteed safe without a human double-checking first," and `undo_batch()` surfaces this rather than silently attempting it.
- **Interaction with the idempotency mechanism (§13A):** a successful undo resets `processed_at`, `approved_by`, `approved_at` to `None`, restores `current_path`, and resets `reversible` to its default — the full detail lives in §13A (since it is that section's own necessary complement) rather than being duplicated here; this bullet exists so a reader of this section alone isn't left thinking undo only touches the filesystem and the log.
- **What undo does *not* attempt to reverse:** `Database/FileIndex/*`/`Database/History/version_history.json` entries (Module 04's own structures, populated at Module 04's processing time, not re-triggered by Module 07's execution — undoing a *move* doesn't change what Module 04 already concluded about duplicate/version relationships, so there is nothing here to roll back) and `Database/Learning/User Corrections.json` entries (a correction is a historical fact about what a human decided, independent of whether the resulting file operation was later undone — never retracted by an undo, consistent with the project's own "never rewrite history" documentation convention applied here to data, not just docs).

## 17. Interaction with the action log

Uses the existing `append_action_log()` (`src/storage/runtime_io.py`) exactly as every module since Module 01 does — no new logging mechanism. Action values used:

## 22. Security considerations

- **The only module so far that performs a real, non-reversible-by-copy filesystem write outside `Database/`/`Runtime/`** — the highest-stakes trust boundary in the pipeline to date. The tier gate (§13) is the single most safety-critical piece of code in this module and must be verified adversarially during both integration testing and UAT (a deliberately forged "approved" decision on a `review_required` record must still never execute — I2), not merely tested against well-behaved input.
- **Path construction must never allow escape outside the destination library root** (e.g. a `suggested_name`/`suggested_destination` or a user-supplied edit containing `../` sequences must be rejected or sanitized before being joined into an absolute path) — a new adversarial-input class this module introduces that no earlier module needed to consider, since no earlier module ever turned a string into a real filesystem write location.
- **No code execution risk from file content** — Module 07 never opens or reads a file's *content*, only moves it by path, so the file-content-derived code-execution class of risk `ENGINEERING_STANDARD.md` §20 asks every module to consider does not apply here in the way it does to Modules 02/03's provider calls.
- **Undo must not be exploitable to relocate a file outside the pipeline's known locations** — the same path-construction discipline applies symmetrically to `undo_batch()`'s replay path.

## 23. Performance expectations

A measured (not estimated) number for the real Module 01→07 chain against `Tests/Large Batch/` or equivalent is required at release time (§0.3), following the exact precedent Module 05's F3 and Module 06's PCV-check-12 finding both already established for this project. Module 07's own per-file cost is expected to be dominated by the actual filesystem move (an OS-level operation, not something this pipeline's own code controls the cost of) rather than by `ExecutionEngine`'s own decision logic, which is O(1) per file plus a bounded collision-suffix retry loop — a different cost profile than any earlier module's, worth measuring rather than assuming comparable to Module 06's 40.122-second baseline.

## 25. Auditability requirements

- Every executed action fully reconstructable from `Runtime/Logs/action_log.jsonl` alone, without needing to re-run the pipeline (`ARCHITECTURE_DECISIONS.md` decision 10) — `move_rename`'s `details` must disclose whether the executed name/destination differed from Module 05's original suggestion (§8.1), so a future reader of the log doesn't need to cross-reference `Database/Learning/` to know a deviation occurred.
- `approved_by`/`approved_at` persisted directly on `FileRecord` (not only in the log) — these are Module 07's own owned fields specifically so a reader of `Database/Metadata/metadata_store.json` alone, without the log, can answer "was this auto-filed or human-approved, and when" for any given file.
- Every user correction captured to `Database/Learning/User Corrections.json` regardless of whether v1 acts on it (G7) — the same "collect real data now, even before the feature that consumes it exists" discipline that field's own README already states as its purpose.
- `reversible = false` is itself an auditable, visible fact on the record (§15) — never a silent internal flag invisible to a future reader deciding whether to trust an automated undo.

## 27. Confirmed architectural decisions (resolved by this design, not open)

- **The three-layer Engine/Provider pattern does not apply to Module 07** — replaced by a two-stage `preview_batch()`/`execute_batch()` shape with a plain `ApprovalDecision` input rather than a Provider ABC (§2). This is itself a candidate for a new, numbered entry in `Governance/ARCHITECTURE_DECISIONS.md` (the next sequential number after decision 19) once this design is reviewed and frozen — not added preemptively here, since `ARCHITECTURE_DECISIONS.md` records decisions already made and frozen, not proposals still in design review (per that document's own header).
- **`current_path` is Module 07's one disclosed exception to the ownership-boundary rule** — already true in the current, frozen source and schema doc (§8), formalized as an explicit contract statement by this design rather than left as an implicit code comment.
- **An edited suggestion is never written back onto `suggested_name`/`suggested_destination`** — `current_path` plus the action log's `to` value plus `Database/Learning/User Corrections.json` together fully capture the executed outcome without needing a new `FileRecord` field (§8.1).
- **`auto`-tier records execute without a human decision step; `approval_required` always requires one** — directly evidenced by `FileRecord.approved_by`'s own pre-existing `"auto" | "user"` enum (§0.1 G4), not invented by this design.
- **Undo replays in reverse chronological order at the batch level** — required for correctness once a version-chain archive and a subsequent rename can both appear in the same batch (§15).
- **Scheduled/unattended execution mode only ever executes `auto`-tier records.** `approval_required` and `review_required` records are always left in place regardless of execution mode (`README.md`'s Execution Modes table lists Scheduled as "same logic as Manual, triggered on a timer" — with no human present, only the tier that by definition needs no human decision can ever proceed) — this makes Scheduled mode safe by construction and requires no special-casing in `ExecutionEngine` beyond the tier gate (§13) it already has.
- **Determinism is held against five explicitly named controlled variables** (input metadata, `Rules/*.md`, destination library contents, configuration, and collision-relevant filesystem state), not "the input batch" alone — a difference from every earlier module's own determinism guarantee because Module 07 is the first whose output legitimately depends on live, mutable, external state (§7, resolves Design Review finding M1).
- **`needs_execution(record)` — a dedicated idempotency function checking `processed_at is None`** is the sole recognition mechanism for "already executed," applied at CLI-level selection, mirroring Module 04's own `needs_duplicate_detection()` precedent rather than a bare inline field check (§13A, resolves Design Review finding M2). A crash mid-batch is recovered via a fixed, five-step reconciliation procedure against `Runtime/Temp/<batch_id>/plan.json` and the action log, with the real filesystem as the tie-breaker whenever the log and `FileRecord` disagree (§13A).
- **Rule precedence order when `review_required` and a duplicate/version override could both apply: `review_required` is checked first and is absolute** — a duplicate/superseded-version archive move never happens for a `review_required` record; the correct disposition is delayed until a human clears the review, never lost (§11A, resolves Design Review finding M3).
- **G2 is stated in three explicitly distinguished parts — guaranteed design behavior, best-effort crash recovery, and a disclosed, unavoidable OS-level limitation** — rather than a single unqualified claim (§0.1/§13A, resolves Design Review finding M4). No guarantee this design can actually deliver was removed; only a guarantee it could never have literally delivered was restated honestly.
- **Crash-repair conservatively defaults a record's `reversible` to `false`** unless the log entry's own `details` positively indicate a clean, non-collision-suffixed, non-`~ARCHIVE~/` operation — the safe-to-guess-wrong direction, consistent with `reversible = false`'s own purpose of prompting a human double-check (§13A, resolves Design Review finding L3/round-2).
- **`Rules/Folder Rules.md` itself must be updated at Module 07's implementation time to state the §11A precedence order explicitly** — this design resolves the ambiguity architecturally but does not edit the `Rules/` document itself, mirroring `Module 05 Design.md` §10's identical treatment of its own naming-template mismatches (§11A, resolves Design Review finding L5/round-2).

## Addendum — WP-2 Ambiguity Resolution: Edited Destination vs. Archive Override (2026-07-12)

WP-2's own independent implementation audit (`resolve_destination_path()`, `src/pipeline/execution.py`) surfaced a Medium finding: the frozen design never explicitly states whether a human's `APPROVE_WITH_EDIT` destination edit can override the mandatory `~ARCHIVE~/Duplicates/`/`~ARCHIVE~/Old Versions/` placement for an exact-duplicate or superseded-version record. §8.1, §10, and §11A each discuss edits and overrides separately and never state their interaction. Per the project owner's explicit direction, this addendum resolves that ambiguity by clarification, without assuming the implementation choice already made (edit never overrides archive) was correct.

**Question.** Can a human-approved edited destination ever override the mandatory archive placement for an exact duplicate or a superseded version?

**Interpretations considered:**

- **(A) Never** — an edit is silently ignored whenever `override_type` is `exact_duplicate`/`superseded_version`; only an edited *name* is honored for those records. This is what WP-2 actually implemented.
- **(B) Always** — an edited destination is honored uniformly across all four `resolve_precedence()` outcomes except `review_required` (which never reaches destination resolution at all, §11A step 1), with no special case for the archive overrides.
- **(C) Conditionally** — some new gating mechanism (e.g. a required "confirm override" flag) is introduced to allow the edit only with extra friction.

(C) is rejected outright: it requires a new field/mechanism not present anywhere in the frozen design (§8.1's `ApprovalDecision` has no such flag), which is scope expansion, not interpretation — inventing a new decision structure is exactly what this task's own instruction ("do not implement any additional production code," and Frozen Module Change Policy's minimum-clarification-only latitude) forecloses.

**Comparison of (A) vs. (B):**

- **Frozen Module 07 design.** §8.1's own text is unqualified: "Module 07 executes the *edited* value" — no carve-out for a record whose suggested destination happens to be an archive path. §11A's "absolute, no exception" language is used for exactly one of its four steps — step 1, `review_required` — and deliberately not repeated for steps 2/3 (the archive overrides); the section's own reasoning (quoting `ARCHITECTURE_DECISIONS.md` decisions 8/9's "verify at the trust boundary" philosophy) is about Module 07 not blindly trusting *Module 05's automatically-computed* `suggested_destination` string — it says nothing about constraining a *human's* later, explicit choice. §11A's own closing sentence ("checked... before any destination is resolved") positions it as selecting which default rule/path constant applies prior to §11's resolution mechanics — not as a restriction on what §11/§8.1's edit-substitution step may subsequently do. Reading §11A as silently overriding §8.1 for two of its four outcomes is an interpretation this document adds, not one the frozen text states. (B) is the more literal reading; (A) requires inferring an unstated exception.
- **`Rules/Folder Rules.md`.** The override table's "never the normal destination" language is written entirely in the vocabulary of Module 05's automatic category→folder mapping — the document never mentions Module 07, approval, or human edits at all (it predates the approval mechanism's own detailed design). Read in its own context, "never" most naturally means "never *automatically*, via the mapping table," not "never, even overriding an explicit, informed human choice made during the review step this document doesn't discuss."
- **Confidence/review philosophy (`Rules/Confidence Rules.md`, `README.md`).** The entire reason `approval_required` exists as a tier distinct from `auto` is that a human is trusted to catch what the automatic scoring got wrong, case by case. A record only reaches an editable approval row if it is `approval_required` (or `auto`, which never presents an edit opportunity at all — `review_required` is excluded absolutely by §11A step 1 regardless of this clarification). Disallowing the one class of human correction most likely to matter here — "this was flagged as a duplicate/superseded version but that's wrong, file it normally" — silently defeats the reason that tier exists. (B) is consistent with this philosophy; (A) works against it.
- **Rollback semantics (§15).** Neutral. §15's two `reversible = false` triggers are about collision-suffixed paths and archive-*origin* moves, not about whether an edited-away-from-archive destination was chosen at approval time. An edit honored under (B) that lands cleanly, uncollided, outside `~ARCHIVE~/` remains `reversible = true` under the existing rule — no new rollback gap is introduced.
- **User expectations.** A human who explicitly types a new destination during `APPROVE_WITH_EDIT` and has that edit silently discarded — with no error, no warning, the file filed to the archive path anyway — is a more surprising, harder-to-diagnose outcome than honoring the edit. (B) matches the plain meaning of "approve **with edit**"; (A) makes the edit field misleading for two of four possible record states without saying so anywhere the human would see it.
- **Module 08 compatibility.** Neutral either way — Module 08 reads whatever `to` value the action log actually records; nothing in its design depends on which interpretation governs how that value was chosen.
- **The one dimension favoring (A):** none identified beyond restating Folder Rules.md's literal words out of context (already addressed above) — no comparison dimension the project owner named actually supports (A) once each is examined on its own terms rather than assumed.

**Recommendation.** Interpretation **(B)** is both the smaller implementation (a single, uniform edited-destination substitution rule, no per-override-type exception to carve out or maintain) and the one that preserves every safety guarantee actually stated in the frozen design: `review_required` remains completely, unconditionally protected (§11A step 1, untouched by this clarification — no destination is ever resolved for such a record, edited or not); every edit remains fully logged (`Database/Learning/User Corrections.json`, §8.1/G7) and fully reversible under the existing, unchanged §15 rules; nothing about this clarification permits an *automatic* process to place a file outside `~ARCHIVE~/Duplicates/`/`~ARCHIVE~/Old Versions/` — only an explicit, human, per-file `APPROVE_WITH_EDIT` decision can, which is precisely the category of action §10/§8.1 already generally permit and precisely the category of action this project's "never act with full autonomy on uncertain calls" guarantee is not about restricting.

**Clarification (append-only, per `Governance/FROZEN_MODULE_CHANGE_POLICY.md`).** §8.1's edited-destination substitution applies uniformly regardless of `resolve_precedence()`'s outcome, with the sole exception of `review_required` (§11A step 1, which precedes and blocks all destination resolution, edited or not). An edited destination supplied via `APPROVE_WITH_EDIT` is honored even when the record's override type is `exact_duplicate` or `superseded_version`. An edited *name* was never in question — §8.1 already establishes it is always honored regardless of override type. Full record: `Governance/ARCHITECTURE_DECISIONS.md` decision 23.

**Not performed by this addendum, flagged as a required follow-up:**
- `src/pipeline/execution.py`'s current `resolve_destination_path()` implements interpretation (A) — it deliberately ignores `edited_destination` for `exact_duplicate`/`superseded_version`, per its own docstring's self-flagged judgment call. That implementation, and the two WP-2 tests that assert (A)'s behavior (`src/pipeline/test_execution.py`, the `edited_destination` "explicitly NOT honored" cases for `exact_duplicate`/`superseded_version`), now encode the superseded interpretation and require correction. This is production code and test code — out of scope for this clarification task per explicit instruction ("do not implement any additional production code") — and is deferred to a dedicated, separately-approved corrective step before any later work package (WP-7 or wherever real `ApprovalDecision`-driven edits are first wired to execution) depends on this function's behavior.
- `Rules/Folder Rules.md`'s override table wording is not edited by this addendum — per `CLAUDE.md` and this document's own established precedent (§11A's closing paragraph), it is a living business-rules document maintained outside this design's authority. Whether to add a clarifying note there distinguishing "automatic mapping" from "human approval-time override" is the project owner's call, not made here.

## Addendum — WP-9 Governing Rule: Incremental `plan.json` Staging (2026-07-12)

A final architecture integration review covering WP-1 through WP-8 as a composed system (`Module 07 Implementation Plan.md`, performed before WP-9 begins) identified a genuine divergence risk in §18's staging requirement, resolved by the project owner as a governing implementation rule for WP-9. Per this project's "never rewrite history" convention, §9/§12/§18's own already-frozen text is left completely unedited; this addendum records the resolution and where its full reasoning lives.

**Finding.** §18 requires a batch's planned operations to be staged to `Runtime/Temp/<batch_id>/plan.json` "before executing any file in a batch." Read as a single, whole-batch, two-pass operation (resolve every eligible record's destination and collision state first, stage the complete plan once, then execute every record), this creates a real divergence risk: `ExecutionEngine.execute_file()` (WP-7) always independently re-derives destination and the real, execution-time collision check itself, by design (`ARCHITECTURE_DECISIONS.md` decisions 8/9) — it never accepts a pre-resolved path. Two records sharing a destination folder and name would have their collision-suffix outcome depend on processing order (already disclosed, §7) — under a two-pass design, the `to` path staged in the resolve-everything pass could diverge from the `to` path actually used at move time. `reconcile_batch()` (WP-8, §13A) matches a leftover plan entry against the action log by exact `to`-string equality, so a crash after such a divergence would misclassify a genuinely completed operation as `SAFE_TO_RETRY`, risking a duplicate file on retry.

**Decision (governing rule for WP-9, full record: `Governance/ARCHITECTURE_DECISIONS.md` decision 24).** `execute_batch()` stages each record's `plan.json` entry immediately before executing that specific record — never a separate whole-batch resolve-then-execute phase. This guarantees staged and executed `to` values are always identical, satisfies §18's literal requirement, and requires no change to any already-implemented WP-1–8 code (`write_batch_plan()` already supports being called repeatedly with a growing list).

**Not performed by this addendum:** no code was written; WP-9 itself remains unimplemented. `Module 07 Implementation Plan.md`'s WP-9 entry is updated separately with this rule as an explicit implementation constraint.

No implementation code was written to produce this addendum. No frozen module (01–06) was touched, and no already-implemented WP-1–8 code was touched. Module 07's design remains **Frozen** — this addendum records implementation-time guidance for a not-yet-started work package; it does not revise §9, §12, §13A, or §18's own already-frozen wording, all of which remain completely unedited above.
