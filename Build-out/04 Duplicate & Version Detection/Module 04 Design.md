# Module 04 (Duplicate & Version Detection) — Design

Pre-implementation design for Module 04, produced under `Governance/ENGINEERING_STANDARD.md`'s lifecycle (Design → Independent Architecture Review → Design Refinement → Design Freeze → ... ). No implementation code has been written. This is the Design Refinement pass: every item originally flagged as a business-rule judgment call requiring the project owner's confirmation (`ENGINEERING_STANDARD.md` §2) has now been reviewed and decided — see the closing "Confirmed design decisions" section.

There is no pre-existing pointer document for Module 04 (unlike Module 03, which superseded a short pre-design note) — the only prior context is `Build-out/00 Pre-build context & examples/Pre-build context.md`'s original brief (§8, feature 4/5; its tools table), the `duplicate_of`/`version_group_id`/`version_rank` fields already reserved on `FileRecord`, and the `Database/FileIndex/`, `Database/History/`, `core/hashing.py`, and `storage/database.py` stub functions already scaffolded with Module 04's name on them. This design formalizes what those stubs anticipated.

---

## 1. Purpose

Module 04 answers one question for every file Module 01–03 have already discovered, classified, and metadata-extracted: **"have we seen this before, in some form?"** It detects two distinct relationships between a newly-processed file and everything already known to the pipeline:

- **Exact duplication** — byte-identical content already filed under a different `file_id`.
- **Versioning** — different content that is recognizably a later or earlier copy of the same underlying document (e.g. `Resume_v8.pdf` vs. `Resume_v9.pdf`).

Module 04 detects and records these relationships. It does not decide what to do about them (no destination routing, no confidence scoring, no file moves) — that is explicitly reserved for Modules 05–07 (§28).

## 2. Responsibilities

Module 04 owns:
- Exact-duplicate detection via content hash, across every discovered file regardless of category.
- Near-duplicate detection via perceptual hash, scoped to image-family categories (Image, Screenshot).
- Version-chain detection via filename similarity plus version/date signal comparison, scoped to categories where "an updated copy of the same document" is a meaningful concept (§9).
- Populating `duplicate_of`, `version_group_id`, `version_rank`, and a new `duplicate_signals` field (§17) on every record it processes.
- Implementing `Database/FileIndex/` (`hash_index.json`, `phash_index.json`, `name_index.json`) and `Database/History/version_history.json` — currently stub-only.
- Logging one `detect_duplicates_and_versions` action-log entry per file it processes (§18).

Module 04 must never:
- Move, rename, delete, or archive a file on disk. It only ever reads file bytes (for hashing) and metadata (for comparison); it has no filesystem-write role at all in this pipeline. (Module 07's job.)
- Decide a destination folder, even though `Rules/Folder Rules.md` already has override routes for duplicates/superseded versions (`~ARCHIVE~/Duplicates/`, `~ARCHIVE~/Old Versions/`) — Module 04 supplies the *signal* (`duplicate_of`/`version_rank`); Module 05 applies the *routing rule*. (Module 05's job.)
- Compute a confidence score or tier, even though its own signals directly feed two of `Rules/Confidence Rules.md`'s existing deductions (near-duplicate/fuzzy match, version-date conflict) and two of its hard floors. (Module 06's job.)
- Touch any field owned by Module 01, 02, or 03.

## 3. Inputs

**Receives:** `List[FileRecord]` from Module 03 — specifically, every record with `status == "discovered"`. Unlike Modules 02→03's handoff, Module 04 does **not** gate its entire scope on category being set and non-`Unknown` — see §9's explicit scoping rationale: exact-duplicate detection is content-based and runs on every discovered record (including `Category.UNKNOWN` and `Category.ARCHIVE`/`APPLICATION`/`VIDEO`/`AUDIO`), while near-duplicate and version-chain detection are scoped to specific categories.

**Also receives (internally):** nothing beyond the record list and the accumulated `Database/FileIndex/`/`Database/History/` state built up across all prior runs. Unlike Modules 02/03, Module 04 has **no provider dependency at all** — see §14 for why.

## 4. Outputs

**Produces:**
- The same `List[FileRecord]` handed in, enriched in place (mirrors `classify_batch()`/`extract_metadata_batch()`'s shape) — **with one disclosed exception**: Module 04 may also update the `version_group_id` and/or `version_rank` fields of a *different*, earlier-processed record already in the store, as a bounded side effect of detecting a version-chain relationship involving it (§7, §16; post-freeze correction, see note below). This is the one place in the pipeline so far where a module's own batch call can modify a record outside the list it was directly handed — explicitly named here because it is a genuine departure from every earlier module's strictly-in-place-on-the-input-list behavior, not an oversight.
- One `detect_duplicates_and_versions` action-log entry per file processed in `Runtime/Logs/action_log.jsonl` (§18).
- Updated `Database/FileIndex/*.json` and, when a version chain is touched, `Database/History/version_history.json`.

## 5. Module Contract

**INPUT:** `List[FileRecord]` from Module 03, `status == "discovered"` (no category filter at the top level — see §3, §9).

**OUTPUT:** Same records, enriched in place; occasionally one *other*, earlier record's `version_group_id` and/or `version_rank` fields updated as a side effect (§4, §7; post-freeze correction, see note below).

**Guarantees** — fields Module 04 owns and populates:
- `duplicate_of` — the `file_id` of the exact-duplicate original, or `null` if none found.
- `version_group_id` — a UUID shared by every member of a detected version chain, or `null` if the record isn't part of one.
- `version_rank` — `"latest"` | `"superseded"`, populated only when `version_group_id` is set; `null` otherwise.
- `duplicate_signals` (§17) — always a full, populated structure for every processed record (never partially filled), storing only the single best-scoring match per detection type (§17, F4); mirrors `classification_signals`'s "never partially filled" guarantee.

**Determinism guarantee:** given the same input batch and the same accumulated `Database/FileIndex/`/`Database/History/` state, Module 04 always produces the same output. Records within a batch are processed in a fixed order specifically so this holds even when multiple version-chain candidates could otherwise tie (§7, F1).

**DOES NOT MODIFY:** every field owned by Modules 01–03 (`file_id`, `source_id`, `original_name`, `original_path`, `current_path`, `extension`, `mime_type`, `size_bytes`, `created_at`, `modified_at`, `content_hash`, `discovered_at`, `status`, `error`, `batch_id`, `category`, `classification_signals`, `extracted_metadata`); every field reserved for Modules 05–08 (`suggested_name`, `suggested_destination`, `confidence_score`, `confidence_breakdown`, `tier`, `processed_at`, `approved_by`, `approved_at`, `reversible`).

**Provider boundary:** not applicable — Module 04 has no provider (§14).

**Hashing algorithms are explicitly not part of this contract.** The contract guarantees only the *semantic behavior* of `content_hash` and any perceptual-hash values Module 04 derives: stable across runs, deterministic for identical input, and comparable (via equality for `content_hash`, via distance for a perceptual hash). The contract makes no guarantee about, and no other module may depend on, which specific algorithm produces that behavior — not SHA-256 for `content_hash` (Module 01's implementation detail, already frozen), and not any particular perceptual-hash variant, hash size, or Hamming-distance calculation Module 04 itself chooses (§11A covers the consequences of ever changing the latter).

**Verified by:** to be built during implementation (§22) — a Module Contract immutability test (every non-owned field byte-identical before/after), plus a dedicated test for the one disclosed exception (only `version_group_id`/`version_rank`, and only on a record genuinely joining or superseded within a version chain, ever change on a record outside the current call's own processing target — every other field on that other record stays byte-identical).

## 6. Internal architecture

Two layers, not three — the Engine/Provider split Modules 02/03 use exists specifically to separate deterministic decision-making from a judgment call that needs live content understanding. Module 04 has no such judgment call (§14), so there is no Provider layer:

```
detect_duplicates_batch()  (src/pipeline/duplicate_detector.py — batch orchestration,
                             persistence, logging — mirrors classify_batch()/
                             extract_metadata_batch()'s shape)
      -> DuplicateDetectionEngine  (per-file decision-making: exact-hash check ->
                                    near-duplicate check (if applicable) ->
                                    version-chain check (if applicable);
                                    fully deterministic)
```

Supporting helpers (in `duplicate_detector.py`, not `core/`, since they're specific to this module's own naming/version logic — the same rationale Module 03 used for keeping its filename-parsing helpers local rather than adding a new `core/` file for a small, module-specific need):
- `_normalize_filename(name)` — strips extension, common version tokens, and separators for similarity comparison.
- `_parse_version_token(name)` — extracts an explicit version indicator (`_v1`, `_v2`, `_final`, etc.) if present.
- `_best_available_date(record)` — the category-appropriate extracted-metadata date field if present, else `modified_at` (tier-4 fallback, same "flagged, not authoritative" treatment Module 03's timestamp hierarchy already established).

Storage-layer functions this module implements (`storage/database.py`, currently `NotImplementedError` stubs): `lookup_hash()`, `lookup_phash_matches()` (signature extended to accept `category` directly, §16, post-freeze correction #4), `lookup_name_matches()` (signature extended to accept `category` directly, §16, F2), `update_indexes()`, `record_version_history()`.

`core/hashing.py`'s `perceptual_hash()`/`hamming_distance()` (currently stubs) are implemented as part of this module, using the already-approved `imagehash` dependency (`requirements.txt` already names it for this purpose).

## 7. Processing workflow

**Deterministic batch order (F1, confirmed):** records within a batch are processed in a fixed, defined order — `discovered_at` ascending, with `file_id` (lexicographic) as the final tie-break when timestamps are identical. This is not a correctness requirement for any single record's own detection (each record's exact-duplicate/near-duplicate checks are independent of order) — it exists specifically so that version-chain grouping and the side-effect "latest" reassignment (step 3 below) are deterministic even when several candidates that could plausibly form a chain arrive in the same batch. Re-running the same batch, or a batch containing the same records in a different input order, always produces the same result (§5's determinism guarantee).

For each record, in that order, with `status == "discovered"` and Module 04 not yet having settled it — defined precisely as: **`duplicate_signals` is still `None`** (never touched by Module 04 at all), **or** the record is sitting in an unresolved cross-version-group conflict (`duplicate_signals.version_conflict == True` **and** `version_group_id is None` — exactly and only the step-3.3 cross-group outcome below, which is deliberately kept eligible for re-examination on every run until resolved, not a bug) (post-freeze correction #2, applied 2026-07-08 — see the note at the end of this section for why the originally-frozen `duplicate_of`/`version_group_id`/`version_rank`-based check failed to actually deliver "only process once" for the single most common outcome). Every other outcome — an exact duplicate found, a near-duplicate found, a version chain formed or joined (with or without a within-group conflict), or genuinely nothing found at all — always leaves `duplicate_signals` populated in a state that does **not** match the cross-group exception above, and is therefore correctly excluded from re-processing on the next run, the same "only process once" idempotency discipline as Modules 02/03.

1. **Exact-duplicate check (always runs, every category):** compute `lookup_hash(record.content_hash)`. If a match exists (an earlier `file_id` already filed under this hash), set `duplicate_of` to that `file_id`, populate `duplicate_signals.exact_duplicate = True`, and **stop** — a byte-identical file cannot simultaneously be "a different version" of anything (different content is definitionally required for versioning), so steps 2–3 are skipped entirely for this record.
2. **Near-duplicate check (Image/Screenshot only, only if step 1 found no exact match):** compute a perceptual hash via `perceptual_hash()`, compare against `lookup_phash_matches(phash, max_distance, record.category)` — **`category` passed directly, post-freeze correction #4** — within the configured max Hamming distance (§11), scoped to same-category candidates only (§9, F5), and keep only the single nearest same-category candidate (§17, F4) — do not retain a list of matches. If a match is found, populate `duplicate_signals.fuzzy_duplicate = True` and `duplicate_signals.phash_distance`, but do **not** set `duplicate_of` — a near-duplicate is a *signal* for Module 06's confidence deduction and hard floor, not a certain duplicate relationship Module 05/07 should act on the same way as an exact match (§9 explains this distinction in depth). Continue to step 3 regardless of this step's outcome — a near-duplicate image can still independently be part of a version chain.
3. **Version-chain check (scoped categories only, §9; Image and Screenshot never group with each other, §9, F5):** normalize the filename and call `lookup_name_matches(normalized_name, category)` (§16, F2 — `category` is passed directly into the lookup, not filtered afterward from full-record fetches). Then, in this exact order (H1 — this sequencing is load-bearing, not stylistic):
   1. **Collect every candidate above the configured similarity threshold (§10)** — do not discard any yet, even though only one will ultimately be kept.
   2. **Inspect the complete candidate set for cross-version-group membership** — whether the collected candidates, taken together, already belong to more than one distinct *non-null* `version_group_id` (M1). A candidate with `version_group_id == null` is unassigned, not a group of its own — `null` never counts as a distinct existing version group. Cross-group conflict (step 3 below) exists only when two or more candidates carry two or more *different, non-null* `version_group_id` values. If every candidate is unassigned (`null`), or at most one non-null `version_group_id` appears among them, there is no conflict — normal version-chain creation/joining proceeds (step 4).
   3. **If the candidates belong to multiple existing version groups (cross-group conflict, F3, confirmed representation):** do not join or auto-merge any of them. Set `duplicate_signals.version_conflict = True`, leave this record's `version_group_id` and `version_rank` both `null`, log the conflict with `conflict_type: "cross_group"` plus the list of conflicting group IDs (§18), and **stop duplicate/version-chain resolution for this record here** — none of the remaining sub-steps below run for it this pass.
   4. **Only if no cross-group conflict exists:** select the single best-scoring candidate from the collected set (§17, F4) — every other candidate is discarded at this point, not before, and not retained as an alternative — and continue using that candidate only:
      - If the matched candidate already has a `version_group_id`, join that group (do not create a new one); if not, mint a new `version_group_id` **shared between this record and the matched candidate** — meaning the matched candidate's own `version_group_id` field is also set to this new ID as part of this same step (post-freeze correction: an earlier pass of this design under-scoped the side-effect exception to `version_rank` only, which made this exact case — the matched candidate being a previously-ungrouped, already-processed record — impossible to actually implement; see the note at the end of this section and §4/§5).
      - Determine `version_rank` for this record by comparing parsed version tokens and `_best_available_date()` against every other member of the group (§10). If the two signals (version token order, date order) disagree, set `duplicate_signals.version_conflict = True` (feeds `Rules/Confidence Rules.md`'s existing `-25` deduction), log the conflict with `conflict_type: "date_token_disagreement"` (§18), and fall back to the defined tie-break rule (§10) rather than guessing silently. (`version_conflict` is deliberately one shared boolean for both this case and the cross-group case above — §17, F3 — the two are distinguished only in the action log's `conflict_type`, never on `FileRecord` itself.)
      - **Side effect (the one exception to in-place-only enrichment, §4):** the matched candidate's `version_group_id` is set as described just above (first-time group creation only — a no-op when joining an already-existing group, since it's unchanged); additionally, if this record is determined to be the new `"latest"`, update the *previous* latest member of the same group — a different, already-processed record — setting only its `version_rank` to `"superseded"`. No field other than `version_group_id`/`version_rank` on any other record is ever touched by Module 04. This is persisted via `record_version_history()` and `save_file_record()` calls scoped to exactly those fields.

   **Post-freeze correction (implementation-discovered defect, applied 2026-07-08):** the version originally frozen limited this section's disclosed side effect to `version_rank` only (§4/§5 said so explicitly), but this sub-step's own plain language — "mint a new `version_group_id` shared between this record and the matched candidate" — requires the matched candidate's `version_group_id` to be set too whenever a group is created for the first time; otherwise a previously-ungrouped candidate would never actually join the chain within the same pass that discovers it. This was found while implementing this exact step, not anticipated by any of the five prior review passes. Resolved by broadening the disclosed exception (§4/§5) to cover both fields — no other mechanic changes: a candidate that already belongs to a group is unaffected (its `version_group_id` doesn't change, since it already agrees), and no field besides `version_group_id`/`version_rank` is ever written to another record.

   **Post-freeze correction #2 (implementation-audit-discovered defect, applied 2026-07-08):** the idempotency check originally frozen at the top of this section (gating on `duplicate_of`/`version_group_id`/`version_rank` all being `None`) does not actually detect "already processed by Module 04" — all three of those fields legitimately stay `None` forever after a fully correct run that found no exact duplicate and no version chain, which is very likely the single most common real-world outcome. As originally specified, such a record would be re-selected, re-processed, and re-logged on every subsequent batch run, indefinitely — confirmed empirically during the Independent Implementation Audit (`Module 04 Implementation Audit.md`, finding H1) by running one genuinely unique record through `detect_duplicates_batch()` twice and observing two near-identical log entries for the same, never-changing record.

   **Root cause:** the check relied on three *outcome*-describing fields (which legitimately stay unset for a negative outcome) instead of a dedicated *has-this-record-been-touched-at-all* signal. `duplicate_signals` already is exactly that signal — §5/§17 already guarantee it is always populated with a full, real instance once Module 04 has processed a record, regardless of what (if anything) was found — but the idempotency check never consulted it.

   **Why the fix is `duplicate_signals is None` plus one narrow, already-approved exception, not simply `duplicate_signals is None` alone:** the fifth architecture-review pass (`Module 04 Design Review.md`) explicitly, deliberately decided that a record stuck in the cross-group-conflict state (step 3.3 above) *should* keep being re-examined on every run rather than becoming idempotent — "staying visible until resolved" was named as correct behavior, not a defect, specifically because v1 has no tooling to ever manually resolve a cross-group conflict any other way. A bare `duplicate_signals is None` check would have silently overridden that already-settled decision (freezing an unresolved conflict in place forever, unseen, on the very next run) — which the requirement to *preserve Module 04's behavior* rules out. The cross-group-conflict outcome is the **only** outcome where `version_group_id` stays `null` *and* `duplicate_signals.version_conflict` is `true` simultaneously (verified by exhaustive case analysis over every other outcome: exact-duplicate, near-duplicate, version-chain-joined-or-created either with or without a within-group token/date conflict, and "nothing found" — none of the others produce this exact combination), so it can be named precisely using fields that already exist, with no new field added to `FileRecord` or `duplicate_signals`.

   **Why this restores true idempotency for both files with matches and files with no matches:**
   - **Files with matches** (exact duplicate, near-duplicate, or a version chain joined/created — with or without a within-group conflict): every one of these paths populates `duplicate_signals` and leaves the record in a state other than the cross-group exception (either `version_group_id` ends up non-`null`, or `version_conflict` stays `false`, or both) — so the record is correctly excluded from re-processing on the next run, exactly as it already was under the old (accidentally-correct-for-this-case) check.
   - **Files with no matches:** `duplicate_signals` is still populated (a full instance with every field at its "nothing unusual" default) and `version_conflict` is `false` — so the exception clause does not apply, and the record is now correctly excluded from re-processing on the next run, fixing the defect.
   - **The one deliberately-preserved exception** (cross-group conflict): `duplicate_signals.version_conflict == true` and `version_group_id is null` together — the record remains eligible for re-examination on every run, unchanged from the already-approved fifth-review decision.

   **Determinism, Module Contract, and scope:** unaffected. This changes only which records `detect_duplicates_batch()`'s (and `main.py`'s equivalent CLI filter's) *internal* eligibility check selects for processing — it is not part of the Module Contract (§5), does not change any guarantee Module 04 makes to another module, and does not alter what value ends up in any field for any record that does get processed. No new field, no new business rule, no change to any detection algorithm.

   This correction is documented here per the Frozen Module Change Policy and is considered re-frozen as of this entry — implementation of the corresponding code change (the two call sites that currently gate on the old three-field check: `detect_duplicates_batch()` and `main.py`'s `detect_duplicates()`) follows from this record, not the other way around.
4. **Update indexes:** call `update_indexes(record)` unconditionally at the end of processing, regardless of whether any match was found, so future records can be compared against this one.
5. **Persist and log:** save the record (and, if applicable, the one other superseded record), append one `detect_duplicates_and_versions` action-log entry (§18).

A single record's failure at any step (§21) never aborts the batch — the same resilience pattern every earlier module already establishes.

## 8. Duplicate detection strategy

**Exact duplication** is determined solely by `content_hash` equality (SHA-256, already computed by Module 01 — no rehashing). This is treated as certain, not probabilistic: a SHA-256 collision between two genuinely different files is cryptographically infeasible, so an exact-hash match is never itself flagged for review the way a near-duplicate is — it is a fact, not a suggestion. (What Module 05/06/07 *choose to do* with that fact — archive it, at what confidence tier — is their decision, not restated here.)

**Near duplication** applies only to Image/Screenshot categories, using a perceptual hash (`imagehash`, via `core/hashing.py`) and a maximum Hamming-distance threshold (§11). Unlike exact duplication, this is explicitly probabilistic — two genuinely different photographs can have a small perceptual-hash distance by coincidence, and two deliberately-similar-but-distinct images (e.g. two frames from a burst-mode shoot) are a real, disclosed edge case (§26). This is why a near-duplicate is recorded only as a *signal* (`duplicate_signals.fuzzy_duplicate`/`phash_distance`), never as `duplicate_of` — `Rules/Confidence Rules.md`'s existing hard floor ("near-duplicate/fuzzy match → never `auto`, at most `approval_required`") already guarantees a human sees it before anything happens, which is the correct treatment for a probabilistic signal, not a certain fact.

## 9. Version detection strategy — and why exact-dup scope differs from near-dup/version scope

Three different scopes, stated explicitly rather than left to be inferred:

- **Exact-duplicate detection runs on every `status == "discovered"` record, regardless of `category`** (including `None`... no — `content_hash` is only `null` when `status == "unreadable"`, and Module 04 never sees unreadable records at all, since they're filtered upstream at `status == "discovered"`; so in practice this means every category, including `Category.UNKNOWN`). Rationale: byte-identical content is byte-identical content regardless of what Module 02 concluded about it — an `Unknown`-category file downloaded twice is exactly as much a duplicate as a classified one, and there's no reason to withhold that fact from Module 06/07 just because classification couldn't resolve a category.
- **Near-duplicate (perceptual hash) detection is scoped to Image and Screenshot only.** A perceptual hash is a specific technique for comparing *visual* similarity; applying it to a PDF's rendered first page, a video's first frame, or an audio waveform would each be a materially different, unbuilt technique — out of scope for v1, consistent with "avoid over-engineering."
- **Version-chain detection is scoped to Invoice, Resume, Bank Statement, Contract, Document, Image, and Screenshot** — explicitly **excluding** Archive, Application, Video, Audio, and Unknown. Reasoning per excluded category: Application installers already have their own `version` field (Module 03's filename-parsed app version) — that's a different concept (the *software's* version) from *this download being a newer copy of a previous download*, and conflating the two would be a real design error, not a simplification. Archive/Video/Audio downloads don't commonly arrive as evolving "v1/v2/final" documents the way text-bearing files and photos do, and the original brief's own example (`Resume_v8` vs. `Resume_v9`) is squarely a text-document pattern. `Unknown` is excluded because a category Module 02 couldn't determine at all offers no reliable signal that two similarly-named files are actually the same *kind* of thing, let alone versions of each other. **Confirmed as designed** — this category list is the settled v1 scope.
- **Image and Screenshot are strictly different categories for both near-duplicate and version-chain grouping (F5, confirmed).** A Screenshot never groups with a plain Image, and vice versa — no category-equivalence mapping exists in v1. `category` equality is exact, matching how it is already treated as a strict, non-overlapping partition everywhere else in this pipeline (Modules 02/03). A future version could introduce an equivalence mapping if real-world data shows this is too strict, but that would be a new, separately-reviewed decision, not a v1 concern.

## 10. Exact matching vs. similarity matching

| | Exact matching | Similarity matching |
|---|---|---|
| **Used for** | Duplicate detection (content) | Version-chain grouping (filename), near-duplicate detection (image) |
| **Mechanism** | SHA-256 equality (`content_hash`) | `rapidfuzz` filename-similarity score; perceptual-hash Hamming distance |
| **Certainty** | Certain (cryptographic) | Probabilistic (a threshold, not a proof) |
| **Recorded as** | `duplicate_of` (a fact) | `duplicate_signals`/`version_group_id` (a signal requiring review) |
| **Confidence treatment** | No deduction defined (Folder Rules routes it directly) | Explicit deductions + hard floors already exist in `Rules/Confidence Rules.md` |

**Version-similarity scoring (rule structure confirmed; numeric threshold is a configurable implementation parameter, not an architectural constant — §15):** normalize both filenames (strip extension, strip a recognized version token, lowercase, collapse separators), score via `rapidfuzz.fuzz.ratio()`, and treat a score ≥ **90** (out of 100, confirmed default for v1, tunable later without a design change) as a candidate match, only within the same `category` (§9, F5). Determine rank by: (a) if both filenames have an explicit, parseable version token (`_v1`/`_v2`/...; `_final` treated as higher than any preceding numbered version, but lower than any later-numbered version discovered afterward — i.e. `_final` means "the last one *I* made," not "immutably the last one ever"), the higher token wins; (b) else, compare `_best_available_date()` across the group, later wins; (c) if (a) and (b) disagree, set `version_conflict = True`, log `conflict_type: "date_token_disagreement"` (§7, §18), and prefer the *filename token's* answer as the tie-break default (a human-authored version number is treated as more deliberate than a possibly-unreliable extracted or filesystem date) — but the conflict is always visible via the signal, never silently absorbed. The rule structure itself (a threshold exists, `_final`'s ordering, filename-wins-on-conflict) is confirmed and settled for freeze; the exact numeric default (90) may be revisited later purely as parameter tuning, not a reopened architectural decision.

**Post-freeze correction #3 (implementation-audit-discovered gap, applied 2026-07-08) — candidate-selection tie-break when two or more above-threshold candidates score identically:** §7 step 3.4 says to "select the single best-scoring candidate" (F4) but, as originally frozen, never stated how to choose among two or more candidates that score *identically* on `fuzz.ratio()` — a real, reachable case (e.g. two files that both normalize to the exact same string, one already belonging to a version group and one not). Found during the Independent Implementation Audit (`Module 04 Implementation Audit.md`, finding M1): an implementation-time tie-break had been added silently (preferring a candidate that already belongs to a version group over one that doesn't) without being reported through this design, contrary to this project's standing discipline that an under-specified, outcome-affecting rule discovered mid-implementation is reported for explicit confirmation, not resolved unilaterally.

**Confirmed rule:** when two or more candidates tie on `fuzz.ratio()` score, prefer whichever candidate already has a non-`null` `version_group_id` over one that doesn't (`version_group_id is not None` as the secondary sort key, descending). Rationale: joining an already-established, real version group is always at least as correct as minting a redundant new one on an arbitrary tie, and without a defined tie-break the outcome would depend on the underlying index's iteration order — an accidental, undocumented form of non-determinism-adjacent fragility this design otherwise explicitly avoids (§7's whole deterministic-batch-order rule exists for exactly this class of concern). If every tied candidate is equally grouped (or equally ungrouped), the tie is broken by whichever the underlying scan encounters first — an occurrence rare enough (multiple *identical*-scoring, *identically*-grouped-or-ungrouped candidates) that no further tie-break is specified for v1.

This is a small, disclosed addition to §10's already-confirmed rule structure, not a new architectural decision — it resolves an implementation-time gap the same way F1's batch-order tie-break resolved an analogous gap during architecture review, just discovered one stage later than it should have been.

## 11. Hash usage strategy

- **SHA-256 (`content_hash`)** — already computed by Module 01 for every readable file. Module 04 never recomputes it; it only reads the existing field and looks it up against `Database/FileIndex/hash_index.json`.
- **Perceptual hash (`imagehash`, via `core/hashing.py`'s `perceptual_hash()`)** — computed by Module 04 itself, once per Image/Screenshot record, and compared via `hamming_distance()` against every **same-category** entry in `Database/FileIndex/phash_index.json` within a maximum distance (**post-freeze correction #4** — see the note at the end of this section; category scoping was always the confirmed intent per §9/F5, but was not carried into this section's mechanism until this correction). **Confirmed default: max distance 5** (out of a typical 64-bit perceptual hash's range) — a smaller distance means stricter/fewer false positives, a larger one catches more true near-duplicates at the cost of more false positives. Like §10's similarity threshold, this is a configurable implementation parameter, not an architectural constant (§15) — the number is settled as a v1 default, tunable later without reopening this design.

**Post-freeze correction #4 (design-internal inconsistency, applied 2026-07-08 — see §16 for the mechanism, §9 for the original "confirmed" decision this corrects the follow-through on):** this section's description of `lookup_phash_matches()` originally said it compares "against every entry" in `phash_index.json`, with no category qualifier — silently inconsistent with §9/F5's "confirmed" requirement that Image and Screenshot are strictly separate categories "for both near-duplicate and version-chain grouping." The inconsistency was traced to its root: the Design Review's own finding F5 ("'Same category' ambiguity for Image vs. Screenshot grouping... by extension, near-duplicate") was resolved at the scoping level by a clarifying sentence added to §9, but that resolution was never propagated into this section or into §16's concrete schema/signature for `lookup_phash_matches()` the way the same review's sibling finding F2 *was* propagated into `lookup_name_matches()`'s signature. Confirmed via independent verification (2026-07-08) that this is a design-completeness gap, not an implementation defect: no version of this section, §7 step 2, or §16 ever instructed an implementer to scope `lookup_phash_matches()` by category, so the shipped implementation (a plain, category-blind scan) faithfully followed what was actually specified. This correction closes that gap by extending this section's own description, matching §16's corresponding extension below.
- All three index lookups are designed as fast, targeted structures (`Database/FileIndex/*.json`) specifically so Module 04 never needs to re-scan the entire cumulative `metadata_store.json` to answer "have we seen this hash/phash/name before" — a deliberate performance improvement over the O(N×M) `save_file_record()` pattern every earlier module already discloses as a known limitation (§20). This is about avoiding a full-store scan for *candidate retrieval* specifically — `lookup_phash_matches()` and `lookup_name_matches()` (both category-scoped, §16) still perform one additional, separate, targeted read of the metadata store to cross-reference each already-narrowed-down candidate's `category`, exactly the same secondary lookup pattern both functions now share (§20).

### 11A. Hashing algorithm changes are a storage/index migration event, not a contract change

Stated explicitly because it is a real, non-obvious operational consequence of §11's design, not because it changes anything about Module 04's behavior today:

- **Existing persisted indexes become incompatible with newly generated values after an algorithm or parameter change.** `Database/FileIndex/hash_index.json` and `Database/FileIndex/phash_index.json` are only internally comparable because every value in them was produced by the same algorithm with the same parameters. If `content_hash`'s algorithm were ever changed (Module 01's territory) or Module 04's perceptual-hash algorithm, hash size, or Hamming-distance calculation were ever changed, newly generated values would not be meaningfully comparable to previously-persisted ones — not merely "less accurate," but comparing values from two different value spaces entirely. A hash-index lookup against a mismatched-algorithm value returns meaningless results, not a degraded-but-safe answer.
- **Such a change requires a deliberate migration or a complete index rebuild** — recomputing every existing record's hash/phash under the new algorithm and repopulating the affected index file(s) from scratch — before the new algorithm can be trusted for comparison against historical data. Partial migration (old and new algorithm values coexisting silently in the same index) is not an acceptable state.
- **Such a change is governed by `Governance/FROZEN_MODULE_CHANGE_POLICY.md`** once Module 04 is itself frozen — it is treated the same as any other post-freeze defect/enhancement to a frozen module's internals: it requires explicit project-owner authorization, a version bump, and a regression re-run, not an informal in-place edit.
- **This migration requirement is an implementation and operations concern, not part of the Module Contract** (§5) — it does not change what Module 04 guarantees to other modules (the semantic behavior of its output fields is unaffected either way), and no other module needs to know or care which algorithm is in use at any given time. It matters only to Module 04's own internal state and whoever operates a future algorithm change.

## 14. AI vs. deterministic responsibilities

**Module 04 is fully deterministic. It has no provider, no live-Claude-judgment step, and no Engine/Provider split.** This is a deliberate departure from Modules 02/03's pattern, not an oversight — stated explicitly because the original pre-build brief's own tools table (`Pre-build context.md` §9) listed Module 04 as "Claude (human can override 'latest')... occasional semantic override needed," which could be read as anticipating a provider-style judgment step the way Modules 02/03 have one.

Why this design instead concludes no provider is needed: every decision Module 04 makes — hash equality, perceptual-hash distance, filename similarity score, version-token parsing, date comparison — is a computation over already-extracted, already-structured data (bytes, a hash, a filename string, an already-extracted date field). None of it requires *reading and understanding a document's content* the way classification or metadata extraction does; the "judgment" the original brief anticipated is really "what do we do when the deterministic signals disagree," which this design resolves not by asking Claude to adjudicate, but by **flagging the disagreement** (`version_conflict`) and deferring the actual decision to the human review path Module 06/07 already provide for exactly this purpose (`Rules/Confidence Rules.md`'s hard floors already force `review_required`/`approval_required` rather than silent auto-resolution). This is the same "deterministic before AI" principle every earlier module follows, applied here to the whole module rather than to a subset of its categories, and is a genuine, considered scope decision worth your explicit sign-off given it changes the anticipated shape of the module (no `MetadataExtractionProvider`-style class exists for Module 04 at all).

**Confirmed:** proceed without a provider for v1, consistent with the reasoning above. If a future version finds the deterministic signals insufficiently discriminating (e.g. too many false-flagged version conflicts), a live-judgment provider could be added later without breaking this module's contract, the same way Module 03 accepted an unused-in-v1 `MetadataExtractionProvider` scaffold for its deterministic-only categories (`Module 03 Design.md` §27) — but that scaffold would be new work, not something this design pre-builds speculatively.

## 16. Database changes

First real implementation of `Database/FileIndex/` and `Database/History/`, both currently empty placeholders per `Database/README.md`:

- **`Database/FileIndex/hash_index.json`** — `{ "<sha256>": "<file_id>" }`. One entry per distinct content hash ever filed; `update_indexes()` adds a new entry only if the hash isn't already present (the *first* file filed under a given hash is definitionally never anyone else's duplicate).
- **`Database/FileIndex/phash_index.json`** — `{ "<perceptual_hash>": ["<file_id>", ...] }`. Unlike the hash index, this is a list per key (or, more precisely, `lookup_phash_matches()` must scan for any key within the max distance, not just an exact key match — a perceptual hash index can't be a simple dict lookup the way an exact hash index can, since "within distance N" isn't a dict key operation; implementation will need a linear or bucketed scan over existing entries, not a single dict access). **`lookup_phash_matches(phash, max_distance, category)` (post-freeze correction #4, confirmed) accepts `category` as a direct parameter**, mirroring `lookup_name_matches()`'s existing treatment immediately below exactly: the lookup resolves candidate `file_id`s against their stored `category` internally (cross-referenced against the metadata store, since — like `name_index.json` — the index itself doesn't store category) and returns only same-category matches within the max distance — never filtered post-hoc by the caller. This is not a change to `phash_index.json`'s own key/value shape, the same latitude §16 already establishes for `name_index.json`'s equivalent extension.
- **`Database/FileIndex/name_index.json`** — `{ "<normalized_name>": ["<file_id>", ...] }`. **`lookup_name_matches(normalized_name, category)` (F2, confirmed) accepts `category` as a direct parameter** rather than requiring the caller to fetch each candidate's full record for post-hoc filtering — the lookup resolves candidate `file_id`s against their stored `category` internally (a small cross-reference inside the function, not a change to `name_index.json`'s own key/value shape) and returns only same-category matches. This is an interface change to an already-unimplemented stub, not a Module Contract change — internal storage-layer signatures are never part of the contract other modules depend on (§5's hashing-algorithm precedent applies equally here). **`name_index.json` is a candidate-retrieval index, not the fuzzy matcher itself (M2)** — an exact-key lookup into it is not sufficient on its own to satisfy §10's similarity-matching design. `lookup_name_matches()` performs candidate retrieval (same-`category` entries from `name_index.json`) followed by fuzzy similarity evaluation (`rapidfuzz.fuzz.ratio()` against each retrieved candidate's normalized name, per §10), returning only those clearing the threshold. The storage format above does not constrain the matching algorithm — a future implementation may optimize candidate retrieval (e.g. bucketing by normalized-name prefix or length) without changing this Module Contract, the same latitude `phash_index.json`'s lookup already has (§20, §27).
- **`Database/History/version_history.json`** — one entry per `version_group_id`, exact shape already sketched in `Build-out/08 Logging & Reporting/Metadata & Log Schema.md`: `{"version_group_id": "...", "files": [{"file_id": "...", "filename": "...", "rank_at_time": "superseded"|"latest", "superseded_at": "..."}]}`. `record_version_history()` appends to (or updates) this structure every time a version-chain relationship is created or a rank changes.

`storage/database.py`'s already-stubbed functions (`lookup_hash`, `lookup_phash_matches`, `lookup_name_matches`, `update_indexes`, `record_version_history`) are implemented to this shape during Module 04's implementation phase. `lookup_name_matches()`'s signature is extended to accept `category` (above, F2); `lookup_phash_matches()`'s signature is likewise extended to accept `category` (above, post-freeze correction #4); the other three functions' existing signatures are unchanged.

## 17. FileRecord changes

`duplicate_of`, `version_group_id`, `version_rank` already exist on `FileRecord` (reserved for this module since Module 01's schema was first drafted) — no change needed to populate them.

**Confirmed addition:** a new `duplicate_signals` field, mirroring `classification_signals`'s existing pattern exactly — a small, typed dataclass (`DuplicateSignals`, to live in a new `src/models/duplicate.py`, matching how `Category`/`ClassificationSignals` got their own `src/models/classification.py`), with fields: `exact_duplicate: bool`, `fuzzy_duplicate: bool`, `phash_distance: Optional[int]`, `version_conflict: bool`. Always a full, populated instance on every record Module 04 processes (never partially filled), `None` until Module 04 runs — the exact same guarantee shape `classification_signals` already has.

**Single-best-match limitation (F4, confirmed):** `duplicate_signals` records only the single best-scoring match per detection type — one nearest perceptual-hash candidate, one best-scoring version-chain candidate — never a list of several simultaneous candidates. This is an intentional v1 limitation, disclosed here so it is never mistaken for an oversight: if a future need arises to surface multiple candidates to a human reviewer, that requires a schema change at that time, not something this design speculatively builds for now.

**`version_conflict`'s dual meaning (F3, confirmed):** this one boolean is deliberately reused to flag either kind of version-chain ambiguity Module 04 can detect — a within-group filename-token/date disagreement (§10), or a cross-group conflict where a record's above-threshold candidates span two different pre-existing `version_group_id`s (§7 step 3). `FileRecord` itself does not distinguish which kind occurred — kept to one field, consistent with `classification_signals`'s own minimal-fields precedent — the distinction is carried only in the action log's `conflict_type` detail (§18), never on `FileRecord`. In the cross-group case specifically, `version_group_id` and `version_rank` are both left `null` on the record — no automatic merge, no guessed assignment — until manual review resolves it.

No other `FileRecord` field changes proposed.

## 19. Security considerations

- **No new code-execution surface.** Perceptual hashing reads image bytes via `Pillow`/`imagehash`, the same library already vetted and in use for Module 02's EXIF/vision-support reads — no new dependency risk.
- **No archive extraction, no shelling out, no `eval`** — Module 04 never opens an archive's contents (that's Module 03's already-bounded `zipfile.namelist()`-only territory) and has no reason to.
- **Bounded blast radius for a false positive.** A wrongly-flagged near-duplicate or version conflict can, at worst, cause a file to be routed to manual review it didn't strictly need (Module 06's hard floors already force this for any fuzzy/conflicting signal) — it can never cause a silent, wrong, irreversible action, because nothing in this pipeline ever deletes a file (`CLAUDE.md`'s own non-negotiable) and Module 04 itself never touches the filesystem at all beyond reading bytes. A wrongly-*missed* duplicate (a false negative) is symmetrically bounded: the file is simply treated as new, which is the same outcome as if Module 04 didn't exist for that file — never worse than the pre-Module-04 baseline.
- **Hash-collision risk is not a real concern for `content_hash`** (SHA-256) at any practical scale; perceptual-hash "collisions" (two different images landing within the threshold distance) are an expected, accepted, and already-mitigated-by-hard-floor part of how a perceptual hash works, not a security defect.

## 20. Performance considerations

- `lookup_hash()`/`lookup_name_matches()` are designed as targeted index lookups specifically to avoid the O(N×M) full-store scan pattern every earlier module's `save_file_record()` usage already discloses as a known limitation — a deliberate improvement, not an accidental one, worth calling out explicitly since it's the first module to actually build the `FileIndex/` structures that were sketched from the start for this exact reason.
- `lookup_phash_matches()` cannot be a simple dict lookup (§16) — it requires scanning existing perceptual-hash entries for anything within the max distance, plus (post-freeze correction #4) a `load_metadata_store()` scan to cross-reference each within-distance candidate's stored `category` — the same cost profile `lookup_name_matches()` already has (both now share "scan same-scope entries, then cross-reference category against a full metadata-store load"). At v1 volumes (a single user's Downloads folder) this is expected to be fine; flagged here as the first thing to revisit (e.g. a bucketed/BK-tree structure, or storing category alongside each index entry to avoid the metadata-store cross-reference) if perceptual-hash volume ever grows enough to matter — informational, not a blocking concern for this design.
- `save_file_record()`'s existing O(N×M) full-store-rewrite cost, already disclosed by Modules 02 and 03 as their own inherited problem, is now also Module 04's problem for the same reason — restated here per `ENGINEERING_STANDARD.md` §21's requirement that each additional module re-disclose a shared inherited cost as it becomes newly relevant, not rely on an earlier module having mentioned it once.
- No new measured performance number is claimed by this design document itself (design phase, no code yet) — a real large-batch measurement is required at the implementation/release stage, per `ENGINEERING_STANDARD.md` §21.

## 21. Failure handling

Same two-layer discipline as every earlier module (`ENGINEERING_STANDARD.md` §13):
- **`DuplicateDetectionEngine`-level (anticipated failures):** `perceptual_hash()` raising on a corrupted/unreadable image (possible even for an Image/Screenshot record that Module 02/03 already opened successfully once, since a file can change or a decode edge case can differ between libraries) → that record's near-duplicate check is skipped, `duplicate_signals.fuzzy_duplicate` stays `False`, a sanitized `error_detail` is captured, and the record's *other* checks (exact-duplicate, version-chain) still proceed normally — a single check's failure never disables the whole record's processing. Similarly, a `lookup_*()` call raising (e.g. a corrupted index file) is caught, treated as "no match found" for that lookup only, with the same diagnostic capture.
- **Batch-orchestration-level (unanticipated failures):** a single record's genuinely unexpected exception is caught by `detect_duplicates_batch()`'s own outer safety net, logged as an `error` action, and the batch continues — the same resilience pattern as `classify_batch()`/`extract_metadata_batch()`.

No failure mode in Module 04 ever produces a fabricated `duplicate_of`/`version_group_id` — an honest `null` (nothing found / couldn't determine) is always preferred to a guess, the same "Unknown instead of guessing" principle as every earlier module (`Governance/ARCHITECTURE_DECISIONS.md` #7).

## 22. Test strategy

Named here as a commitment this design makes (`ENGINEERING_STANDARD.md` §2's requirement that a design name concretely which regression tests it commits to) — not built during this design phase:

- Exact-hash match / no-match, including two records with the same hash but different categories (an edge case §26 names explicitly).
- Perceptual-hash boundary cases: distance exactly at the threshold, one above, one below (mirroring Module 03's exact boundary-testing discipline for its own numeric rules).
- Filename normalization and similarity-scoring boundary cases (score exactly at threshold, one above, one below); version-token parsing (`_v1`/`_v2`/`_final`/no token/malformed token).
- Version-conflict detection: filename-order and date-order agreeing, disagreeing, and one signal missing entirely.
- The side-effect update (§7 step 3, §4): verify *only* `version_group_id`/`version_rank` ever change on another record (first-time group creation sets `version_group_id` on the matched candidate; supersession sets `version_rank` on the previous latest) — every other field on that other record byte-identical, via the same immutability-test discipline Module 03 uses for its own contract.
- A Module Contract immutability test for the *currently-processed* record, matching every earlier module's own such test.
- `Database/FileIndex/*.json`/`Database/History/version_history.json` (de)serialization round-trips.
- Action-log shape for `detect_duplicates_and_versions`, including the "second line, not a rewrite" behavior for a side-effect update (§18), and both `conflict_type` values (F3).
- Deterministic batch-processing order (F1): the same input batch, re-run or supplied in a different input-list order, produces byte-identical output; an explicit test for two version-chain candidates with identical `discovered_at` resolving via the `file_id` tie-break.
- Category-scoped `lookup_name_matches()` (F2): a same-normalized-name, different-category candidate is never returned as a match.
- Category-scoped `lookup_phash_matches()` (F5, post-freeze correction #4): a same-(or within-distance)-perceptual-hash, different-category candidate (e.g. an Image and a Screenshot that happen to share or nearly share a perceptual hash) is never returned as a match — the near-duplicate counterpart to the `lookup_name_matches()` test immediately above.
- Cross-group conflict handling (F3): a record whose above-threshold candidates span two existing `version_group_id`s ends up with `version_group_id`/`version_rank` both `null` and `duplicate_signals.version_conflict = True`, `conflict_type == "cross_group"` — never an automatic merge.
- Single-best-match retention (F4): when multiple perceptual-hash or filename-similarity candidates clear the threshold, only the single best-scoring one is retained in `duplicate_signals`.
- Image vs. Screenshot exact-category enforcement (F5): a Screenshot never groups with a same-named/same-content Image, and vice versa — **for both version-chain grouping (already tested) and near-duplicate grouping (post-freeze correction #4 — this half was not actually tested until this correction; see the dedicated bullet above)**.
- A drift-guard test once §10/§11's thresholds are formalized into `Rules/Duplicate & Version Rules.md` (§15) — mirroring Module 03's `Rules/Confidence Rules.md` citation/table cross-check test, so this class of drift can't recur a third time.

All are now implemented (`pipeline/test_duplicate_detector.py`, `storage/test_database.py`) and verified passing as of the second implementation audit. Finding H1 additionally required a regression test this list didn't originally name (the idempotency correction, §7 post-freeze correction #2, and its precise negative case — a resolved cross-group conflict must **not** become idempotent) — added alongside the others.

**Second status note (added post-UAT-Run-1, 2026-07-08, post-freeze correction #4):** UAT's first real run (`Tests/Module 04 UAT Plan.md`, Finding UAT-1) discovered — and independent verification confirmed as a design-completeness gap, not an implementation defect — that the near-duplicate half of the F5 bullet above was never actually implemented or tested; only the version-chain half was. The design is corrected here (§7 step 2, §11, §16, and the dedicated test bullet added above) to carry §9/F5's already-confirmed decision through to `lookup_phash_matches()`'s mechanism, exactly as §16's F2 correction already did for `lookup_name_matches()`. ## 26. Known edge cases

- **Two exact-hash-duplicate records with different `category`.** Possible in principle (if Module 02's judgment path produced a different answer across two separate runs on identical bytes, or if one was processed before a Module 02 fix and one after). Hash equality is authoritative for `duplicate_of` regardless of category agreement — Module 04 does not require or check category agreement for exact matches (only for near-duplicate/version-chain matches, where category scoping is load-bearing). The mismatch itself is not actively flagged in v1 — a disclosed, accepted limitation, not a blocking gap.
- **A version chain's "latest" determination changing as new versions arrive over time.** Explicitly designed for (§7 step 3's side effect) — not an edge case the design fails to handle, but one worth naming so a future reader doesn't mistake the side-effect update for a bug.
- **Perceptual near-duplicate false positives** (two genuinely different but visually similar images, e.g. burst-mode photos) — mitigated by the threshold (§11) and by the hard floor that guarantees human review, never silent auto-action.
- **Two intentionally-identical files that aren't really "the same download twice"** (e.g. a blank template reused for two unrelated purposes) — content-only duplicate detection cannot distinguish this from an accidental re-download; accepted as a known limitation of a purely content-based approach, no semantic understanding attempted in v1 (consistent with §14's no-provider scope decision).
- **A category with no reliable date field at all** (Video/Audio are excluded from version-chain scope entirely per §9, so this specifically concerns text/image categories with a sparse or missing date field) — falls back to `modified_at`, flagged via the same tier-4-fallback discipline Module 03 already established, not silently treated as authoritative.
- **Zero-byte or unreadable files** never reach Module 04 at all (filtered at Module 01/the `status == "discovered"` gate) — not a case this module needs to handle.
- **A new file's candidates spanning two pre-existing version groups.** Handled explicitly (§7 step 3, F3): flagged via `duplicate_signals.version_conflict`/action-log `conflict_type: "cross_group"`, left unassigned (`version_group_id`/`version_rank` both `null`) pending manual review, never auto-merged.

## 27. Future extensibility

- A future version could add a live-judgment provider if deterministic signals prove insufficiently discriminating in practice (§14) — deferred, not pre-built speculatively.
- `Rules/Duplicate & Version Rules.md` (§15, recommended, not created) would let the thresholds in §10/§11 be tuned without touching this design document, the same relationship `Rules/Confidence Rules.md` has to `Confidence & Review`'s architecture.
- A future SQLite migration (`ROADMAP.md` Version 2, already anticipated for `Database/` generally) would let `lookup_phash_matches()` use a real indexed nearest-neighbor structure instead of a linear scan (§20) — this module's `storage/database.py` function signatures are written to not presume JSON specifically, so this migration shouldn't require a contract change, only an implementation change.
- Multi-document PDF splitting (`ROADMAP.md` Version 2) would, once built, likely need Module 04 to reconsider whether a "duplicate" check on a batch-invoice-export file is even meaningful pre-split — flagged as a forward-compatibility consideration for whoever builds that feature, not resolved here.

## 28. Explicit ownership boundaries

| | Module 04 owns | Module 04 must never do | Belongs to |
|---|---|---|---|
| Duplicate detection | `duplicate_of`, exact/fuzzy signals | Decide what happens to a duplicate | Module 05 (routing), 06 (scoring), 07 (execution) |
| Version detection | `version_group_id`, `version_rank`, `version_conflict` signal | Decide which version is "correct" to keep long-term (only flags latest/superseded) | Module 07 (execution — archiving superseded, never deleting) |
| Filesystem | Reads file bytes (for hashing) | Move, rename, archive, or delete any file | Module 07 |
| Destination | Nothing | Choose or suggest a folder | Module 05 |
| Confidence | Nothing (supplies raw signals only) | Compute a score or tier | Module 06 |
| Database | `FileIndex/`, `History/` | Modify `Metadata/`'s Module 01–03-owned fields | N/A — read-only there |
| Reporting | Nothing (supplies log entries only) | Generate the Duplicate Report itself | Module 08 |
