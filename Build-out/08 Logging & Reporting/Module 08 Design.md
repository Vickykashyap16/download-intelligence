## 0. Acceptance criteria (defined before the design below; the rest of this document must satisfy these)

### 0.1 What Module 08 guarantees

- **G1 — Writes only to `Runtime/Reports/*`.** Module 08 never writes any `FileRecord` field, never writes to `Database/Metadata/`, `Database/FileIndex/`, `Database/History/`, or `Database/Learning/`, and never writes to `Runtime/Logs/action_log.jsonl`. This is a stronger and simpler claim than any earlier module's ownership boundary, because Module 08 is the first module in this pipeline that owns **zero** `FileRecord` fields — its entire output surface is report files, not record enrichment.
- **G2 — Read-only with respect to pipeline state.** Running Module 08, once or repeatedly, never changes any `FileRecord`, never triggers or influences a re-execution of any earlier module, and never re-derives or second-guesses any upstream module's conclusion (`category`, `tier`, `duplicate_of`, etc.) — it only aggregates and presents what already happened. This is the same "does not do a later/earlier module's job" discipline `Governance/ARCHITECTURE_DECISIONS.md` decision 3 establishes, applied here in the read direction rather than the write direction.
- **G3 — Every reported figure is traceable.** Every count, table row, and total in a generated report is derived directly from a real `Runtime/Logs/action_log.jsonl` entry or a real `Database/Metadata/metadata_store.json` field — never an estimate, an interpolation, or a fabricated placeholder. This extends decisions 7/19's "never guess, always disclose" philosophy from judgment calls to aggregation: an honest "no data for this period" is always preferred to a plausible-looking invented number.
- **G4 — A report-generation failure never affects a file operation that already happened.** This is a direct, binding carry-forward of the pre-design note's own stated edge case (`08 Logging & Reporting.md`, "Edge cases": *"Log/report write fails → never let a logging or reporting failure block the actual file move; surface the failure in the next Daily Summary instead"*) — confirmed here as still correct and made a formal guarantee rather than left as an informal note.
- **G5 — Idempotent regeneration over unchanged source data, unconditionally.** Re-running report generation for the same period, against an unchanged action log and metadata store, produces byte-for-byte identical output — full stop, no carve-out. **Revised in this corrective pass (resolves Design Review Round 1 finding M1):** the original wording reserved an exception for a possible "generated at [wall-clock timestamp]" line; no report template introduces one. Where a report needs to convey how current its content is (relevant chiefly to a still-open Daily Summary or, depending on OD-1, a continuously-updated Duplicate/Storage Report), it does so with a **data-derived "as of" marker** — e.g. "reflects activity through [the latest action-log entry's own `timestamp` field among the entries included]" — never a live `now()` call at render time. A data-derived marker is itself part of the source data (the same unchanged action log G5 is already defined against), so it does not reintroduce the ambiguity the original wording was trying to carve around; a wall-clock marker would. See §7 for the full reasoning and the alternatives considered.
- **G6 — Never permanently deletes or silently overwrites a previously-generated report for a closed period.** A closed period's report (e.g. yesterday's Daily Summary) is treated as a historical record once written, consistent with `CLAUDE.md`'s non-negotiable ("never permanently delete anything") applied here to generated artifacts, not just user files — see §11/OD-2 for exactly what "closed" means and what happens if source data changes after a period closes.
- **G7 — No new external dependency beyond read access to `Database/`, `Runtime/Logs/`, and (for the Storage Report only, see OD-4) the destination library's real folder structure.** Module 08 introduces no provider, no network call, no new third-party dependency.

### 0.2 What Module 08 explicitly does not guarantee

- **NG1 — Does not guarantee real-time or continuous reporting.** v1 report generation is invoked the same way every other module's batch function is — Manual or Scheduled (`README.md`'s Execution Modes table) — never a live dashboard or push notification.
- **NG2 — Does not guarantee retroactive correction of an already-closed report if the underlying data changes afterward** (e.g. an undo executed the day after that day's Daily Summary was generated). See OD-2 — this is a real, disclosed open question, not silently resolved by this design.
- **NG3 — Does not guarantee any rendering beyond plain Markdown files.** No charts, no HTML dashboard, no interactive UI in v1 — matches every report example already committed to `Metadata & Log Schema.md`.
- **NG4 — Does not guarantee trend prediction or forecasting.** The Weekly Summary and Storage Report show historical rollups only; no projection of future growth or behavior.
- **NG5 — Does not guarantee protection against `action_log.jsonl` growing unboundedly.** Module 08 reads the log as it exists; log rotation/archival remains an explicitly deferred `Version 2` concern (`Governance/ARCHITECTURE_DECISIONS.md` decision 10), not something this module's design solves.
- **NG6 — Does not guarantee learning or personalization from report content.** Purely descriptive output — `Database/Learning/User Corrections.json`'s own passive-capture-only status (NG6 in Module 07's own contract) is unaffected by anything Module 08 does.

### 0.3 Measurable release criteria

- Unit suite: 100% pass, including a zero-write immutability test — the standing per-module requirement (`ENGINEERING_STANDARD.md` §11), here simplified to "assert no `FileRecord` field and no `Database/*` file changes at all" rather than "assert non-owned fields are unchanged," since Module 08 owns none.
- Integration Test Plan executed against the real Module 01→08 chain: report content verified against a real, hand-computable action log/metadata store snapshot; a forced report-generation failure confirmed not to affect or roll back any already-completed Module 07 execution (G4); multi-day and multi-batch rollup correctness; full regression.
- UAT run as a real user against a real external Downloads-like folder and a real external destination-library-like folder, through the real CLI, with the generated Markdown reports actually read and checked for human clarity and correctness — not just machine-checked totals.
- A measured (not estimated) performance number for report generation against `Tests/Large Batch/` or equivalent, and an explicit statement of how that cost scales with a growing action log/metadata store (§20).
- Release Audit: all 13 Pipeline Contract Verification checks pass, or every exception is explicit and approved; zero unresolved Critical/High/Medium findings.

### 0.6 Correction to the superseded pre-design reference — read this before anything else below

`Build-out/08 Logging & Reporting/08 Logging & Reporting.md` (the pre-build spec note, written before any module existed) states Module 08's "Logic" as including: *"Database (Metadata/FileIndex/History) — written/updated the moment a file is processed... Logs (`action_log.jsonl`) — append-only, one line (JSON) per action... tagged with a `batch_id`."* Read literally, this assigns Module 08 responsibility for writing the metadata store, the FileIndex, the version history, and the action log itself.

**That is no longer accurate, and this design does not carry it forward.** Modules 01 through 07 already write all of that themselves, as they were each built: every module calls `save_file_record()` (`src/storage/database.py`) to persist its own owned fields the moment it processes a record (`Governance/ARCHITECTURE_DECISIONS.md` decision 11), `Database/FileIndex/*` and `Database/History/version_history.json` are written exclusively by Module 04 (`Release/Module04/MODULE_CONTRACT.md`), and every module calls the shared `append_action_log()` primitive (`src/storage/runtime_io.py`, implemented at Module 01's own implementation time specifically *because* every module needs it) to write its own action-log entries as it goes. `src/pipeline/reporting.py`'s existing scaffold stub (`finalize_batch()`'s docstring: *"update Database/Metadata + FileIndex + History for every processed file, append the action log entries"*) still reflects this same stale, pre-implementation assumption and needs a corrected docstring at implementation time (§25, a required follow-up, not performed by this design document itself, mirroring how Module 07's design flagged `Rules/Folder Rules.md`'s own wording as needing an implementation-time update rather than editing it here).

**Module 08's real, current-architecture scope is narrower and purely additive: read-only aggregation and report generation over data every earlier module already wrote.** This is not a scope cut imposed by this design — it is a correction that recognizes what the codebase has actually already become since that pre-design note was written, the same class of correction Module 03's own implementation applied to its own superseded pre-design pointer note (`Release/Module03/IMPLEMENTATION_AUDIT.md` finding F3). The rest of this document is written against this corrected scope throughout.

## 1. Purpose

The pipeline's read-only aggregation and human-communication layer: turn everything Modules 01–07 have already recorded — the cumulative metadata store and the append-only action log — into plain-language summaries a human can actually read without inspecting raw JSON. Module 08 is the last module in the strict linear chain (`Release/DEPENDENCY_DIAGRAM.md`) and the first module in this pipeline that changes nothing about any `FileRecord` or any prior module's persisted state — every module 01–07 writes at least one field or Database structure; Module 08 writes none.

## 2. Architectural note: no Engine/Provider pattern, and no Provider-shaped decision layer at all

`Governance/ARCHITECTURE_DECISIONS.md` decisions 4–6 define the Engine → Provider pattern for judgment-dependent logic (Modules 02/03); decisions 5/6's "fully deterministic, no Provider" precedent already covers Modules 04–06; decision 22 covers Module 07's plain-data-structure alternative for a human-decision input. Module 08 fits none of the "needs some external input beyond already-structured data" shapes those decisions address — it reads only its own two already-fully-structured sources (the metadata store, the action log) and performs pure computation (counting, grouping, summing) over them. There is no judgment call, no human decision, and no non-deterministic input anywhere in this module's scope. **Decision: Module 08 has no Engine/Provider layer and no equivalent decision-channel abstraction — a single-layer `report_batch()`/`generate_<report>()` shape, batch-orchestration function directly, mirroring Modules 04/05/06's "two layers, not three" precedent, simplified further to one layer, since there is no per-file Engine decision to separate from batch orchestration in the first place** (see §9). *(Wording tightened in the Round 3 cleanup pass — resolves Design Review Round 2 finding C1: the original "arguably one layer" hedge is replaced with a direct statement, matching the decisive tone every other module's design uses to state its own architecture.)*

## 3. Responsibilities

1. **Daily Summary generation** — after a day's run(s), aggregate that day's action-log entries and the resulting `FileRecord` states into `Runtime/Reports/Daily Summary/summary_YYYY-MM-DD.md`: files scanned, auto-filed, approval-required, review-required, duplicates found, versions archived, errors, and a per-file table (original name, new name, destination, category, confidence, tier) — matching `Metadata & Log Schema.md`'s own already-committed worked example exactly.
2. **Weekly Summary generation** — roll up a week's Daily Summaries into `Runtime/Reports/Weekly Summary/`, trends over time (volume, tier distribution, error rate).
3. **Duplicate Report generation** — a report over every `duplicate_of`/`version_group_id`/`version_rank`-bearing record in the metadata store: what was found, what happened to it (archived, review-required, etc.).
4. **Storage Report generation** — space used per destination folder/category, growth over time (see OD-4 for the open question of exactly what data this reads).
5. **Read-only aggregation helpers** — functions that load and summarize the action log/metadata store for the above four report types; not a general-purpose query interface (`ROADMAP.md`'s "future ideas" search/query interface is explicitly out of v1 scope, unrelated to this module).

## 4. Responsibilities explicitly reserved for later work (per `ARCHITECTURE_DECISIONS.md` decision 3)

- **Any FileRecord field enrichment, of any kind.** Module 08 is a consumer of the fully-populated record, never a producer of any part of it — if a report needs a value no earlier module currently populates, that is a gap in an earlier module's scope or a genuinely new field requiring its own architectural decision, never something Module 08 computes and stores itself.
- **Active learning / auto-application of `Database/Learning/User Corrections.json`.** `ROADMAP.md` Version 3 scope, unrelated to reporting.
- **Log rotation, archival, or pruning of `action_log.jsonl`.** Explicitly deferred (`ARCHITECTURE_DECISIONS.md` decision 10) — Module 08 reads the log as-is, however large it has grown.
- **A query/search interface over `Database/Metadata/`.** `ROADMAP.md`'s own "future ideas" list, not committed v1 scope and not part of this module's four named report types.
- **Any UI, dashboard, or notification mechanism beyond writing a Markdown file to `Runtime/Reports/`.** Deferred, unnamed, no roadmap commitment currently exists for it.

## 5. Inputs

**Revised in this cleanup pass (resolves Design Review Round 2 finding N1) — split into precise per-report-type sources, matching §9/§19's own already-correct detail rather than the single, imprecise grouping originally stated here.**

**Receives (for Daily Summary and Duplicate Report):** the full contents of `Runtime/Logs/action_log.jsonl` (via `read_action_log_entries()`, already implemented at Module 07's WP-8 — no new read primitive required) and the full contents of `Database/Metadata/metadata_store.json` (via `load_metadata_store()`, already implemented since Module 01) — not a `List[FileRecord]` handed in by a caller the way every earlier module receives one, since Module 08 is not processing a specific batch's records forward through the pipeline; it is summarizing everything recorded so far, scoped by time window (a day) or by signal type (duplicate/version presence).

**Receives (for Weekly Summary):** already-written `Runtime/Reports/Daily Summary/` files for the reported week only as its primary source — not the raw action log or metadata store directly (§9's own already-established design; this section previously, inaccurately, implied otherwise by grouping Weekly Summary with Daily Summary/Duplicate Report above). **One narrow, disclosed exception, added during the Round 3 cleanup pass to keep this section consistent with §12's L1 fix:** `generate_weekly_summary()` additionally consults `Runtime/Logs/action_log.jsonl` for the single, bounded purpose of distinguishing "genuinely no activity that day" from "that day's Daily Summary generation previously failed" when a requested day's Daily Summary file is missing (§12) — a read-scope exception, not a write-ownership one (§0.5/§8's "no disclosed exception" claim is about what Module 08 writes, unaffected by this).

**Receives (for Storage Report only, pending OD-4):** either `size_bytes` values already present in `Database/Metadata/metadata_store.json` (no new dependency), or — if OD-4 resolves toward measuring real, current on-disk usage — read access to the destination library's real folder structure, a new kind of dependency no earlier deterministic module has needed for read purposes beyond Module 01's own initial Downloads-folder scan.

**Also receives (internally):** nothing else. No provider, no human decision input, no configuration beyond knowing where `Database/`/`Runtime/` live (already-established path constants, `src/storage/database.py`/`runtime_io.py`) and, for Storage Report under OD-4, the same `destination_root` configuration Module 07 already reads (`ARCHITECTURE_DECISIONS.md` decision 20) — reused, not reintroduced.

## 6. Outputs

**Produces:**
- `Runtime/Reports/Daily Summary/summary_YYYY-MM-DD.md` — one file per calendar day with reportable activity.
- `Runtime/Reports/Weekly Summary/summary_YYYY-Www.md` — one file per reported week (ISO week numbering; filename pattern confirmed against `Runtime/Reports/README.md` during this cleanup pass, resolves Design Review Round 2 finding L3 — see §17/§25).
- `Runtime/Reports/Duplicate Report/*.md` — what was found and its disposition (archived, kept, overridden by the user — the three outcome categories `Runtime/Reports/README.md` already names) — see OD-1 for open questions about single-current-state vs. dated-snapshot shape.
- `Runtime/Reports/Storage Report/*.md` — see OD-1/OD-4.
- **No `FileRecord` field changes of any kind.** No `Database/*` writes of any kind. No `Runtime/Logs/action_log.jsonl` writes of any kind.

## 7. Determinism guarantee

**Revised in this corrective pass (resolves Design Review Round 1 finding M1).** Given an unchanged action log and metadata store (the exact byte content Module 08 reads), report generation always produces byte-for-byte identical output — same counts, same table rows, same totals, same everything — every time it runs. There is no carve-out, because no report template includes any wall-clock-derived content.

**Alternatives considered for how (or whether) a report conveys its own recency:**

- **(A) No recency marker at all.** The smallest possible fix to the original ambiguity — simply omit any "generated at" line, matching `Metadata & Log Schema.md`'s own already-committed Daily Summary worked example, which has no such line. Sufficient for Daily/Weekly Summary, whose filenames/headers already carry an inherent date scope (`summary_YYYY-MM-DD.md`, `# Daily Summary — 2026-07-05`). **Rejected as the sole answer** because it silently loses real information for whichever of Duplicate Report/Storage Report OD-1 resolves toward a single, continuously-updated "current state" file — such a file has no filename-embedded date the way Daily Summary does, so a reader would have no way to judge how current its content is at all.
- **(B) A wall-clock "generated at [now()]" line, with G5 scoped to exclude it.** The design's original resolution. **Rejected on reconsideration:** it reintroduces exactly the ambiguity M1 flagged (what, precisely, counts as "content" versus "metadata" for the purposes of an idempotency test), and — unlike (A) or (D) — it conveys *when the code happened to run*, not *how current the underlying data is*, which is a less useful signal for a reader trying to judge a report's trustworthiness (a report re-run seconds apart with no new activity would show two different "generated at" values despite being informationally identical, actively suggesting a false impression of change).
- **(C) A monotonic run counter ("regenerated N times").** Considered and rejected quickly: it answers a question ("has this been run before") no report reader has, at the cost of a new piece of state Module 08 would need to track and persist somewhere — in direct tension with §15's confirmed decision that Module 08 introduces no new `Database/` structure.
- **(D) A data-derived "as of" marker — the latest action-log entry's own `timestamp` field among the entries a report actually includes.** **Chosen.** This conveys genuine, useful recency information (unlike (A)), is itself part of the same unchanged source data G5 is already defined against rather than a live clock read (unlike (B), so it introduces no idempotency ambiguity at all — re-running against identical source data reproduces the identical marker), and requires no new state (unlike (C), since `read_action_log_entries()` already returns each entry's own `timestamp`).

**Why (D) is the smallest solution with the lowest long-term maintenance cost:** it requires no new field, no new Database structure, no new test-harness carve-out logic (G5's idempotency test becomes a plain full-content equality assertion, not a "compare everything except line N" assertion), and it remains correct and useful regardless of how OD-1 is eventually resolved — a genuine per-period file (Daily/Weekly) and a continuously-updated current-state file (a candidate OD-1 outcome) both benefit from a data-derived "as of" marker identically, so this resolution does not need to be revisited once OD-1 resolves. **Compared against the precision discipline this project's own design-review process already established** (`Governance/ENGINEERING_STANDARD.md` §3; the same standard `Module 07 Design Review.md`'s own Round 1 M1 finding applied — a module's determinism/idempotency guarantee should be scoped against explicitly named variables, not left to informally-understood exceptions; **correction, Round 4 cleanup pass, resolves Design Review Round 3 finding N4:** this section previously mis-cited `ARCHITECTURE_DECISIONS.md` decision 13, which is "Independent module versioning" and does not govern this claim — no Architecture Decision actually governs guarantee-scoping precision, so the citation is corrected to the governance document and prior-module precedent that actually do): (D) achieves a *stronger*, not merely differently-worded, guarantee than the original draft — full byte-for-byte equality, unconditionally — which is a better outcome than the review's own suggested "smallest fix" (option (A) above) once (A)'s information-loss risk under a plausible OD-1 resolution was identified.

## 8. DOES NOT MODIFY (exhaustive ownership boundary)

Module 08 never sets or touches any of the following, under any code path:

- **Every `FileRecord` field owned by Modules 01–07** — `file_id`, `source_id`, `original_name`, `original_path`, `current_path`, `extension`, `mime_type`, `size_bytes`, `created_at`, `modified_at`, `content_hash`, `discovered_at`, `status`, `error`, `category`, `classification_signals`, `extracted_metadata`, `suggested_name`, `suggested_destination`, `naming_signals`, `duplicate_of`, `version_group_id`, `version_rank`, `duplicate_signals`, `confidence_score`, `confidence_breakdown`, `tier`, `batch_id`, `processed_at`, `approved_by`, `approved_at`, `reversible` — every one is read, none is ever rewritten. **Clarified in this cleanup pass (resolves Design Review Round 2 finding N2):** "read" here means loaded into memory as part of the full `FileRecord` object `load_metadata_store()` already returns (there is no partial-load mechanism in this codebase — every module loads whole records). For `extracted_metadata` specifically, "loaded" is as far as it goes: per §13/§18's M4 resolution, no report-generation function ever accesses or displays its content — the field is present in the loaded object but never used.
- **`Database/Metadata/metadata_store.json`** — read via `load_metadata_store()`, never written.
- **`Database/FileIndex/*.json`, `Database/History/version_history.json`** — Module 04's exclusively; Module 08 does not read these directly either (it reads the `duplicate_of`/`version_group_id`/`version_rank`/`duplicate_signals` fields Module 04 already wrote onto each `FileRecord`, which is sufficient for the Duplicate Report — no direct FileIndex/History access is needed or performed).
- **`Database/Learning/User Corrections.json`** — not written; may optionally be read for a report's informational purposes only (e.g. "N corrections logged this period") — see §25 item 2 for the confirmed disposition (narrow, contained question, deliberately not promoted to a numbered open decision); if read, it is read-only, exactly like every other source.
- **`Runtime/Logs/action_log.jsonl`** — read via `read_action_log_entries()`, never appended to. This is the single most important line in this section: every earlier module writes to this file; Module 08 is the first and only module in the pipeline that reads it back as a primary data source without ever adding to it.
- **`Runtime/Temp/*`** — Module 07's exclusively; Module 08 has no interaction with in-flight batch staging.

**No disclosed exception.** Unlike Module 04 (`version_group_id`/`version_rank` on another record) and Module 07 (`current_path`), Module 08 has zero disclosed exceptions to its "writes nothing outside its own output folder" rule — the cleanest ownership boundary of any module in the pipeline, a direct consequence of its purely read-and-summarize purpose.

## 9. Internal architecture

One layer, not two or three (§2):

```
generate_daily_summary(date) -> str        (reads action log + metadata store, writes Reports/Daily Summary/)
generate_weekly_summary(week) -> str       (reads prior Daily Summaries + action log, narrowly, per §5/§12; writes Reports/Weekly Summary/)
generate_duplicate_report() -> str         (reads metadata store, writes Reports/Duplicate Report/)
generate_storage_report() -> str           (reads metadata store [+ destination library, per OD-4], writes Reports/Storage Report/)
```

Each function is independently callable and has no dependency on the others completing first, except `generate_weekly_summary()`, which reads already-written Daily Summary files rather than re-deriving a week's *aggregation* from the raw action log directly — a deliberate choice: it keeps Weekly Summary consistent with whatever Daily Summary already reported for each day (including any day-level nuance a future Daily Summary template might add) rather than risking two different aggregation code paths silently producing different numbers for the same underlying data. (One narrow, disclosed exception: it also consults the raw action log solely to disambiguate a missing day, never to re-derive a day's actual figures — §5/§12.) Each function follows the same internal shape: **load** (action log and/or metadata store) → **filter** (to the relevant time window or signal type) → **aggregate** (count, group, sum) → **render** (to the fixed Markdown template already committed in `Metadata & Log Schema.md`) → **write** (to the correct `Runtime/Reports/` subfolder).

## 10. Report generation lifecycle

1. **Trigger.** See OD-3 — whether this is its own explicit CLI step (`python src/main.py report`, matching every other module's own explicit-step precedent) or automatically invoked after Module 07's `execute()`. This design does not resolve that question unilaterally.
2. **Load.** Read the full action log and/or metadata store (or, for Weekly Summary, the relevant already-written Daily Summary files).
3. **Filter and aggregate.** Scope to the report's own window/signal type; compute every count/table row per §3's four report types, per the exact shapes already committed in `Metadata & Log Schema.md`.
4. **Render and write.** Produce the Markdown file at its fixed path; never overwrite a previously-closed period's report (G6/OD-2).
5. **Report the report.** A CLI-level summary of what was generated (file path, counts) — the same "tell the user what just happened" pattern every earlier module's CLI wiring already follows (`src/main.py`).

## 11. Persistence model — report file semantics

- **Daily Summary — one file per calendar day, append-within-day, never rewritten across days.** If Module 08 is invoked more than once on the same day (multiple batches), the same day's file is updated to reflect the day's cumulative activity so far — this is not a violation of G6, since the day is still "open" until it ends; G6's "never silently rewritten" guarantee applies to a **closed** period (a prior day), never today's still-accumulating one. The exact boundary of "when does a day close" is not resolved by this design — **promoted in this cleanup pass to OD-5 (§22)**, resolving Design Review Round 2 finding L2.
- **Weekly Summary — one file per reported week**, generated from already-closed Daily Summaries only (§9) — never generated from a still-open, current day's partial data.
- **Duplicate Report / Storage Report — shape genuinely open (OD-1).** The pre-design note calls both a "running view," which most naturally reads as a single, continuously-updated current-state file rather than a dated-snapshot series — but this is in tension with G6's "never silently rewrite a closed period" guarantee, since a single current-state file is, by definition, always being rewritten. This design does not resolve the tension unilaterally; see OD-1.

## 12. Failure model

Two layers, per the established project-wide philosophy (`ARCHITECTURE_DECISIONS.md` decision 18), scaled down to match Module 08's genuinely lower stakes — a report is regenerable from source data at any time, unlike a real filesystem move:

**Layer 1 — anticipated failures, handled with a named, honest result, never a silent guess and never a crash:**
- `action_log.jsonl` doesn't exist yet (fresh install, no activity) → treated as "nothing to report," mirroring `read_action_log_entries()`'s own already-implemented "returns `[]` if the log doesn't exist" precedent — not an error.
- A malformed/corrupt line in `action_log.jsonl` (e.g. a truncated write from a genuinely interrupted process, per Module 07's own disclosed G2 OS-level limitation) → skipped, with an explicit, visible note in the generated report itself ("N log lines could not be parsed and were excluded") rather than either crashing the whole report or silently under-counting with no disclosure — this is the same "honest disclosure over silent gap" principle decisions 7/19 already establish, applied to a parsing failure instead of a judgment fallback.
- A `FileRecord` a report needs to reference is missing an expected field (should not happen given every earlier module's own ownership guarantees, but handled defensively rather than assumed impossible) → the report shows an explicit "unknown"/omitted cell for that value, never a fabricated placeholder (G3).
- `Runtime/Reports/` is not writable (permissions, disk full) → the report generation call fails clearly and visibly; critically, **this failure never reaches backward to affect anything Module 07 already did** (G4/I4) — by construction, since Module 08 never touches `Database/` or `Runtime/Logs/` at all, there is no shared-state path through which a Reports/ write failure could corrupt or roll back a real file operation.
- **A day within a requested week has no Daily Summary file (§9's dependency).** **Added in this cleanup pass (resolves Design Review Round 2 finding L1).** Two distinct causes must never be conflated: (a) genuinely no activity that day (an expected, legitimate case), or (b) that day's own Daily Summary generation previously failed under one of this section's own Layer 1 cases and was never produced (a reporting gap, not a quiet day). `generate_weekly_summary()` distinguishes them by checking whether `Runtime/Logs/action_log.jsonl` has any entry timestamped that day: zero entries → the week's report shows that day as "no activity"; one or more entries but no corresponding Daily Summary file → the week's report shows that day as "report unavailable for this date," visibly, never silently treated as zero activity. This directly extends G3's "never fabricate, always disclose" principle to Weekly Summary's own aggregation step, not just to Daily Summary's.

**Layer 2 — the outer safety net**, at whatever orchestration point invokes Module 08 (the CLI, or a post-step hook per OD-3): a report-generation exception is caught, logged, and does not propagate to fail the step that triggered it — directly implementing the pre-design note's own already-stated requirement (§0.1 G4).

**No crash-reconciliation mechanism is needed, unlike Module 07.** Module 08 has no multi-step filesystem-plus-log-plus-record sequence that could be interrupted mid-way in a way that corrupts state outside its own output folder — the worst case of an interrupted report generation is a partially-written or missing report file, which is always safely regenerable from the still-intact action log and metadata store on the next run. This is a genuine, favorable difference from Module 07's failure model, not an oversight in this one.

## 18. Security considerations

- **No file-content-derived code-execution risk.** Module 08 never opens or interprets the *content* of any user file — it reads only its own two structured, pipeline-generated data sources (the action log, the metadata store) and, under OD-4, folder/file-size metadata from the destination library — never a user file's bytes.
- **Read access to the destination library (if OD-4 resolves toward real filesystem measurement) is a new dependency class**, though read-only and lower-risk than Module 07's write access to the same location — still worth an explicit adversarial test (a destination folder containing a symlink loop, an unreadable subfolder, or a permissions-denied entry should degrade to an honest "could not measure" note, per §12's Layer 1 philosophy, never a crash and never a fabricated size).
- **No sensitive-value exposure risk from `extracted_metadata`, by construction rather than by later verification.** **Revised in this corrective pass (resolves Design Review Round 1 finding M4).** The original wording deferred to an implementation-time check that report templates "never surface a full `extracted_metadata` value" — the same class of audit-time-only safeguard `ARCHITECTURE_DECISIONS.md` decision 8 explicitly rejects for exactly this reason ("would make the privacy guarantee only as strong as the [implementer's] behavior on any given call"). **Alternatives considered:** (a) a small, named, per-category allow-list of `extracted_metadata` fields safe to display (mirroring Module 03's own closed-taxonomy pattern) — rejected as unnecessary, not unsafe: re-reading `Metadata & Log Schema.md`'s own already-committed worked examples for all four report types (Daily Summary's per-file table: original name, new name, destination, category, confidence, tier; the Duplicate/Storage Report descriptions) confirms none of them actually requires any `extracted_metadata` field to fulfill its stated purpose — inventing an allow-list for a capability nothing currently needs would be scope the project owner never asked for and a business-rule confirmation this design doesn't need to request. (b) **Chosen:** Module 08 does not read or display `extracted_metadata` content in v1 at all (§3/§13) — the conservative default per `ARCHITECTURE_DECISIONS.md` decision 3's "defer genuinely useful but non-essential capability to an explicit future decision" discipline, the same treatment Module 03 itself gave naming-suggestion logic when it deferred that entirely to Module 05 rather than guessing at scope. This resolves the privacy question structurally (there is nothing to leak because the field is never read for this purpose) rather than by a later, template-by-template check, and is smaller than option (a): no allow-list to define, confirm, or maintain. If a genuine future need to surface a specific `extracted_metadata` field in a report ever arises, that is itself a new, explicit, project-owner-confirmed decision at that time (matching decision 5's "a new, disclosed reason" bar for any similar future extension) — not something this corrective pass decides preemptively.
- **No path-escape risk analogous to Module 07's.** Module 08 writes only to its own fixed, pre-determined `Runtime/Reports/` subfolder paths (never a user-suppliable or file-content-derived path), so the adversarial-input class Module 07's §22 named (a `../`-bearing suggested name/destination) does not apply here — worth stating explicitly so a future reviewer doesn't need to re-derive why this module's security surface is narrower.

## 19. Performance expectations

A measured (not estimated) number is required at release time (§0.3), matching every module's own precedent. **Revised in this corrective pass (resolves Design Review Round 1 finding M3).** The original wording described Module 08's cost as one blended "O(M) — a full read of the entire action log and/or entire metadata store" figure. This conflated two data sources that are not the same size and do not grow at the same rate, and — per §5 — are not both read by every report type:

- **`Database/Metadata/metadata_store.json` — O(records).** Exactly one record per file ever discovered, read via `load_metadata_store()`. This is the *only* source `generate_storage_report()` reads (absent OD-4 resolving toward a real filesystem walk) and the primary source `generate_duplicate_report()` reads.
- **`Runtime/Logs/action_log.jsonl` — O(log lines), a strictly larger and faster-growing number.** Per `Metadata & Log Schema.md`'s own committed action vocabulary, a single file that reaches full, successful, `auto`-tier execution accumulates roughly seven entries (`discover`, `classify`, `extract_metadata`, `detect_duplicates_and_versions`, `suggest_naming_and_destination`, `score_confidence`, `move_rename`) — more if it triggers `error`/`reject`/`undo` along the way, fewer if it exits the pipeline early (e.g. `status == "unreadable"` after `discover`, or left `review_required` and never executed). This is read by `generate_daily_summary()` and `generate_weekly_summary()` (via already-written Daily Summaries, §9) and by `generate_duplicate_report()` for `detect_duplicates_and_versions` entries.

**Both figures must be measured and disclosed separately at release time** — not blended into one number — in Module 08's own `KNOWN_LIMITATIONS.md`, re-disclosing the same underlying "full-file-read cost" pattern `ARCHITECTURE_DECISIONS.md` decision 11 already named for `save_file_record()`, now applied to a full-file *read* rather than a full-file read-modify-write. This remains, if anything, more consequential than any earlier module's per-file cost, since Module 08 reads the entire relevant source(s) on every single invocation regardless of how narrow that invocation's reporting window is (a Daily Summary for one day still loads the whole cumulative action log to filter it down) — but the two sources must be measured and reasoned about independently, since a future scale investigation (e.g. the kind `Governance/PROJECT_RETROSPECTIVE_2026-07-13.md` already performs for `save_file_record()`) needs to know which of the two actually dominates at real scale, and they will not dominate equally.

## 22. Open architectural decisions requiring project owner confirmation

- **OD-1 — Duplicate Report / Storage Report persistence shape.** Single, continuously-updated "current state" file (matching the pre-design note's "running view" language) versus dated historical snapshots (matching Daily Summary's own established one-file-per-period pattern and G6's "never silently rewrite a closed period" guarantee). These two readings are in real tension (§11) and this design does not resolve it unilaterally.
- **OD-2 — Retroactive correction of a closed report.** If an undo or some other state change happens after a Daily Summary's day has closed, does that day's report stay exactly as originally written (treated as a historical record, consistent with this project's own "never rewrite history" documentation convention — decisions and CHANGELOG entries are never edited after the fact, only appended to), or does it get regenerated/appended-to with a correction note? This directly affects G6's precise definition and NG2's scope.
- **OD-3 — What triggers report generation.** Its own explicit CLI step (`python src/main.py report`, matching every other module's own explicit-step precedent: `scan`, `classify`, `extract`, `detect_duplicates`, `suggest_naming`, `score_confidence`, `preview`/`execute`), or an automatic hook invoked after Module 07's `execute()` completes (closer to the pre-design note's "the moment a file is processed" framing, though that framing is otherwise superseded per §0.6)? This affects §10's lifecycle and OD-5's "day boundary" question directly (promoted from §11 to a numbered decision in this same cleanup pass — the two are closely related but not identical: OD-3 is about what invokes report generation at all, OD-5 is about when a day's own report is considered closed once generation has been invoked).
- **OD-4 — Storage Report's data source.** Derived entirely from `size_bytes` values already present in `Database/Metadata/metadata_store.json` (no new dependency, but reflects file size *at discovery time*, not current on-disk reality if a filed file were ever modified in place — though v1's pipeline has no mechanism for that to happen post-filing, making this likely a non-issue in practice), or computed by actually walking the real destination library's folder structure (a new, disclosed read-only filesystem dependency, §18)? This determines whether Module 08 needs any new dependency at all.
- **OD-5 — The reporting "day boundary" rule.** **Added in this cleanup pass (resolves Design Review Round 2 finding L2 — promoted from an inline aside in §11 to a numbered open decision, matching OD-1 through OD-4's own treatment, since this question is load-bearing for §0.1 G6's precise application and was previously easy to lose track of relative to the other four.)** When is a reporting "day" considered closed for Daily Summary purposes, given the pipeline has no long-running process to observe a clock boundary crossing on its own (NG1 — "invoked... Manual or Scheduled... never a live dashboard or push notification," §0.2; **correction, Round 4 cleanup pass, resolves Design Review Round 3 finding N3:** previously mis-cited as "`ARCHITECTURE_DECISIONS.md` decision 17/NG4" — decision 17 is "Dependency chain (strictly linear in v1)" and NG4 is "Does not guarantee trend prediction or forecasting," neither of which supports this claim; NG1 is the correct citation and needs no `ARCHITECTURE_DECISIONS.md` decision number, matching NG1's own citation of only `README.md`'s Execution Modes table)? Candidate interpretations: (a) midnight in a fixed, configured local timezone, evaluated at the next invocation after that boundary has passed; (b) the day is "closed" the moment any invocation occurs on a later calendar date, with no independent clock-watching at all (the simplest option, requiring no new dependency — closure is inferred from invocation timing alone, not observed in real time); (c) an explicit, separate "close the day" action distinct from report generation itself (more moving parts, not clearly justified by anything named so far). Requires project-owner confirmation before implementation, since it directly determines when §0.1 G6's "never silently rewrite a closed period" guarantee begins applying to a given day's file.

Per this project's "never rewrite history" documentation convention, the original §11/§22/§24/§25 text above is left completely unedited — it remains an accurate record of what was true at design-freeze time (2026-07-14). This addendum records what was subsequently resolved, when, and where the full decision record lives; it does not alter the design's architecture. OD-2 and OD-4 remain deliberately deferred, exactly as §24 itself allowed ("more contained and could plausibly be resolved during implementation planning instead") — not resolved by this addendum.

- **OD-1 (Duplicate Report / Storage Report persistence shape) — RESOLVED.** Full record: `Governance/ARCHITECTURE_DECISIONS.md` decision 25. Both report types are a single, continuously-updated current-state file each (`Runtime/Reports/Duplicate Report/duplicate_report.md`, `Runtime/Reports/Storage Report/storage_report.md`), never dated snapshots. §0.1 G6's "closed period" protection is clarified to govern period-based reports (Daily Summary, Weekly Summary) only — Duplicate Report and Storage Report have no period concept and are therefore outside G6's scope by definition, not by exception. **Not yet implemented:** `Module 08 Implementation Plan.md` WP-3/WP-5's own finalization of `write_duplicate_report()`/`write_storage_report()` is this decision's implementation scope, not performed by this addendum.
- **OD-3 (report generation trigger) — RESOLVED.** Full record: `Governance/ARCHITECTURE_DECISIONS.md` decision 26. Report generation is its own explicit CLI step, `report()` in `src/main.py`, matching every other module's own explicit-verb precedent — never an automatic hook into Module 07's `execute()`. A single `report()` call invokes all four `generate_*()` functions in one pass. `execute()` itself is not modified in any way to support this decision. **Not yet implemented:** `Module 08 Implementation Plan.md` WP-6's own CLI wiring is this decision's implementation scope, not performed by this addendum.
- **OD-5 (day-boundary rule) — RESOLVED.** Full record: `Governance/ARCHITECTURE_DECISIONS.md` decision 27. A Daily Summary's calendar day is closed the first time `report()` is invoked with a current UTC date later than that summary's own date — inferred entirely from invocation timing, using the UTC timestamps every action-log entry already carries. No timezone configuration is introduced. **Not yet implemented:** `Module 08 Implementation Plan.md` WP-1's shared day-filter helper and WP-2's G6/day-boundary test already anticipated this exact mechanism (bucket by the entry's own `timestamp`, never wall-clock "today") and require no rework as a result of this decision.

No implementation code was written to produce this addendum. No frozen module (01–07) was touched. Module 08's design remains **Frozen** — this addendum records a decision-confirmation step the design itself anticipated (§22, §24 Risks) as a prerequisite to implementation, not a design revision. Per the project owner's explicit instruction, WP-1 has not been started; only WP-0 (this addendum and its corresponding `Governance/ARCHITECTURE_DECISIONS.md` entries) has been performed.
