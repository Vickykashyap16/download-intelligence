## Verification of F1–F4

### F1 (required vs. optional metadata) — RESOLVED, verified
§7 no longer infers required/optional status from the naming templates silently — it states, category by category, required fields, optional fields, and category-specific prohibited examples, on top of a universal prohibited-metadata rule (any field outside the required+optional list). §7 also adds, as general rules applying to every category: extraction priority (required fields never sacrificed for optional ones), minimum acceptable extraction (at least one required field found), a mechanical definition of "incomplete" (`extraction_complete = all(...)`, not a judgment call), and four explicit, independently-checkable stopping rules (before starting, after a deterministic-only pass, after one provider call with no retry, and never cross-method per field). §7A explains consumption by Module 05 (naming, including where Module 05 — not Module 03 — resolves compound slots and filesystem-date fallbacks), Module 06 (the taxonomy is now the definition `Confidence Rules.md`'s deduction formula always assumed existed), and Module 07 (inherits the privacy guarantee passively for a future approval preview). Critically, §7's own opening paragraph and the document's closing paragraph both now state explicitly that the required/optional assignments are this design's *proposal*, not a settled business rule, and require your confirmation before being treated as final — this was the crux of the original finding (a business-rule judgment being made unilaterally) and is now handled as a flagged, pending decision rather than quietly assumed. **Verified resolved**, with one residual, correctly-disclosed dependency: the specific per-category assignments in §7's table still need your explicit sign-off (noted below as the one item this review cannot itself close out, since it is a business decision, not an architectural one).

### F2 (redaction) — RESOLVED, verified
§18 replaces the earlier qualitative "pattern check... 8+ digits... common account/routing-number-like patterns" description with an exact, four-step, testable rule scoped to `account_last4`: strip non-digits, redact to `null` if the result is longer than 4 digits, pass through unchanged at 4 digits or fewer (including empty), run exactly once, never retroactively. §18 also directly answers every part of the original request in its own named subsection: what remains visible (`bank_name`, `statement_period`, `account_last4` when ≤4 digits), what is stored internally (nothing beyond what's visible — redaction is deletion, not relocation), what appears in logs (the field *name* only, in `redacted_fields`, never the value), and what later modules may access (only the post-redaction dict, with no path back to the original value for any module, present or future). §20's test list now specifies the exact boundary cases (4 digits passes, 5 digits redacted, empty passes, a *different* category's long numeric field must not be redacted) — a deterministic rule with a corresponding deterministic test, not a described intention. **Verified resolved.**

### F3 (privacy scope) — RESOLVED, verified, and the reasoning holds up under a second look
§18 generalizes the closed-taxonomy control (no field outside required+optional, for any category) as the universal, primary privacy control — directly answering "should this apply to categories beyond Bank Statement" for the general case: yes, and it already did, via §12's existing key-validation mechanism, now named explicitly as a privacy control rather than only a data-integrity one. For the *value-level* check specifically, §18 gives a principled reason `account_last4` alone gets the extra rule: it is the only field in the entire taxonomy whose defined purpose is to be a truncated-safe view of a value that is itself prohibited. This review re-derived the same conclusion independently by re-reading §7's full per-category table: no other field (Invoice's `amount`, Contract's `effective_date`, Resume's `candidate_name`) is a deliberately-truncated view of something worse — they're either meant to be captured in full or are free descriptive text with no single unsafe superset value. The design also correctly explains why a blanket digit-pattern scanner across every field would be actively harmful, not merely unnecessary (it would misfire on fields meant to hold long numbers, like `invoice_number`), which is a genuine over-engineering trap avoided, not just asserted to be avoided. The user's prompt specifically asked about Medical documents and IDs as illustrative examples; §7's table correctly does not invent categories outside the current v1 taxonomy (Invoice, Resume, Bank Statement, Contract, Document, Image, Screenshot, Application, Archive, Video, Audio, Unknown, per `Rules/Classification Rules.md`) — this review confirms that was the right call: answering for hypothetical categories not in scope would have been scope creep the user's own "avoid over-engineering" instruction warned against. **Verified resolved**, reasoning independently re-checked, not just re-stated.

### F4 (timestamp source priority) — RESOLVED, verified
§9A defines exactly the four-tier hierarchy the prompt's own example suggested (embedded metadata → EXIF → container metadata → filesystem timestamps), maps each tier to what's actually implemented in v1 (tier 1: Audio only, pending `mutagen`; tier 2: Image/Screenshot via existing `core/exif.py`; tier 3: nobody, deferred; tier 4: never written into `extracted_metadata` by Module 03 at all, reserved for Module 05's own naming fallback), and gives a substantive reason for the order — each tier answers a progressively less specific question, and a field named `capture_date`/`content_date`/`recording_date` promises the answer to the *most* specific question, not the least. This review specifically checked the claim that tier 4 is "never" used by Module 03 against §7's table and §9's table and found it consistent in both places — Video's `content_date` and Audio's `recording_date` are both documented as `null` rather than filesystem-time-populated, with the fallback responsibility explicitly assigned to Module 05 instead (§7A). This is a stronger, more precise answer than the original design draft gave (which had ambiguously implied Module 03 might use `FileRecord.modified_at` directly for Video's naming slot) — the remediation didn't just add a hierarchy, it corrected a real latent inconsistency the first review's F4 finding had only partially characterized. **Verified resolved.**
