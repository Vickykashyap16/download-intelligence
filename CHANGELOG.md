## 2026-07-06 — Module 01 review fixes: file_id redesign, schema sync

Review caught a real bug in the file_id design from the initial Module 01 pass, plus three confirmations and a documentation gap. Fixed before Module 02 starts depending on this module's output.

**file_id redesign.** The original design hashed the absolute path to get a stable, deterministic file_id, specifically to avoid content-hash collisions (two identical-content files merging onto one record). That reasoning was right, but the fix was wrong: Module 07 moves and renames every file it files, so a path-derived identity either goes stale the moment that happens, or requires perfect discipline everywhere to never re-derive it from a file's current location — one careless `generate_file_id(new_path)` call anywhere in the future and a record silently orphans itself from its own history.

Fixed by separating three concerns that had been conflated:
- **`file_id`** — now a UUID4, assigned once at first discovery, never recomputed from anything about the file. Permanent for the life of the record.
- **`content_hash`** (renamed from `hash_sha256`) — SHA-256 of file bytes, unchanged in purpose: what Module 04 compares across *different* records to find duplicates.
- **`current_path`** (new field) — the live "where is this file right now" pointer. Module 01 sets it at discovery (equal to `original_path` at that moment); Module 07 is responsible for updating it after every move/rename. `original_name`/`original_path` now unambiguously mean "first ever seen as," never touched again.

Trade-off accepted: a random UUID4 alone can't tell you "have I already discovered this exact file" across repeated scans the way a deterministic hash could. Solved with `find_by_current_path()` (`src/storage/database.py`) — before minting a new ID, Module 01 checks whether a record already exists at that exact `current_path`; if so, it reuses that file_id and `discovered_at` rather than creating a duplicate entry for a file nobody's moved yet. This is re-identification for idempotent re-scanning only, not duplicate detection (still Module 04's job, still based on comparing `content_hash` across *different* paths).

Bonus catch from the same fix: if a re-scanned path's `content_hash` no longer matches what's on record, that's now flagged in the action log (`content_changed_since_last_scan`) instead of being silently overwritten — covers "file edited in place" or "a different file landed at this exact name."

**Also removed:** `new_name`/`new_path` fields — fully redundant with `current_path` once it existed (in v1 a file moves exactly once; both would always hold identical values after execution). One field, one meaning, no duplicated concept.

**Confirmed, no code change needed:**
- `metadata_store.json` is the complete cumulative file database, not a per-scan snapshot — strengthened the wording in `Database/README.md` to say so explicitly, since the code already worked this way but the doc didn't spell it out.
- Recursive folder scanning is intentionally out of scope for v1 — already correctly implemented (`scan_source()` only calls `directory.iterdir()`, explicitly skips subdirectories) and documented (`Rules/Ignore Rules.md` "Source scope").

**Documentation sync:** `Build-out/08 .../Metadata & Log Schema.md` had gone stale the moment Module 01 was actually built — updated to match `src/models/file_record.py` exactly (the canonical source going forward if they ever drift again), including the identity-fields explanation above.
