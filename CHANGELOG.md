## 2026-07-20 — Module 08 WP-6 pre-implementation review: report() CLI invocation placement resolved (decision-recording only, no code)

Following a dedicated pre-implementation engineering review of WP-6 (`report()` CLI wiring), one scope question was surfaced that decision 26 (OD-3's resolution, 2026-07-14) did not address: whether `report()` should be appended to `main.py`'s automatic `if __name__ == "__main__":` chain (`scan() → classify() → extract() → detect_duplicates() → suggest_naming() → score_confidence()`) or remain a separate, explicitly-invoked command like `execute()`/`undo()`. Resolved: `report()` is a separate, explicitly-invoked CLI command, never part of the automatic chain. Reports summarize completed execution results; Daily Summary's own content derives from `move_rename`/`archive_duplicate`/`archive_superseded_version`/`score_confidence` action-log entries that don't exist yet at the point `score_confidence()` finishes in the chain (before `execute()` has run) — auto-chaining would produce technically honest but systematically premature output on every default invocation. Keeping `report()` explicit preserves a predictable workflow and mirrors `execute()`/`undo()`'s own existing "consequential enough to require deliberate invocation" precedent.

**Documentation updated:** `Governance/ARCHITECTURE_DECISIONS.md` (decision 31, following the established five-part format); `Module 08 Design.md` (a new, dated "Addendum — WP-6 `report()` CLI Invocation Placement" section appended after the existing addenda, append-only, which also records WP-5's now-complete status without editing the earlier addendum's own stale "WP-5 has not been started" sentence); `Module 08 Implementation Plan.md` (a non-destructive status note appended to WP-6, confirming decision 26 already eliminated its "touches `execute()`" branch entirely and decision 31 now resolves the CLI-placement question).

This decision governs CLI orchestration only — no report-generation logic (decisions 25/28/29/30, the four `generate_*()` functions) is affected. No implementation code was written. WP-6 is now architecturally frozen and implementation-ready; its own implementation is not performed by this entry.

## 2026-07-20 — Module 08 Implementation Planning: Storage Report scope clarified, Open Decision OD-4 resolved (decision-recording only, no code)

Following the project owner's approval of a dedicated architecture review into the tension between `Module 08 Design.md` §3's "growth over time" wording and decision 25's fixed current-state-file persistence shape, both that ambiguity and Open Decision OD-4 (Storage Report's data source, §22) — the one substantive gap left blocking WP-5 after WP-0 — were resolved. - **"Growth over time" resolved:** satisfied entirely by Storage Report's own current aggregate totals per category/destination, computed fresh from the metadata store on every invocation — never a persisted historical snapshot, rolling window, or run-to-run delta. Chosen because no code path in this pipeline currently causes a filed file's real size to diverge from its recorded value, because a filesystem-measurement answer would require extending I5 to name a source it does not currently name, and because it keeps Module 08's dependency footprint at exactly what WP-1 through WP-4 already established — the smallest option consistent with decisions 3/12's discipline. This resolution was the direct, motivating reason the growth-over-time ambiguity was resolved first: settling Storage Report as a current aggregate (not a trend) removed the one plausible reason to prefer live filesystem measurement.

## 2026-07-06 — Module 01 review fixes: file_id redesign, schema sync

Review caught a real bug in the file_id design from the initial Module 01 pass, plus three confirmations and a documentation gap. Fixed before Module 02 starts depending on this module's output.

**file_id redesign.** The original design hashed the absolute path to get a stable, deterministic file_id, specifically to avoid content-hash collisions (two identical-content files merging onto one record). That reasoning was right, but the fix was wrong: Module 07 moves and renames every file it files, so a path-derived identity either goes stale the moment that happens, or requires perfect discipline everywhere to never re-derive it from a file's current location — one careless `generate_file_id(new_path)` call anywhere in the future and a record silently orphans itself from its own history.

Fixed by separating three concerns that had been conflated:
- **`file_id`** — now a UUID4, assigned once at first discovery, never recomputed from anything about the file. Permanent for the life of the record.
- **`content_hash`** (renamed from `hash_sha256`) — SHA-256 of file bytes, unchanged in purpose: what Module 04 compares across *different* records to find duplicates.
- **`current_path`** (new field) — the live "where is this file right now" pointer. Module 01 sets it at discovery (equal to `original_path` at that moment); Module 07 is responsible for updating it after every move/rename. `original_name`/`original_path` now unambiguously mean "first ever seen as," never touched again.

Trade-off accepted: a random UUID4 alone can't tell you "have I already discovered this exact file" across repeated scans the way a deterministic hash could. Solved with `find_by_current_path()` (`src/storage/database.py`) — before minting a new ID, Module 01 checks whether a record already exists at that exact `current_path`; if so, it reuses that file_id and `discovered_at` rather than creating a duplicate entry for a file nobody's moved yet. This is re-identification for idempotent re-scanning only, not duplicate detection (still Module 04's job, still based on comparing `content_hash` across *different* paths).

Bonus catch from the same fix: if a re-scanned path's `content_hash` no longer matches what's on record, that's now flagged in the action log (`content_changed_since_last_scan`) instead of being silently overwritten — covers "file edited in place" or "a different file landed at this exact name."

**Also removed:** `new_name`/`new_path` fields — fully redundant with `current_path` once it existed (in v1 a file moves exactly once; both would always hold identical values after execution). One field, one meaning, no duplicated concept.

**Confirmed, no code change needed:**
- `metadata_store.json` is the complete cumulative file database, not a per-scan snapshot — strengthened the wording in `Database/README.md` to say so explicitly, since the code already worked this way but the doc didn't spell it out.
- Recursive folder scanning is intentionally out of scope for v1 — already correctly implemented (`scan_source()` only calls `directory.iterdir()`, explicitly skips subdirectories) and documented (`Rules/Ignore Rules.md` "Source scope").

**Documentation sync:** `Build-out/08 .../Metadata & Log Schema.md` had gone stale the moment Module 01 was actually built — updated to match `src/models/file_record.py` exactly (the canonical source going forward if they ever drift again), including the identity-fields explanation above.

Scope: discover supported files in a source directory, filter per Rules/Ignore Rules.md, generate a stable file_id, hash content, collect basic file info, and persist to Database/Metadata — nothing past that (no classification, naming, dedup, or file moves).

Implementation decisions worth recording:

- **`FileRecord` extended** with fields the frozen schema doc didn't include: `extension`, `mime_type`, `size_bytes`, `created_at`, `modified_at` (raw filesystem info Module 01 collects), plus `discovered_at`, `status`, `error` (ingest-tracking — `status` is `discovered` or `unreadable`). The schema in `Build-out/08 .../Metadata & Log Schema.md` predates any module actually being built and only had `hash_sha256`; these were added rather than overloading `extracted_metadata` (reserved for Module 03's category-specific content fields).
- **`file_id` = SHA-256 of the absolute path, not file content.** A content hash would make two files with identical bytes collide onto one file_id, silently merging their Database records before Module 04 ever gets to decide what to do about the duplicate. `hash_sha256` (content hash) stays a separate field for Module 04 to compare across records — that's still its job, not Module 01's.
- **Unreadable/locked files still get a FileRecord** (`status: "unreadable"`, `error` populated, `hash_sha256: null`) rather than being dropped, so Module 02 can route them per the existing "locked file, needs manual review" handling already described in `Rules/Classification Rules.md`. Genuinely unsupported/ignored/zero-byte/directory entries get a lightweight `SkippedEntry` instead — no FileRecord, since Module 01's spec only asks for records on valid files.
- **`append_action_log()` gained a `discover` action** (found + queued), a small documented addition to the `move_rename | archive_duplicate | archive_superseded_version | skip | error | undo` vocabulary in the schema doc, which was written before an ingest stage existed.
- **`storage/database.py`'s `load_metadata_store()`/`save_file_record()` and `storage/runtime_io.py`'s `append_action_log()` are now implemented** (Module 01 depends on both for its own explicit requirements); the FileIndex/History (Module 04) and report-writing/Temp (Module 07/08) functions in those same files remain stubs, out of scope here.
- **v1 supported-type list is narrower than the full category taxonomy** in `Rules/Classification Rules.md` (e.g. no `.exe`/`.msi`) — followed the Module 01 spec's explicit list (`.pdf`, image types, `.docx`, `.txt`, `.zip`, video/audio types, `.dmg`/`.pkg`) as given; worth reconciling with the fuller taxonomy before Module 02.
- **`config/sources.yaml` is now actually read** (via `load_source_config()`), the one config file that survived the "no YAML mirrors" decision. `path` must be filled in with a real directory before `python -m src.main` can run against a real Downloads folder; `scan_source()`/`build_ingest_queue()` can be called directly with an explicit path for testing without touching the config file.

## 2026-07-05 — Implementation start: src/ scaffold + Runtime/Database restructure

The frozen architecture had no folder for actual code — "Automation" build type assumed Claude would reason/bash its way through every run rather than rely on reusable scripts. Once implementation started, that assumption didn't hold up for the more mechanical steps (hashing, EXIF, JSON storage, file moves), so the structure was extended:

- **Added `src/`** — standard Python package layout, one module per `Build-out/` pipeline step (`pipeline/step01_watch_ingest.py` … `step08_logging_reporting.py`), plus `config/` (machine-readable YAML mirrors of `Rules/*.md`), `storage/` (Database + Runtime I/O), `models/` (shared data shapes), and `utils/` (hashing, text extraction, EXIF). Python 3 chosen — every tool identified during design was already Python-native. See `src/README.md`.
- **Modules are scaffolded, not implemented.** Every function is a signature + docstring raising `NotImplementedError`. Real logic gets built one module at a time, in order, starting with the deterministic steps (01, 04, 06, 08) before the Claude-judgment-dependent ones (02, 03, 05, 07).
- **Established the deterministic-vs-judgment module contract:** steps that are pure computation (ignore filtering, hashing, confidence scoring, logging) are ordinary testable code; steps that require reading/understanding a file's content (classification, metadata extraction) expose a request/response dataclass contract that Claude fills in live during a run rather than the code calling an LLM API itself.
- **Added `Runtime/`** (`Logs/`, `Reports/`, `Temp/`), moving the former top-level `Logs/` and `Reports/` under it — groups all *operational output* the automation writes while running, separate from `Database/`'s durable structured records. Added `Runtime/Temp/` for in-flight batch state, so a crash mid-execution can resume or be cleanly discarded instead of leaving things half-done (previously undocumented).
- **Moved `Learning/` under `Database/Learning/`** — it's persistent structured data like `Metadata/`/`FileIndex/`/`History/`, not operational output.
- Old top-level `Logs/`, `Reports/`, `Learning/` folders are now deprecated stubs, same treatment as `Registry/` (can't be deleted from this workspace).
